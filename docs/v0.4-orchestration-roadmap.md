# AgentCall v0.4 编排路线图

v0.4 是在 v3.1 MCP/ACP 落地之后形成的新设计方向：AgentCall 不应该在
ACP 和 PTY/Claude Code 会话之间二选一，而应该教主进程，尤其是 Codex，
如何同时协调这两种运行时。

## 问题

真实工程里，子 agent 经常失败，不是因为模型不够聪明，而是因为上下文不够。
它拿到的是一句任务，而不是一份可以执行的项目简报。它可能不知道当前项目阶段、
主进程已经做过的判断、其他子 agent 的工作、相关文件、风险边界，以及最终必须
回传什么证据。

反问有用，但反问不够。大规模 agent 系统需要一个主进程负责组装上下文、路由任务、
观察执行、合并报告，并且只在缺失信息真的是产品判断或风险偏好时才打扰人。

## 核心判断

AgentCall v0.4 本质上是四个部分协作：

```text
MCP      = Codex 可以调用的控制面
hooks    = Claude Code 会话发出的事件面
state    = AgentCall 维护的共享项目事实
playbook = 教 Codex 什么时候用哪条路径的操作规则
```

不要指望一段很长的 prompt 就能让 Codex 稳定成为超级组织者。组织能力应该被写进
工具 schema、路由工具、事件日志和验收协议里。

## 两条运行时路径

### ACP：Agents As Tools

ACP 是短生命周期的专家调用路径。

- 主进程保留最终控制权。
- 子 agent 接收 context packet。
- 子 agent 执行一个 bounded lifecycle。
- 子 agent 必须返回结构化 report。
- 主进程验证 scope、tests、lifecycle、risks 和 acceptance。

适合使用 ACP 的任务：

- 检查一个聚焦 diff
- 修一个小 bug
- 总结相关文件
- review 一个 plan
- 分类风险
- 调查一个边界清楚的测试失败

### Claude Code 会话：Handoffs

Claude Code session 是长生命周期 handoff 路径。

- 子 agent 拥有连续的本地工作上下文。
- AgentCall 观察 pid/status、transcript、hooks、tool calls 和 checkpoints。
- 主进程可以注入上下文、请求报告、暂停、恢复或重新分配。
- 用户仍然可以通过 PTY 风格 UI 看见和控制这个会话。

适合使用 handoff session 的任务：

- 大实现切片
- 深度 repo 探索
- 迭代调试
- 长迁移
- 模糊设计 spike

## 为什么 PTY 回来了

旧 PTY 原型的价值不在于“能往窗口里打字”。真正的价值是：Claude Code 本身已经是
一个 handoff runtime。PTY 是这个 runtime 的人类可见外壳。真正的数据源应该是：

```text
agent status
hooks
transcript
statusLine JSON
checkpoint reports
```

屏幕抓取和原始 stdin 只是 fallback 控制手段，不应该是主要协调机制。

## MCP 工具面

MCP server 应该暴露一组 Codex 能稳定使用的控制 API：

```text
agentcall_capabilities
agentcall_route_task
agentcall_context_packet_create
agentcall_delegate_acp
agentcall_session_spawn
agentcall_session_list
agentcall_session_status
agentcall_session_tail
agentcall_session_send
agentcall_checkpoint_request
agentcall_events_tail
agentcall_reports_list
```

`agentcall_route_task` 应该返回建议，而不是要求 Codex 自己记住策略：

```json
{
  "recommended_runtime": "acp",
  "reason": "bounded review task with low continuity needs",
  "required_context": ["diff", "acceptance_criteria"],
  "expected_output": "ChildReport"
}
```

或者：

```json
{
  "recommended_runtime": "claude-code-session",
  "reason": "large exploratory implementation needing long-lived context",
  "checkpoint_policy": "on stop, on idle, and after file edits",
  "expected_output": "CheckpointReport"
}
```

## Hook 事件面

Claude Code hooks 应该进入 AgentCall event bus。Codex 不应该手动监听一堆原始
hook，而是通过 MCP 查询整理后的状态。

目标事件：

```text
SessionStart
UserPromptSubmit
PreToolUse
PostToolUse
Notification
Stop
SubagentStop
PreCompact
SessionEnd
```

这些事件可以用来：

- 注册 session
- 注入 context packet
- 记录 tool calls 和 file edits
- 跟踪 permission wait 和 blocked state
- 更新 file claims
- 在 stop 或 idle 时强制 checkpoint report
- 在 compact 前保存 memory snapshot
- 在 session end 时释放 claim 并归档 transcript

## Shared State

AgentCall 应该维护结构化项目状态，而不仅仅是日志。

```text
.agentcall/state/
  project.json
  decisions.ndjson
  risks.ndjson
  file_claims.json
  active_sessions.json
  context_index.json
  memory/
```

这样主进程可以回答：

- 现在有哪些 agent 活跃？
- 谁 idle、blocked 或 waiting for permission？
- 谁正在碰哪些文件？
- 哪些 report 已经 ready？
- 哪些 risk 仍然 open？
- 下一个 child 需要什么上下文？

## Context Packet

每一次子 agent 调用或 handoff 都应该从标准 context packet 开始：

```json
{
  "task_id": "task-0001",
  "phase": "execute",
  "runtime": "acp",
  "objective": "Fix calculator.add",
  "allowed_paths": ["small_project"],
  "acceptance_criteria": ["add(2, 3) returns 5"],
  "relevant_files": [],
  "prior_reports": [],
  "decisions": [],
  "risks": [],
  "forbidden_actions": [],
  "output_contract": "ChildReport"
}
```

这个 packet 必须和 report 一起落盘：

```text
.agentcall/tasks/task-0001/calls/planner-01/input.json
.agentcall/tasks/task-0001/calls/planner-01/prompt.md
.agentcall/tasks/task-0001/calls/planner-01/context.json
```

否则我们只能审计 child 做了什么，不能审计 child 当时知道什么。

## 反问策略

子 agent 不应该在上下文薄的时候随便问人。它应该先返回结构化 sufficiency 结果：

```json
{
  "status": "need_context",
  "missing": ["current architecture decision", "test command"],
  "can_parent_resolve": true,
  "recommended_parent_action": "attach recent decision and package test command"
}
```

主进程处理顺序：

1. 自动补缺失的文件、报告、决策或测试命令。
2. 派另一个子 agent 做 bounded context-gathering task。
3. 只有缺失信息是无法推断的产品判断、风险偏好或意图歧义时，才问用户。

反问应该是 fallback 和路由信号，不是主要协作机制。

## 协作模式

有用的 child roles：

- `repo-scout`：寻找相关文件并说明为什么相关
- `plan-critic`：在执行前 review plan
- `executor`：做 bounded change
- `diff-reviewer`：检查真实改动，只在有风险时报告
- `test-analyst`：调查失败并提出下一步检查
- `context-summarizer`：把历史 report 压缩成 memory patch
- `handoff-owner`：为一个探索性切片持有长 Claude Code 会话

主进程接受干净工作，不写形式主义 review。只有当存在风险、漂移、证据不足、
blocked state 或低信心时，才触发 review。

## 实现阶段

### Phase 1：落盘子调用输入

- 增加 `calls/<call_id>/input.json`
- 增加 `calls/<call_id>/prompt.md`
- 增加 `calls/<call_id>/context.json`
- 在 report 里加入 context sufficiency 字段

### Phase 2：Router 和 Context Builder

- 增加 `agentcall_route_task`
- 增加 `agentcall_context_packet_create`
- 让 ACP calls 消费持久化 context packet
- 教 Codex 在选择 runtime 前先问 router

### Phase 3：Hook Ingestion

- 增加 hook receiver
- 把 Claude Code hook payload 归一化成 AgentCall events
- 跟踪 session lifecycle、tools、file edits、permission waits 和 stop events

### Phase 4：Claude Code Session Adapter

- 按 pid/status/session id 注册长生命周期 session
- 查询 status 和 transcript paths
- 请求 checkpoints
- 发送受控指令
- 通过 MCP 暴露 session status

### Phase 5：UI 和 Supervision

- 在同一个 board 展示 ACP calls 和 Claude Code sessions
- 展示 idle/running/blocked/needs-permission 状态
- 展示 reports、checkpoints、file claims 和 recent events
- 保留 PTY attach 作为可见控制层，但不把它当作事实来源

## 风险

- Claude Code status 和 hooks 暴露的结构可能比预期少。
- hook payload 可能随着版本变化。
- parent router 如果太聪明，可能会把重要选择从用户面前藏起来。
- 长生命周期 session 容易漂移，必须有严格 checkpoint policy。
- Codex 可能过度 delegation，所以工具返回值必须带强建议和限制。

## v0.4 完成标准

v0.4 成功时，Codex 应该能够：

1. 询问 AgentCall 一个任务该怎么 route。
2. 不手写长 prompt 就创建 context packet。
3. 通过 ACP 派发 bounded work。
4. 为长任务 spawn 或 attach Claude Code handoff session。
5. 查询谁 active、idle、blocked 或 ready for review。
6. 请求 checkpoint 并检查 reports。
7. 接受干净工作而不写 review artifact。
8. 只有在 AgentCall 无法补全上下文时才问人。
