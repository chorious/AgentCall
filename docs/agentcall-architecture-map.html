AgentCall 当前架构：一个本地多 Agent 控制面

AgentCall 不是“后台自动化平台”，而是 Codex 指挥 Claude Code PTY worker 的本地控制面。Codex 负责监督和验收；daemon 负责状态权威、路由、PTY、hooks、store、projection；Claude worker 负责可见的工程执行。

Parent

Codex 做决策

拆任务、派 worker、读 board/session、验收 report。正常情况不读 raw terminal 当状态源。

Authority

Daemon 写状态

routes、sessions、events、claims、leases、bindings、projections 都以 Rust daemon 为 live authority。

Runtime

Claude Code 做工作

Daemon-owned PTY worker 在配置的 claude_workspace 中运行，任务目标 workspace 由 route 描述。

Path A

派工路径

  1. Codex 调 agentcall_route。
  2. MCP 转发到 /api/mcp/call。
  3. daemon 校验 route、租约、write/report/scratch 边界。
  4. 创建 handoff/context packet。
  5. 启动 PTY worker，写 route/session projection。
Path B

观测写入路径

  1. Claude worker 执行工具或出现菜单。
  2. hook POST /api/hooks/ingest。
  3. daemon 绑定 wrapper session，记录 file claim / policy / report evidence。
  4. append_agent_event 更新 event + projection。
  5. store writer 顺序落库。
Path C

观察读取路径

  1. Codex 调 agentcall_board 或 agentcall_session。
  2. daemon 读取 projection / summary。
  3. 返回 compact state、why、primary_action。
  4. 只有 debug/raw 才展开 terminal/events。

状态与读写边界

核心原则：写路径可以重，读路径必须轻。hook、route、command 可以写 store；board/session 默认应读 projection，不应顺手 cleanup、不应扫大日志、不应重算 PTY 状态墙。

写路径：会改变系统

  • POST /api/routes 创建 route、lease、session。
  • POST /api/hooks/ingest 记录 hook event、policy、claim、report evidence。
  • agentcall_session_send 生成 command envelope 并进入 SessionActor。
  • SQLite 使用 WAL + single daemon writer，避免多 writer 争抢。

读路径：默认不改变系统

  • GET /api/board?view=compact 应只返回当前 compact projection。
  • agentcall_session(view=summary) 是 projection-first summary。
  • runtime_health 用于可用性，不应做维护写入。
  • debug/raw 读取才允许大 payload、大 terminal、事件展开。
当前最容易乱的点：“冷读 projection”不自动等于“返回小而当前的集合”。如果 projection store 里保留大量历史 session，compact board 必须有当前性过滤或冷却索引，否则 HTTP 可能很快但 payload 过大，MCP/前端仍会超时。

Workspace 模型

daemon workspace

AgentCall 仓库

daemon state、logs、runtime config、plugin、docs 所在目录，例如 E:\Project\AgentCall。

target workspace

任务目标目录

agentcall_route(workspace=...) 指向的项目。write_paths、report_path、workspace audit 都围绕它判断。

claude_workspace

Claude 进程 cwd

来自 config/agentcall.local.json。Claude Code PTY worker 在这里启动，并读取该目录 hooks。

report workspace

报告落点

route 明确或 daemon minted 的 report_path。report worker 可共享 workspace lease，但只写 report/scratch。

超时排查入口

1. Transport

MCP 是否旧版本

看 MCP 进程是否来自 target/runtime/<version>，以及 agentcall-version.json 是否匹配 daemon health。

2. HTTP

Daemon 是否慢

直接测 /api/runtime/health、/api/board?view=compact、/api/mcp/tools 的耗时和响应体大小。

3. Payload

Board 是否过胖

如果 HTTP 很快但响应体几百 KB，问题通常不是锁，而是 compact 返回了历史或过多字段。

架构规则

应该坚持
  • Codex 默认读 compact board/session projection。
  • daemon 是 live state authority。
  • SQLite 一条 daemon writer 写入；普通 reads 不做维护写。
  • worker 默认 PTY-first，可见、可 hook、可 report。
  • control token 只在需要控制权时短期暴露。
必须避免
  • board 读路径顺手 cleanup 或扫所有历史 projection。
  • MCP 用旧 debug binary 连接新 daemon。
  • 把 route workspace 误认为 Claude process cwd。
  • 用 raw terminal 文本替代 hook/report/daemon structured state。
  • 让 Python 脚本成为 live state writer。
