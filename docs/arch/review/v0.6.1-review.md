## 一、已经做对的部分（保留，勿回退）

daemon 这次实打实落了 R1 的硬核内核：

- `state_writer: Mutex<()>`（`daemon/main.rs:65`）+ `write_json_file` 临时文件 `fs::rename` 原子落盘（`:626`）—— 单写 + 原子写。
- `event_seq: AtomicU64`（`:64,84`）—— event id 进程内 O(1) 单调发号，干掉 O(n²) 数行。
- 全面 `chrono::Utc::now().to_rfc3339()` —— ISO-8601，修掉「unix:N」格式分裂。
- `expires_at` + `mark_expired_claims_stale`（`:824,893`）—— claim TTL/stale 回收。
- claim 策略入锁：`pre_tool_use_claim_locked` / `post_tool_use_observe_locked` / `release_claims_locked`（`:775,835,866`）。
- 暴露 `POST /api/hooks/ingest`（`:268`）与 `POST /api/events`（`:262`）—— 「外部写都过 daemon」的接收端已就位。
- MCP 增加 `--daemon-url`（`mcp/main.rs:36`）并把 `runtime_health` / `projects` 等转向 daemon。

这部分质量可以，方向正确，不要在整改中破坏它。

## 二、P0 阻断项：single-writer 只做了一半，最热的写路没接进来

**现象**：daemon 造好了 `/api/hooks/ingest` 安全入口，但 Claude Code hook 客户端从未被指过去。

证据：

- `scripts/agentcall-claude-hook.py` 未改，仍 `python -m agentcall hook ingest`（`:74`）。
- `src/agentcall/store.py`、`v2/hooks.py`、`v2/claims.py` **本轮 git diff 内全部未改**；`store.py:209 next_event_id` 仍数行、仍无锁直写。
- 即 Claude hook（每个工具调用都触发，是最高频写者）**绕过 daemon，直接用旧的 lock-free / line-count 路径写同一批 `events.ndjson` 和 `file_claims.json`**。

**后果**（daemon 与任一 Claude hook 在同一 workspace 同时存在时）：

1. **event id 必撞**：daemon `event_seq`（启动 seed 一次）与 Python「数行」各自发 `evt-NNNNNN`，写入同一日志，无协调 → 重号。
2. **`file_claims.json` 丢更新**：daemon `state_writer` Mutex 只约束 daemon 自身线程，**约束不到那个独立 Python 进程**，并发写 last-writer-wins。
3. **TTL 形同虚设**：`mark_expired_claims_stale` 只在 daemon 触碰该数据时生效；hook 路径走 `claims.py`（无 TTL），daemon 不保证会去碰。

**判定**：这是 R1 预警的「半迁移」。v0.6 的命题是「多 agent 并发下可信」，而产生并发的正是 hooks —— 对最该可信的场景，保证当前**不成立**。这一项不关闭，v0.6 验收不算通过。

### 整改要求（Codex）

1. **改 `agentcall-claude-hook.py`：把 ingest 改为 POST `http://127.0.0.1:3293/api/hooks/ingest`**（用标准库 `urllib.request`，不引第三方依赖）。
   - daemon 返回的 JSON 仍按现有逻辑转成 `hookSpecificOutput`（SessionStart/UserPromptSubmit 注入 context；PreToolUse deny）。
   - daemon **不可达时 fallback** 到本地 `python -m agentcall hook ingest`，保证 hook 永不阻断 Claude（fail-open）。
   - 同步处理 codex hook（`agentcall-codex-hook.py`）走同一入口。
2. **让 Python `Store` 退出「直接写共享 state」的角色**：`events.ndjson` / `file_claims.json` / `active_sessions.json` 的写入唯一权威是 daemon。Python 侧保留只读（board/inspect/report 读取）即可。
3. 关闭后用并发测试验证：daemon 在跑时，N 个 hook 进程并发 `PreToolUse Write` 同一文件 → 只允许一个 active owner；event id 全程无重复；`events.ndjson` 全行可 parse。

> 验收口径（呼应 R2）：测试必须「强制竞争下仍单 owner」，不是「跑了没崩」。加同步屏障或高迭代压力循环，且必须用真 OS 进程（非线程，避免 GIL 掩盖 race）。

## 三、P1 结构项：daemon `main.rs` 是 1669 行 monolith

`crates/agentcall-daemon/src/main.rs` = 1669 行、零 `mod`、单文件。PTY 生命周期 + HTTP/WS server + 路由 + claim 策略 + hook ingest + 状态 IO + summary 启发式 + TTL 全挤在一起。唯一接缝是 `route()`/`dynamic_route()`。

**目标**：按职责切模块，`main.rs` 退回「端口汇合」——只做解析参数 / 建 `AppState` / bind 端口 / 注册路由 / run。

建议切法：

```text
main.rs        —— 参数解析、AppState 构建、bind 端口、注册路由、run（汇合点）
state.rs       —— AppState、read/write_json_file（原子写）、event_seq、append_event、next_event_number_from_log
pty/session.rs —— Session、start_session、spawn_reader、spawn_waiter、write_input、resize、stop
claims.rs      —— pre/post/release_*_locked、mark_expired_claims_stale、stale_claim_count
hooks.rs       —— ingest_hook、append_unmatched_hook_locked、unmatched_hooks_state、session_id_from_payload
summary.rs     —— session_summary、looks_like_waiting_for_input、runtime_health、projects_state、board_state
http/mod.rs    —— Request、read_request、route、dynamic_route、json/error/static 响应、url_decode、ws 升级
```

约束：

- 纯结构性重构，**不改行为**；切分前后 `cargo test -p agentcall-daemon` 必须等价通过。
- 锁的获取顺序保持不变（`state_writer` 与 `sessions` 的相对顺序不要在搬动中改变，避免引入死锁）。
- 不顺手「改进」无关代码（遵循 CLAUDE.md 第 3 条 surgical changes）。

## 四、P2 顺带项（hook 接通后大半自动消失）

- `store.py:401 task_status_from_report` 两分支都返回 FAILED —— 死逻辑（OPUS P1，本轮未动）。
- `store.py:209 next_event_id` 数行 O(n²)（本轮未动）。
- `looks_like_waiting_for_input`（`daemon:579`）用 PTY 文本启发式判 `waiting_input`：可接受（daemon 读 PTY、Codex 不读，自洽），但**脆**。建议把判定关键词/正则集中成常量并加注释说明其启发式本质，避免被当成可靠信号。
- `event_seq` 启动时 seed 一次（`next_event_number_from_log`）：一旦 Python 仍在并发追加，seed 即过期 —— 这条只有在 P0 关闭（Python 不再直写）后才真正安全。**P0 是它的前置条件。**

> P2 中的前两条：把 hook 接进 daemon、Python 退出写角色后，这条 Python 写路径整体退役，两个债一并消失，不必单独修。

## 五、整改顺序（给 Codex 的 task 依赖）

```text
CC-1 (P0a) 改 hook 客户端 → POST /api/hooks/ingest + daemon 不可达 fallback
   └─> CC-2 (P0b) Python Store 退出共享 state 写入，确立 daemon 唯一写权威
          └─> CC-3 (验收) hook 并发压力测试：单 owner / 无重号 / 全行可 parse
CC-4 (P1) daemon main.rs 拆模块（行为等价，cargo test 通过）   # 与 CC-1~3 独立，可并行
```

每个 CC 交结构化 report；无风险直接接受，仅 conflict/drift/failed/blocked/low-confidence 触发 review。

## 六、一句话结论

daemon 的 single-writer 内核做得对、要保留；但 **R1 只闭合了 daemon 自己的半圈，hooks 这条最关键的写路仍绕过锁——并发可信的主命题尚未成立**。P0 = 把 hook 接进 daemon 并让 Python 退出直写；P1 = main.rs 拆模块、让 main 回归端口汇合。两者做完，v0.6 才算真正落地，可升 v0.6.1。
