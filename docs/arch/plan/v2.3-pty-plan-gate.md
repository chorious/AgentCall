# AgentCall v2.3: PTY Plan Gate

v2.3 把 PTY 明确定义为可视化 handoff runtime，并把默认工作流改成 `plan_then_auto`：Claude Code 先在 plan mode 中理解任务、提出计划、必要时提问；只有主管通过 `agentcall_session_send(action=approve_plan)` 或 `start_auto` 明确批准后，才进入 auto mode 执行。

这不是让 route 继续“猜任务大小”。v2.2 已经把 ACP 收敛为 5 个 SOP worker；v2.3 则把所有非 SOP、需要可视化协作或可能需要澄清的任务默认交给 PTY plan gate。

## 行为

- `agentcall_route(mode=start, runtime=pty)` 默认使用 `pty_workflow=plan_then_auto`。
- daemon 启动 Claude Code 时默认命令为 `claude --permission-mode plan --session-id <uuid>`。
- route 自动提交 handoff prompt，并要求 Claude 在 plan 阶段只读代码、写计划、必要时提问。
- Claude 调用 `ExitPlanMode` 时，hook 会把 route 标记为 `plan_ready`，summary/board 显示 `workflow_status=plan_ready`。
- 主管可以：
  - `agentcall_session_send(action=approve_plan)`：向当前 TUI 发送 `1`，切换到 auto mode。
  - `agentcall_session_send(action=start_auto)`：同样批准并进入 auto mode，作为语义化别名。
  - `agentcall_session_send(action=revise_plan, text=...)`：要求 Claude 继续修订计划。

## Hook Gate

plan 阶段不是只靠提示词约束。daemon hook policy 会执行硬边界：

- `Read/Grep/Glob` 等读操作允许。
- Claude 自己的 `.claude/plans` 计划文件写入允许。
- 项目实现文件的 `Write/Edit/MultiEdit/NotebookEdit` 会被拒绝。
- 非只读 `Bash` 会被拒绝。
- `ExitPlanMode` 会被允许，并转化为 route 的 `plan_ready` 状态。

这让 plan mode 的“不确定就提问”变成正常协作流程，而不是执行失败。

## 为什么不完全依赖 TUI

TUI 可以作为人类可见界面和最后一层可读信号，但 v2.3 不把 TUI 当状态权威：

- mode/status 优先来自 route/hook。
- `ExitPlanMode` 来自 hook tool event，而不是 grep TUI 文本。
- `approve_plan` 是明确控制动作，daemon 会同步写 route state。
- TUI 的 `1. Yes, and use auto mode` 仍作为实际切换手段，因为真实 smoke 已证明同一 PTY 中可用。

## 退出口

如确实需要跳过 plan gate，可在 route 中显式传：

```json
{
  "runtime": "pty",
  "pty_workflow": "normal",
  "initial_permission_mode": "auto"
}
```

默认路径仍推荐 `plan_then_auto`。ACP 继续只服务明确 SOP；PTY 承担复杂长任务和可视化协作。
