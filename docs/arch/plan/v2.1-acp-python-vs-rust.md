# AgentCall v2.1: ACP Python vs Rust Native 更新结论

Updated: 2026-06-04

## Summary

v2.1 不再把问题简化成“Rust ACP 是否已经替代 Python ACP”。更准确的判断是三层：

- **ACP 传输层**：Rust native ACP 已经优于 Python reference。它具备 timeout、stdout/stderr 并发读取、进程退出检测，并且已经通过真实 `@agentclientprotocol/claude-agent-acp` smoke。
- **Claude 启动 cwd**：ACP 已修正为与 PTY 一致。实际 child cwd 强制使用 daemon local config 的 `claude_workspace`；route 请求里的 `workspace` 只表达任务目标。
- **子智能体协作层**：修复前 Rust ACP route 只是裸 bounded invocation；v2.1 已补上 wrapper binding、context packet 默认化和 child lifecycle projection，使 ACP 调用能被 Codex/board 识别成受控子 Agent 生命周期。

## 当前证据

真实 ACP smoke 已确认：

- daemon health 显示 `acp_command_configured=true`。
- 本机 local config 使用 `C:\Program Files\nodejs\npx.cmd` 启动 ACP adapter。
- ACP route 完成，返回 `adapter=rust_native_acp`。
- adapter agent 是 `@agentclientprotocol/claude-agent-acp@0.40.0`。
- `stop_reason=end_turn`，child process exit code 为 `0`。
- smoke 任务实际创建 `.agentcall/reports/acp_real_smoke_v2.md`。
- route result 中 `cwd=D:\guKimi`，`workspace=E:\Project\AgentCall`。

这说明 ACP 不是“配置看起来可用”，而是真实启动、交互、执行并退出成功。

## 相对 OPUS 原文的订正

OPUS 原文中对传输层的主要判断仍成立：Rust ACP transport 比 Python reference 更稳。Python reference 的 `timeout_seconds` 没有真正包住 `_read()`，stderr 也不是并发 drain；Rust 版本在这些点上已经更接近可用控制面。

Windows `npx` 问题已通过 daemon local config 的 `acp_command` 解决。本机使用绝对路径 `C:\Program Files\nodejs\npx.cmd`。Rust 仍不应该假设裸 `npx` 在 daemon detached 环境里一定可解析。

Rust ACP `cwd` 已不再直接使用 route `workspace`。当前实现应保持：`workspace` 是任务目标，`cwd` 是 Claude/ACP child process 启动目录，并且由 `configured_claude_workspace(state)` 决定。

OPUS 对“Rust route 只返回原始文本、缺 report/schema 编排”的批评在修复前成立。v2.1 已增加最小 JSON/markdown report validation，并把 validation 结果写入 route result；完整 JSON schema 级验证仍可作为后续增强。

## 修复前关键发现

这次 ACP hooks 实际触发了。daemon events 里可以看到：

- `hook.SessionStart`
- `hook.UserPromptSubmit`
- `hook.PreToolUse`
- `hook.PostToolUse`
- `hook.Stop`

但是这些 hook 全部是 `binding_source=unbound`、`wrapper_session=null`。直接原因是 ACP spawn 没有像 PTY 一样注入 `AGENTCALL_WRAPPER_SESSION`。

当前 Rust ACP route 只写：

- `route.acp_invocation`
- `route.recorded`

旧 Python orchestrator 会写：

- `child.call_started`
- `agent.state_changed`
- `child.report_received`

所以 Codex 子智能体监听没有被唤起，不是因为 ACP 没跑，而是因为它跑成了一次裸 bounded invocation，没有被投影成受控 child lifecycle。

此外，ACP `session/new` 仍传 `mcpServers: []`。Rust client 目前只处理 `session/request_permission`。ACP updates 中出现 `availableCommands` 只表示 Claude ACP session 看到了命令列表，不等于 Codex 子智能体监听已经被调用。

如果 route 请求没有传 `task_id/call_id/role/phase/allowed_paths`，当前实现只会生成裸 route，不会生成 context packet，也不会形成可追踪 child-call identity。

## v2.1 已落地修正

### P0: ACP wrapper binding

- `AcpInvocation` 增加 `wrapper_session` 字段。
- wrapper session 默认使用 `call_id`，缺失时使用 `invocation_id`。
- ACP child process spawn 注入 `AGENTCALL_WRAPPER_SESSION=<wrapper_session>`。
- route result 增加 `binding_gate`，要求真实 hook 后看到 `binding_source=env`。

### P0: ACP child lifecycle projection

`mode=start,runtime=acp` 现在会在 daemon 内投影为子生命周期事件：

- start 前写 `child.call_started`。
- start 前写 `agent.state_changed`，状态为 `running`。
- 完成后写 `child.report_received`。
- 完成后写 `agent.state_changed`，状态为 `completed` 或 `failed`。

这些事件必须带：

- `task_id`
- `call_id`
- `role`
- `phase`
- `runtime=acp`
- `workspace`
- `cwd`
- `invocation_id`

### P1: report contract parity

- ACP prompt 改回 child invocation 语义，包含 context packet、mode rules、allowed paths、acceptance criteria。
- Rust route 对 ACP 文本结果做最小 JSON/markdown report 提取，并记录 validation 状态。
- 不再把 raw `text` 误当作已验收 report。
- report readiness 是交付物维度，不替代 liveness/status。

### P1: context packet 默认化

- 对 `mode=start,runtime=acp`，如果没有传 `task_id/call_id`，daemon 自动生成稳定 child identity。
- 默认 `task_id=route_id`。
- 默认 `call_id=invocation_id`。
- 如果调用方传了 context 字段，则使用调用方字段，不得丢失 role、phase、allowed paths。

### P2: route id 恢复与后续风险

- daemon 重启后 route id 现在会从既有 `routes.json` 和 event log 恢复 next seq，避免覆盖旧 route。
- ACP/Claude 输出仍可能出现破折号等 mojibake，需要进入 UTF-8 输出修复或报告清洗，但不阻塞 v2.1 主线。

## Test Plan

### Rust unit

- ACP invocation spawn 注入 `AGENTCALL_WRAPPER_SESSION`。
- forced ACP route 返回 `workspace=<任务目标>` 且 `cwd=<claude_workspace>`。
- ACP route 缺 `task_id/call_id` 时自动生成 child identity。
- ACP route 写出 child lifecycle projection events。

### Real smoke

- 启动 daemon。
- 调用 `agentcall_route(mode=start,runtime=acp)`，不传 `adapter_command`。
- 断言 ACP 完成、process exit code `0`、`cwd=D:\guKimi`。
- 检查 events：存在 hook events 且 `binding_source=env`、`wrapper_session=<call_id|invocation_id>`。
- 检查 board：ACP invocation 作为 child lifecycle 可见，而不是仅作为 route record 可见。

### Regression

- `cargo test -p agentcall-daemon`
- `cargo test --workspace`
- `python -m pytest -q tests\test_v08b_acp_parity.py`
- smoke 不得修改源码，只允许写 `.agentcall/reports` 或明确允许的报告路径。

## Assumptions

- v2.1 不恢复 Python orchestrator 作为默认 live 路线。
- Python reference 继续作为 parity/reference，不作为控制面权威。
- ACP 与 PTY 共享 `claude_workspace` cwd 规则。
- route `workspace` 是任务目标，不是 Claude/ACP child process cwd。
- Codex 子智能体监听需要 daemon-visible child lifecycle 和 hook binding，不是单纯 ACP 协议成功。
