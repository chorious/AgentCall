# AgentCall v0.6 修改方向：多 Agent 并发验收

v0.6 的主目标从“做出更完整的 PTY/summary 体验”调整为：**证明 AgentCall 在多 Agent 并发下仍然可信**。

上一轮 Android_pet 实战已经证明 PTY handoff、hooks、report 闭环能跑通。现在更重要的问题不是继续堆功能，而是验证核心承诺：多个 agent 同时工作时，文件 claim、事件日志、session 状态、report 验收和可视化 summary 是否仍然正确。

## 核心判断

AgentCall 的长期形态仍然是：

```text
Skills = 教 Codex 何时调度、何时验收、何时 review
MCP    = 给 Codex 和外部进程的结构化控制面
Hooks  = 写回真实执行事实
PTY    = 人类可视化和 handoff 外壳
```

但 v0.6 的优先级要改：

```text
并发可信度 > 状态聚合 > PTY 体验 > ACP/PTY 路由扩展
```

原因很简单：如果多 agent 并发下 claim 会双写、event id 会撞、summary 会误判，那么 PTY 画面再漂亮也只是把不可信系统可视化出来。

## v0.6 要回答的问题

v0.6 要用测试和真实运行回答这些问题：

1. 两个 agent 同时写同一个文件时，是否只能有一个获得 claim？
2. 多个 agent 同时写不同文件时，是否不会互相误伤？
3. 一个 agent 崩溃或 hook 丢失后，claim 是否会被 TTL/liveness 回收？
4. Python、Rust hook、Rust daemon 同时写事件时，event id 和 JSON 行是否稳定？
5. hooks 里的 `Read/Glob/Grep` 是否不再被误报成 write？
6. 不同项目的 PTY session 是否能在同一个 daemon/端口下稳定区分？
7. Codex 是否能不看 PTY，只靠 `session_summary` 判断每个 child 的状态？
8. child 漏报 changed_files 时，parent 是否能通过 git diff 或 workspace validator 发现？

## 测试路线

v0.6 不应该一上来就用 4 个真实 Claude Code 跑并发。真实 Claude 适合最终验收，不适合作为第一层并发测试。建议分四层：

### L0：纯 Store/Claim 并发单元测试

目标：不用 Claude、不用 PTY，直接压 AgentCall 的状态层。

场景：

- N 个进程同时 claim 同一个文件。
- N 个进程同时 claim N 个不同文件。
- claim 后不 release，模拟 session 崩溃。
- 同时 append events，检查 event id 唯一、NDJSON 无坏行。
- 同时读写 `active_sessions.json` 和 `file_claims.json`，检查无丢更新。

验收：

- 同文件只允许一个 active owner。
- 不同文件全部成功。
- claim 过期后可回收。
- events 全部可 parse。
- event id 无重复。

### L1：Hook Handler 并发测试

目标：压真实 hook 入口，验证 Claude Code hooks 的语义。

场景：

- 并发发送 `PreToolUse Write` 到同一文件。
- 并发发送 `PreToolUse Read` 到同一文件。
- `PostToolUse Read` 不产生 `observed_write`。
- `PostToolUse Edit/Write` 才产生写事件。
- 缺 `session_id` 时不能全部坍缩成 `"unknown-session"`。

验收：

- 冲突 deny 可以稳定出现。
- read 不污染 write claim。
- unknown session 进入 `unmatched_hooks` 或派生稳定 fallback id。

### L2：模拟 Agent 工作流并发测试

目标：不用真实 Claude，模拟多个 child lifecycle，同步验证 report/summary。

场景：

- 4 个 fake child 同时处理 4 个互不重叠任务。
- 2 个 fake child 同时试图改同一个文件。
- 一个 child 写了文件但 report 漏报 changed_files。
- 一个 child report 成功但测试失败。
- 一个 child 卡住不写 report。

验收：

- disjoint work 全部完成。
- conflict work 至少一个被拒绝或标记 blocked。
- 漏报 changed_files 被 git diff/workspace validator 抓到。
- 没 report 的 child 不会被判 completed。
- `session_summary` 能准确区分 completed / failed / blocked / drifted。

### L3：真实 Claude PTY 并发验收

目标：最终用真实 Claude Code 验证体验和链路。

建议先跑 2 个，再跑 4 个：

1. 两个 Claude PTY，同项目不同文件。
2. 两个 Claude PTY，同项目同文件，验证 claim conflict。
3. 四个 Claude PTY，四个不同任务，验证 summary 和 viewer。
4. 一个 Claude 故意卡住或等待输入，验证 `needs_user_input`。
5. 一个 Claude 试图越界，验证 drift warning。

验收：

- 人可以在同一个 viewer 看到所有 session。
- Codex 可以通过 MCP/API 拉到每个 session 的 summary。
- 不需要读 PTY 画面也能知道谁完成、谁卡住、谁越界。
- report 完成后 parent 可以验收，不机械 review。

## 架构调整

### 1. 状态权威优先

v0.6 必须先定状态写入策略。短期建议：

- Python 和 Rust 都可以保留，但写共享 state 时必须使用同一套锁和原子写协议。
- `events.ndjson` append 必须有单写保护。
- state json 写入必须使用临时文件 + atomic rename。
- event id 生成不能靠“数现有行数”。

长期方向：

- Rust daemon 成为 session registry、event writer、file claim writer 的权威。
- Python CLI/MCP 作为 daemon/API 的薄封装。

### 2. File Claim 变成真正并发原语

v0.6 的 claim 字段建议至少包含：

```json
{
  "file": "src/app.py",
  "session_id": "agent-a",
  "project": "AgentCall",
  "workspace": "E:\\Project\\AgentCall",
  "status": "active",
  "claimed_at": "2026-06-03T12:00:00Z",
  "updated_at": "2026-06-03T12:01:00Z",
  "expires_at": "2026-06-03T12:06:00Z",
  "owner_pid": 12345,
  "transcript_path": "C:\\Users\\MUSHI\\.claude\\projects\\...",
  "last_tool_name": "Edit"
}
```

规则：

- `Write/Edit/MultiEdit` 才 claim。
- `Read/Glob/Grep` 只 observe。
- active claim 过期或 owner dead 后可回收。
- 同一 session 可以 refresh 自己的 claim。
- 不同 session 写同一文件必须 conflict。

### 3. Session Summary 作为并发验收接口

`session_summary` 仍然重要，但它必须服务并发验收。

最小字段：

```json
{
  "session": "androidpet-v11",
  "project": "Android_pet",
  "transport": "pty",
  "status": "working",
  "workspace": "E:\\Project\\Android_pet",
  "claude_workspace": "D:\\guKimi",
  "last_hook_at": "2026-06-03T03:28:52Z",
  "last_tool": "Edit",
  "claimed_files": ["app/src/main/Foo.kt"],
  "files_written": ["report_v1.1.md"],
  "report": null,
  "needs_user_input": false,
  "warnings": [],
  "conflicts": []
}
```

状态枚举：

```text
starting
prompt_submitted
working
waiting_input
report_written
completed
failed
blocked
drifted
stale
```

### 4. 单 Daemon 单端口，多项目多 Session

v0.6 继续坚持一个 daemon：

```text
http://127.0.0.1:3293
```

多项目通过 registry 区分：

```text
/api/projects
/api/projects/{project}/sessions
/api/sessions/{session}/summary
/api/sessions/{session}/events
/api/sessions/{session}/terminal
/viewer?project=Android_pet
/viewer?session=androidpet-v11
```

每个 PTY session 启动时登记：

```json
{
  "session": "androidpet-v11",
  "project": "Android_pet",
  "workspace": "E:\\Project\\Android_pet",
  "claude_workspace": "D:\\guKimi",
  "command": ["claude", "--permission-mode", "auto"],
  "transport": "pty"
}
```

## v0.6 实施顺序

### Phase 1：并发测试先行

- 增加 L0 store/claim 并发测试。
- 增加 L1 hook handler 并发测试。
- 这些测试一开始可以失败，用来锁定真实问题。

### Phase 2：状态写入修复

- 统一 Python/Rust 时间戳为 ISO-8601。
- 引入锁或 daemon 单写队列。
- 修 event id 生成。
- state json 改原子写。

### Phase 3：Claim TTL 与语义修正

- 修 `Read` 被记为 write 的问题。
- 增加 claim TTL。
- 增加 owner PID/liveness 探活。
- 缺 session id 的 hook 进入 unmatched，而不是全归 unknown-session。

### Phase 4：Summary 与 Validator

- 增加 `session_summary`。
- 增加 `project_sessions`。
- report 验收加入 git diff/workspace validator。
- `_validate_changed_files_exist` 改用真实 child workspace。

### Phase 5：真实多 Claude 验收

- 2 个 Claude PTY，不同文件。
- 2 个 Claude PTY，同文件冲突。
- 4 个 Claude PTY，不同任务。
- 1 个 Claude 卡住或等待输入。
- Codex 只用 summary 验收，不默认读 PTY。

## MCP/API 调整

v0.6 最小工具面：

```text
agentcall_session_summary
agentcall_project_sessions
agentcall_file_claims
agentcall_unmatched_hooks
agentcall_runtime_health
agentcall_concurrency_probe
```

`agentcall_concurrency_probe` 可以先是测试/诊断工具，用来快速检查：

- 当前锁实现是否启用。
- event id 是否可能重复。
- stale claims 数量。
- unmatched hooks 数量。
- active sessions 数量。

## Skill 更新

v0.6 的 Codex skill 应写成短规则：

- 多 agent 任务先查 `runtime_health`。
- 派工前查 `project_sessions` 和 `file_claims`。
- 子 agent 完成后先查 `session_summary`，再读 report。
- 没有风险就接受，不机械 review。
- conflict、stale、drifted、failed、blocked 才触发 review 或二次派工。
- PTY 只做人类可视化，不作为 Codex 默认状态来源。

## 非目标

v0.6 暂时不做：

- 大规模 swarm。
- 完整分布式 parent/child 调度。
- 每项目独立 daemon/端口。
- 全量重写 Python。
- 把长 SOP 塞进 Codex 上下文。
- 用 PTY 画面替代 hooks/transcript/summary。

## 完成标准

v0.6 完成时，必须满足：

- 同文件并发 claim 测试稳定拒绝冲突。
- 不同文件并发 claim 测试稳定通过。
- event id 无重复，events.ndjson 无坏行。
- stale claim 可以自动回收。
- `Read/Glob/Grep` 不再生成 write claim。
- `session_summary` 能表达 completed/working/waiting/blocked/drifted/stale。
- 2 个真实 Claude PTY 可以并发跑不同任务。
- 2 个真实 Claude PTY 抢同一文件时能看到明确冲突。
- 4 个真实 Claude PTY 可以在同一个 viewer/API 下被区分和验收。

## 一句话结论

v0.6 的主题不是“更多 agent”，而是“多个 agent 同时工作时，AgentCall 仍然可信”。
