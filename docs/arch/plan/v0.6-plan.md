# AgentCall v0.6 Plan：并发可信底座 + Event-first Summary + Claude Code 协作

v0.6 综合三份输入：第一版 v0.6 的 event-first/session summary，第二版 v0.6 的多 Agent 并发验收，以及 `OPUS_Review.md` / `v0.6-plan-risk.md` 的质量门。

## 核心决策

- v0.6 直接采用 daemon 单写：Rust daemon 是 `events`、`file_claims`、`active_sessions`、`session_summary` 的唯一权威写者。
- Python 胶水可以保留为 CLI、测试和编排入口，但不得直接写共享状态；涉及共享状态的写操作必须调用 daemon/API。
- Codex 不默认读 PTY。Codex 读 `session_summary`；daemon 可以解析 PTY buffer/提示符来生成 `waiting_input`。
- `D:\guKimi` 是本机 Claude Code 启动配置，必须可配置，不作为发布硬编码默认。

## 优先级

```text
并发可信度 > 状态聚合 > PTY 体验 > ACP/PTY 路由扩展
```

如果多 Agent 并发下 claim 会双写、event id 会撞、summary 会误判，那么 PTY 画面再漂亮也只是把不可信系统可视化出来。

## 状态与 Claim 要求

- event id 必须单调且并发下不重复。
- 新事件统一 ISO-8601 时间戳；读取端兼容历史 `"unix:*"` 和坏行。
- `Write/Edit/MultiEdit/NotebookEdit` 才能 claim。
- `Read/Glob/Grep` 只 observe，不能产生 `file_claim.observed_write`。
- claim 增加 `workspace`、`project`、`expires_at`、`owner_pid`、`transcript_path`、`last_tool_name`。
- stale claim 采用 TTL 惰性回收为主，PID liveness 只做辅助提示。
- 缺 session id 的 hook 不能全部归并为 `unknown-session`；应进入 unmatched 或生成稳定 fallback id。

## Session Summary

`session_summary` 是 Codex 默认状态入口，最小状态枚举：

```text
starting
prompt_submitted
working
waiting_input
report_written
completed
failed
blocked
drifted
stale
```

最小字段：

```json
{
  "session": "androidpet-v11",
  "project": "Android_pet",
  "transport": "pty",
  "status": "working",
  "workspace": "E:\\Project\\Android_pet",
  "claude_workspace": "D:\\guKimi",
  "last_hook_at": "2026-06-03T03:28:52Z",
  "last_tool": "Edit",
  "claimed_files": ["app/src/main/Foo.kt"],
  "files_written": ["report_v1.1.md"],
  "report": null,
  "needs_user_input": false,
  "warnings": [],
  "conflicts": []
}
```

`waiting_input` 由 daemon 侧解析 PTY 或 hook notification 推导；Codex 只消费 summary，不直接刮屏。

## Claude Code 协作

依赖 Claude Code，但必须 bounded lifecycle，按顺序派工：

```text
CC-A 并发 failing tests
  -> CC-B daemon 单写 + claims 修复
       -> CC-C summary / API / MCP
       -> CC-D validators / OPUS P1/P2
```

- **CC-A**：只写 L0/L1/L2 failing tests，不改生产实现，交 `report_v0.6_concurrency_tests.md`。
- **CC-B**：实现 daemon 单写、event id、claim TTL、Read/write 语义，交 `report_v0.6_state_claims.md`。
- **CC-C**：实现 session summary、project sessions、runtime health，交 `report_v0.6_summary_api.md`。
- **CC-D**：修 report/schema/workspace/git diff/ACP timeout/config，交 `report_v0.6_validators.md`。

Codex 每轮先读 report 和 summary；无风险直接接受，不机械 review。只有 conflict、drift、failed、blocked、low confidence 才 review 或二次派工。

## Test Plan

### L0 Store/Claim 并发

- 使用真实 OS 进程，不用线程。
- 用 barrier 或可注入延迟强制竞争。
- N 进程抢同一文件，只允许一个 active owner。
- N 进程写不同文件，全部成功。
- 并发 append events，event id 无重复，NDJSON 无坏行。
- stale claim 可 TTL 回收。

### L1 Hook 并发

- 并发 `PreToolUse Write` 同文件稳定 deny。
- `PostToolUse Read` 不产生 `observed_write`。
- 缺 session id 不坍缩为统一身份。

### L2 模拟工作流

- 4 个 fake child 并发不同任务。
- 2 个 fake child 同文件冲突。
- child 漏报 changed_files 时由 validator 抓到。
- 无 report 不得判 completed。

### L3 真实 Claude PTY

- 2 个 Claude PTY 不同文件。
- 2 个 Claude PTY 同文件冲突。
- 4 个 Claude PTY 不同任务。
- 1 个 Claude 等待输入，summary 标记 `waiting_input`。
- Codex 只用 summary + report 验收。

## MCP/API 目标面

```text
agentcall_session_summary
agentcall_project_sessions
agentcall_unmatched_hooks
agentcall_runtime_health
agentcall_concurrency_probe
```

`agentcall_concurrency_probe` 只做诊断，不作为验收依据。真正验收必须落在 L0/L1/L2/L3 测试。

## OPUS 修复项

- 修 `task_status_from_report` 死逻辑。
- report 校验统一走 JSON schema。
- `_validate_changed_files_exist` 使用 child workspace + git diff/workspace validator。
- 移除 `executor-02` 魔法后缀。
- ACP client 落实 timeout，stdout/stderr 并发读取，超时 kill。
- 清理硬编码 `D:\guKimi`，改为 env/config 注入。

## 完成标准

- daemon 成为共享状态唯一写者。
- 同文件并发 claim 稳定冲突，不同文件并发稳定通过。
- event id 无重复，新事件 ISO-8601，旧事件读取容错。
- stale claim 可回收。
- `Read/Glob/Grep` 不再污染 write claim。
- `session_summary` 能表达 `working/completed/waiting_input/blocked/drifted/stale`。
- 4 个真实 Claude PTY 可在同一 daemon/API/viewer 下区分和验收。
