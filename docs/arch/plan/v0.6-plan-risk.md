# AgentCall v0.6 计划评审：风险与待决项

> 评审者: Claude Opus 4.8 · 日期: 2026-06-03
> 对象: v0.6 综合计划（并发可信 + Event-first + OPUS 质量门）与 `docs/arch/plan/v0.6-direction.md`

## 总评

方向对，优先级对，OPUS 覆盖完整——但有一个没拍板的根本岔路，会让 Phase 1/2 有返工风险。

这份计划最大的价值是把「先修可信底座，再做 summary，最后真实验收」的顺序定死，且 L0→L3 测试分层克制（真实 Claude 放最后）。问题集中在几个被一句话带过、实际却是整个 v0.6 成败所在的技术决策。

---

## 做对的地方（确认而非凑数）

- 优先级 `并发可信 > 聚合 > PTY` 正确，与 OPUS P0 一致。「把不可信系统可视化出来」的判断成立。
- L0「先写 failing test，不碰生产实现」是真 TDD，能锁死问题、防止 CC-B 自证。
- bounded CC 任务 + 强制 report，是拿项目自己的 SOP 吃自己的狗粮，立场一致。
- Read/Glob/Grep 只 observe 的诊断准确：现状 `observe_post_tool_use`（`claims.py:96` / `hook:269`）对任何带 `file_path` 的工具（含 Read）都会 append 一条 `file_claim.observed_write`，确需修。
- 非目标段写得好（不做 swarm / 不做每项目独立 daemon），挡住了范围蔓延。

---

## 关键风险 / 缺口（按影响排序）

### R1 [必须先拍板] 「短期双语锁」与「长期 daemon 单写」互斥，计划两个都要

这是最大隐患。计划短期 = Python + Rust hook + Rust daemon 共用同一套锁和原子写协议；长期 = 收敛到 daemon 单写。但：

- OPUS #1（双实现漂移）不会被「大家用同一套锁」修好，反而加重——现在要让两个 codebase 各自正确实现「跨进程锁 + 原子写 + 单调 id」并永远同步。
- 跨 Python 与 Rust 的进程级 advisory lock 在 Windows 上要靠 `LockFileEx`（Rust 侧 `fs2`）对齐 `msvcrt.locking`（Python 侧），这恰是最易错、最难测处。计划只写了「同一套锁」四个字。

**结论**：若终点是 daemon 单写，建议 v0.6 直接做，别留「长期」。否则 CC-B 会先造跨语言文件锁，Phase 2 测过、再在「长期」扔掉。要么明确「v0.6 即 daemon 单写，Python/Rust hook 都改成把写操作 POST 给 daemon」，要么诚实承认「v0.6 接受双语锁技术债，daemon 单写不在本版」。当前是模糊中间态。

### R2 [测试设计] 并发测试「能 pass」≠「修好了」——缺确定性复现

L0/L1 用「N 进程」抢同一文件方向对（必须真 OS 进程，不能用线程，GIL 会掩盖 race，计划未写明）。但 TOCTOU 是概率性的：没有同步屏障（barrier）强制两个 claim 在同一瞬间进入临界区，跑 100 次可能次次侥幸通过，给假绿灯。

CC-A 必须包含：要么可注入的延迟/屏障点，要么高迭代压力循环 + 失败即记录。「测试通过」要定义成「强制竞争下仍单 owner」，而非「跑了没崩」。

### R3 [没定方案] 单调 event id 替代方案缺失，且与 R1 绑死

「不能靠数行数」诊断对，但没给替代。

- 若保留全局顺序 `evt-000123`，必须有单一发号者 = daemon 单写（回到 R1）。
- 若接受 ULID/UUID + 本地计数，放弃「行数=id」便利，换来正确性。

这个选择决定双语锁方案是否可行，必须在 Phase 1 之前定，不能拖到 Phase 2 边写边定。

### R4 [机制缺失] `waiting_input` 可能与「Codex 不读 PTY」自相矛盾

summary 状态枚举 10 个，但计划没说 `drifted` / `waiting_input` / `blocked` 怎么被检测。`waiting_input` / `needs_user_input` 最危险：Claude Code 没有可靠「我在等输入」hook，判断 PTY child 卡在提示符上通常只能刮终端输出——这恰违反「Codex 不读 PTY、只看 summary」的核心主张。

要么找到可靠 hook/信号推导，要么承认该状态由 daemon 侧解析 PTY 得出（daemon 读、Codex 不读，可自洽，但须写明）。

### R5 [排序缺失] 四个 CC 任务没标依赖和并行性——对并发项目而言讽刺

真实依赖：**CC-A → CC-B →（CC-C ∥ CC-D）**

- CC-B 让 CC-A 的 failing test 变绿。
- CC-C 的 `session_summary` 读 CC-B 改完的 claim schema（`expires_at` / `owner_pid`），故 C 依赖 B 字段定稿。
- CC-D 的 git-diff validator 基本独立。

计划平铺四任务，无排序也无并行标注，建议在 task packet 中显式写依赖。

### R6 [稳健性] stale 回收的「谁来回收 + PID 探活」过于乐观

- 谁回收没定：惰性（下个读者把过期当空闲）vs daemon 后台 sweeper。惰性更简单、无状态，建议作主。
- `owner_pid` liveness 在 Windows 上不可靠：PID 会复用，跨用户查存活 racy。建议 TTL 为主、PID 仅辅助提示，计划现把两者并列当等价手段。

### R7 [次要] 缺格式迁移考虑

时间戳从 `"unix:N"` 改 ISO-8601、claim schema 加字段，会让现存 `events.ndjson` / `state/*.json` 变混格式。要么写迁移，要么读取端容错。计划改了格式但未提存量数据。

### R8 [次要] `concurrency_probe` 别变成自我安慰

探针报「锁已启用: yes」测的是配置不是正确性。当诊断工具可以，但验收必须落在 L0/L1 压力测试上，不能落在探针自检上。建议在计划里写死这条边界。

---

## 三条最该改的

1. **先回答再开工**：v0.6 到底是「daemon 单写」还是「双语共用锁」？倾向前者——一次性解决 OPUS #1+#2+#3 三条 P0，而双语锁只解决并发不解决漂移。把决定写进 Phase 0。
2. **给 CC-A 并发测试加「强制竞争」契约**：必须有屏障或高压循环，定义「通过 = 强制同时进入临界区仍单 owner」。
3. **把 summary 状态推导机制补进计划**，尤其 `waiting_input` 的数据来源，否则它会是 v0.6 最先翻车的字段。

---

## 建议补进 v0.6 计划的两段

### Phase 0：决策门（开工前必须关闭）

- 写者模型：daemon 单写 / 双语共用锁，二选一并写明理由。
- event id 策略：全局顺序（需单发号者）/ ULID。与写者模型一致。
- `waiting_input` 数据源：hook 信号 / daemon 解析 PTY。
- stale 回收策略：惰性为主 + TTL，PID 仅辅助。

### CC 依赖图

```text
CC-A (并发 failing tests)
   └─> CC-B (状态写入 + claim 修复)
          ├─> CC-C (summary / API / MCP)   # 依赖 B 的 claim schema
          └─> CC-D (validator + OPUS P1/P2) # 基本独立，可与 C 并行
```
