# AgentCall v4.2 Plan：Readable TUI Cleanup + Low-Friction Session Control

## Summary

v4.2 聚焦 PTY wrapper 的体验问题：Claude Code TUI 里有大量装饰性词、重绘残留、状态短语和权限菜单噪声，Codex 读取这些内容时上下文压力大、误判多、耐心差。最新日志还暴露出更关键的问题：`select_option` 已经可用，但 **policy deny / permission menu 没有被抽象成可恢复状态**，导致 Codex 在菜单、deny、等待、重试之间打转。v4.2 不改变 PTY-first 主线，不重启 ACP，也不把 TUI 提升为状态权威；本轮目标是把 TUI 和 policy 反馈变成更短、更结构化、更适合 LLM 接管的 wrapper 信号。

核心判断：

- **状态权威仍是 hook / daemon / route / report**。
- **TUI 只负责可读摘要、菜单识别、自由文本问题和弱提示**。
- **重复 policy deny 必须升格为 attention，不得继续显示为健康 working**。
- **Claude Code 默认不应是纯 read-only worker，而应是 bounded-write worker：能写，但只能写 route 授权路径、session scratch 和 report path。**
- **权限提示不是问题，裸 TUI 菜单才是问题；wrapper 要说明这是什么菜单、有哪些选项、选项语义是什么。**
- **Codex 默认读 `llm_summary`，只有低置信度或人工排障时展开 clean/raw tail**。
- **session_send 默认不盲打正在工作的 Claude，而是通过 hook additionalContext 队列注入；权限菜单走 `select_option`；强行打断必须显式 `interrupt`。**

## 现状

已经完成的基础：

- `terminal.rs` 已有流式 UTF-8 decoder，避免中文跨 chunk 乱码。
- `clean_terminal_text` 已能去 ANSI、压空行、保留部分 cursor separator。
- `session_summary` 已拆出 `liveness_status`、`attention_status`、`confidence`、`patience_hint`。
- `session_send` 已支持：
  - `select_option`：处理权限/菜单数字选项。
  - `interrupt`：仅用于跑偏或立即回收。
  - 工作中自然语言输入改为 queued supervisor instruction，通过 `PostToolBatch` / context hook 注入。
- Board 已优先展示 compact sessions / attention / runtime health。

仍然糟糕的地方：

- `headline/current_task/last_error/waiting_input` 对 TUI 文字依赖重，容易吃到装饰词。
- clean tail 仍含许多对人有视觉意义、对 LLM 没价值的短语。
- 权限菜单、计划确认菜单、动态 workflow 菜单虽然能识别一部分，但不是统一的 typed interaction model。
- policy deny 被记录为事件，但重复 deny 没有进入 `attention_status`，Codex 会继续按 `working + inside_patience_window` 等待。
- Codex 常把“读一个东西”的任务派成只读，但 Claude Code 真实工作经常需要临时脚本、索引、缓存、抽取结果和报告文件；纯 read-only 会制造不必要的 permission friction。
- deny 后没有自动纠偏上下文，Claude 可能机械重试同一条被拒绝命令。
- `clean_output` 与 `llm_summary` 之间缺少一个稳定的中间层：`semantic_output` / `tui_signals`。
- Codex 看 session 时仍可能被大段 tail 吸走注意力，重复追问或误判 Claude “卡住”。

## TUI 装饰词分类

先不要只写一堆正则。v4.2 应把 TUI 文本分成几类，每类处理方式不同。

### 1. 可删除装饰

对状态判断没有价值，默认从 `semantic_output` 删除：

- 思考/等待装饰：`Thinking...`、`Cogitated for ...`、`Sautéed for ...`、`✻ ...`
- recaps 提示：`recap:`、`disable recaps in /config`
- help/footer：`? for shortcuts`、`Esc to cancel`、`Tab to amend` 等底部提示
- auto-update 噪声：`Auto-update failed` 只进入 warning，不作为 task error
- 纯边框/表格残留：box drawing、重复分隔线、孤立 prompt marker

### 2. 可压缩进程状态

保留为短字段，不进入大段文本：

- token/context 用量：进入 `tokens`、`context_used`
- mode：进入 `mode`
- route/workflow：进入 `workflow_status`、`pty_workflow`
- report 生成提示：进入 `report_ready` / `reports`

### 3. 交互提示

必须保留，但要结构化：

- 权限菜单：`interaction.kind=permission_menu`
- 动态 workflow 菜单：`interaction.kind=workflow_menu`
- plan approve/start auto 菜单：`interaction.kind=plan_menu`
- Claude 自由文本提问：`interaction.kind=question`

这类内容应进入 `llm_summary.interaction`，并决定 `attention_status=waiting_input|needs_permission`。Codex 默认读结构化字段，而不是自己从 tail 里猜。

### 4. 任务内容

真正应该给 Codex 看的内容：

- 当前正在读/改的文件
- 失败原因
- 需要主管决策的问题
- 产出的 report 路径
- 最后一段有意义的自然语言说明

这些进入 `semantic_output` 和 `headline/current_task/last_error`。

## Key Changes

### P0：默认 bounded-write，而不是默认 read-only

AgentCall 对 Claude Code 的默认信任模型应从“只能读”改成“能干活，但写入有边界”。

当前代码已有 `allowed_paths` 与 `containment.mode = prompt_only | enforced`。因此 v4.2 不是从零新增 containment，而是扩展为更明确的三态：

- `prompt_only`：无强制 path policy，仅在 prompt 中提示边界。
- `enforced_readonly_bash`：写工具按路径强制，Bash 仍只允许只读命令。
- `bounded_write`：写工具按路径强制，并允许有限、可解析的 Bash 写入 scratch/report。

首版 v4.2 默认采用 **`enforced_readonly_bash`**，不是一步到位放开任意 Bash 写入。这是为了消解当前代码冲突：现有 `hooks.rs` 在 enforced 模式下对非只读 Bash 是整条 deny，不能仅靠创建 scratch 目录解决。

默认 route 策略：

- route 启动时自动生成 session 专属 scratch：
  - `.agentcall/workspaces/<session_name>/`
- route 默认把以下路径加入 writable allowed set：
  - `report_path`
  - session scratch
  - 调用方显式传入的 `allowed_paths`
- `Write/Edit/MultiEdit` 只能写 writable allowed set 内路径。
- Bash 首版仍默认只读；非只读 Bash 被 deny，但 deny summary 应区分“合理缺少可写脚手架”与“危险越界”。
- Claude 需要临时脚本时，首版推荐写到 session scratch 的脚本文件，再通过只读/白名单执行策略执行；如果执行会产生写入，必须进入 P0b 的有限 Bash 写入判定。
- 只读审计任务必须显式声明 `read_only=true` 或不给 writable allowed set；它不应是默认。
- 如果 Claude 需要新增写入路径，应报告 blocker；Codex 再扩 allowed_paths 或重新派工。

目标不是放宽到“随便写”，而是给 worker 一个可工作的沙盒。Claude Code 是打工的人，不是只能看资料的审计机器人。

建议 route/summary 暴露：

```json
{
  "containment": {
    "mode": "enforced_readonly_bash",
    "read_only": false,
    "writable_paths": [
      "docs/ReportNReview/Report_x.md",
      ".agentcall/workspaces/session-x/"
    ],
    "scratch_path": ".agentcall/workspaces/session-x/",
    "bash_write_policy": "readonly_only"
  }
}
```

#### P0b：Bash 写入路径级判定的设计选项

Opus review 指出的核心冲突必须先拍板：如果要让 Bash 写 scratch，必须从“非只读 Bash 整条 deny”升级为“有限 Bash 写目标解析 + path policy”。

v4.2 采用分阶段策略：

1. **首版安全落地：Bash 仍只读，写入走 Write/Edit/MultiEdit。**
   - 优点：与现有 `hooks.rs` 行为一致，风险最低。
   - 缺点：Claude 用 shell 生成临时文件仍会被拒，需要它改用写工具创建脚本/报告。
2. **后续可选：有限 Bash 写入白名单。**
   - 只允许可解析且目标路径明确的模式，例如：
     - `python script.py --out <scratch/file>`
     - `node script.js --out <scratch/file>`
     - `command > <scratch/file>`
     - `command | tee <scratch/file>`
   - 所有解析出的写目标必须位于 writable allowed set。
   - 任何无法可靠解析的 shell 特性直接 deny：
     - 多段 `; && ||`
     - subshell / command substitution
     - wildcard 写目标
     - 删除、移动、覆盖 git 状态的 destructive 命令
   - 这一路线必须配测试，不作为首版 P0 的默认能力。

首版 acceptance 不要求 Bash 写 scratch 成功，只要求：

- 写工具可写 scratch/report/allowed_paths。
- 非只读 Bash 被 deny 后，summary 不再伪装成健康 working。
- deny summary 给出 `suggested_allowed_paths` 或 “改用 Write/Edit 写入 scratch 脚本/报告” 的建议。

### P1：policy deny loop 升级为可恢复 attention

当前真实问题不是 claim 冲突，而是策略拒绝被当成普通工作流噪声。例如同一 session 反复尝试：

```text
git clone https://github.com/Holic75/KingmakerRebalance.git --depth 1
```

并反复收到：

```text
PTY path policy denies non-read-only bash when allowed_paths are enforced
```

这时 summary 不应继续显示：

```text
liveness_status=working
attention_status=none
patience_status=inside_patience_window
```

v4.2 新增 policy denial tracker：

- 按 `wrapper_session + tool + normalized_command_or_path + deny_reason` 聚合。
- `normalized_command_or_path` 首版规则：
  - Write/Edit/MultiEdit：使用 normalized file path。
  - Bash：移除连续空白、统一路径分隔符、保留命令主干和显式目标路径；不尝试语义等价归一化。
  - 无法提取稳定目标时，使用 command hash，避免过度合并不同命令。
- 同一组合连续出现 2-3 次，标记：
  - `liveness_status=blocked`
  - `attention_status=blocked_by_policy`
  - `patience_status=attention_required`
  - `status_source=policy`
- 默认阈值为 2 次连续 deny；后续可配置。
- counter reset 条件：
  - 同一 wrapper 发生成功 tool event。
  - 命令/path hash 改变。
  - route 更新 allowed_paths / containment。
  - session stop/end。
- summary 给出明确建议：
  - 不要继续等待。
  - 区分 deny 类型：
    - `dangerous_or_out_of_scope`：危险命令、越界写入、跨 owner 文件，建议改任务 / interrupt / stop。
    - `missing_scratch_or_report_path`：任务合理但缺少可写 scratch/report，建议补授权路径。
    - `needs_supervisor_decision`：需要人判断是否扩 allowed_paths。
  - 需要 supervisor 改 allowed_paths / 改任务 / 要求 Claude 报告 blocker / interrupt / stop。
- Board attention 展示 `blocked_by_policy`，并显示最后一次 deny reason 与重复次数。

建议新增字段：

```json
{
  "policy_block": {
    "active": true,
    "tool": "Bash",
    "command": "git clone ...",
    "reason": "PTY path policy denies non-read-only bash when allowed_paths are enforced",
    "repeat_count": 3,
    "category": "dangerous_or_out_of_scope",
    "recommended_action": "interrupt_or_send_blocker_instruction",
    "suggested_allowed_paths": []
  }
}
```

如果是缺少 scratch/report 授权，建议字段应是：

```json
{
  "policy_block": {
    "active": true,
    "category": "missing_scratch_or_report_path",
    "recommended_action": "extend_allowed_paths",
    "suggested_allowed_paths": [
      ".agentcall/workspaces/<session_name>/",
      "docs/ReportNReview/<report>.md"
    ]
  }
}
```

### P2：deny 后自动注入纠偏上下文

如果 policy deny loop 触发，下一次 hook additionalContext 应自动注入短纠偏提示，避免 Claude 继续撞墙。

例：

```text
AgentCall policy denied your last Bash command repeatedly.
Do not retry the same command.
Use existing references, ask for supervisor permission, or report this as a blocker.
Denied command: git clone ...
Reason: PTY path policy denies non-read-only bash when allowed_paths are enforced.
```

要求：

- 只注入一次或按 deny hash 去重，避免刷屏。
- 注入记录进入 events：`policy_denial.guidance_injected`。
- 不自动放宽权限，不自动修改 allowed_paths。
- 如果 deny category 是 `missing_scratch_or_report_path`，提示 Claude 报告需要的路径，而不是继续重试。
- 如果 deny category 是 `dangerous_or_out_of_scope`，提示 Claude 停止该动作并报告 blocker。
- 如果 session 没有 `PostToolBatch`，summary 显示纠偏提示仍 pending。

### P3：新增 TUI cleaner pipeline

在 daemon 内新增一个分层 pipeline：

```text
raw_output bytes
  -> decoded_output
  -> clean_output       # UTF-8 + ANSI + viewport/tail 基础清理
  -> semantic_output    # 删除装饰、压缩重复、保留任务语义
  -> tui_signals        # menu/question/report/token/mode weak signals
  -> llm_summary
```

建议新增模块或扩展 `terminal.rs`：

- `clean_terminal_text`：继续做基础清理。
- `semantic_terminal_text`：删除装饰词和重复 UI 行。
- `extract_tui_signals`：输出结构化弱信号，不直接改权威状态。

### P4：装饰词规则要可测试、可解释

不要把所有正则散在 `summary.rs`。

新增规则表：

- `drop_line_patterns`
- `compress_line_patterns`
- `warning_patterns`
- `interaction_patterns`

每条规则应有：

- 名称
- 匹配类型
- 输出动作：drop / compress / warning / signal
- 测试 fixture

首版规则少一点，但必须覆盖当前真实痛点：

- `Cogitated/Sautéed/Thinking` 思考装饰
- recap/config 提示
- shortcut/footer
- auto-update failed warning
- dynamic workflow permission menu
- plan approve menu
- Claude permission prompt

### P5：`llm_summary` 新增字段

新增或稳定以下字段：

```json
{
  "semantic_tail": "...",
  "interaction": {
    "kind": "permission_menu|workflow_menu|plan_menu|question|none",
    "recommended_action": "select_option|send|interrupt|wait",
    "options": [
      {"index": 1, "label": "Yes, run it", "semantic": "approve"},
      {"index": 2, "label": "View raw script", "semantic": "inspect"},
      {"index": 3, "label": "No", "semantic": "deny"}
    ],
    "text": "..."
  },
  "policy_block": null,
  "containment": {
    "mode": "enforced_readonly_bash|bounded_write",
    "read_only": false,
    "writable_paths": ["..."],
    "scratch_path": "...",
    "bash_write_policy": "readonly_only|limited_path_checked"
  },
  "output_layers": {
    "raw_available": true,
    "clean_tail_lines": 80,
    "semantic_tail_lines": 30
  }
}
```

`interaction` 是 `tui_signals` 的裁决投影。MCP 默认只返回 `interaction`，不返回完整 `tui_signals`；只有 debug 或低置信度时才展开 `tui_signals`。这样 Codex 不需要同时读两套相似字段。

兼容字段 `headline/current_task/last_error` 继续保留，但来源应优先使用 `semantic_output`。

### P6：MCP 默认返回更小

`agentcall_session` 默认只返回：

- summary 基础字段
- `semantic_tail` 的短尾部
- `interaction`
- `patience_hint`
- warnings

只有 `include=["clean_tail"]` 时才返回当前 clean tail。raw output 继续只用于 debug，不作为 MCP 默认返回。

### P7：权限菜单和一般菜单包装

现有 `select_option` 已经能处理单数字，但菜单识别还偏散。

v4.2 统一：

- `interaction.kind=permission_menu`：必须 `select_option`，禁止自然语言 send，并列出每个选项的 label / semantic。
- `interaction.kind=workflow_menu`：允许 `select_option`，并在返回里说明选择含义，例如 approve / inspect / deny。
- `interaction.kind=plan_menu`：`approve_plan/start_auto` 映射到 option 1，但 summary 应清楚显示这是 plan approval。
- `interaction.kind=question`：可以 `send`，但要求 Codex 用短答，避免灌大段新任务。
- 新增语义辅助：允许 `agentcall_session_send(action="select_option", text="approve")` 作为后续方向，但首版可只在 summary 暴露 semantic，实际仍发数字。

### P8：queued supervisor instruction 可观测性

当前 queued 注入已经存在，但反馈不够清楚。v4.2 增加：

- `pending_supervisor_instructions` 显示文本摘要 hash / action / queued_at。
- `last_supervisor_instruction_injected_at` 显示注入事件。
- 如果 session 长时间没有 `PostToolBatch`，summary 给出明确 warning：
  - hook 没装全
  - worker 需要重启
  - 当前输入不会被听见

### P9：Board 降噪

Board 增加 compact interaction column：

- blocked_by_policy
- permission menu
- workflow menu
- queued instruction pending
- stale without progress
- low confidence

不要把普通 idle / Stop 放进 attention；不要因为装饰词变化制造 attention。

### P10：scratch 生命周期

session scratch 默认位于 `.agentcall/workspaces/<session_name>/`，必须有清理策略，避免长期堆积：

- session active/running 时保留。
- session completed/stopped 后保留 24 小时，方便复盘。
- `python agentcall.py doctor` 可提示 stale scratch 数量。
- 后续新增 `python agentcall.py cleanup` 或 daemon cleanup endpoint，但 v4.2 首版只要求记录 `created_at/session_name/route_id`，不自动删除。

## Non-goals

- 不恢复 ACP。
- 不把 TUI 作为状态权威。
- 不靠 TUI 装饰词判断任务完成。
- 不自动 kill Claude Code。
- 不实现 Claude session 自动续接/恢复。
- 不让 Codex 默认读取 raw terminal。
- 不把所有 Claude Code UI 文案都做成无限正则库。
- 不让多个 Codex 同时控制同一个 Claude worker。
- 不用 cwd/PID/窗口标题猜 wrapper binding。
- 不因 policy deny 自动放宽权限。
- 不把 `git clone`、写文件、外部网络访问等 denied action 静默改成 allow。
- 不把 bounded-write 理解成无边界写入；所有写入必须被 path policy 约束。
- 不让 Bash 通过重定向、临时目录或 shell trick 绕过 writable allowed set。
- 首版不承诺解析任意 shell；不可解析 Bash 写入保持 deny。
- 不在 v4.2 首版自动删除 scratch 目录。

### Unit fixtures

新增 fixtures 覆盖：

- 中文跨 chunk + TUI 装饰词混合。
- Cogitated/Sautéed/recap/shortcut/footer 被删除。
- Auto-update failed 进入 warning，但不进入 last_error。
- dynamic workflow 菜单识别为 `workflow_menu`。
- permission prompt 识别为 `permission_menu`。
- plan approve menu 识别为 `plan_menu`。
- 自由文本问题识别为 `question`。
- 无意义 TUI 噪声不生成乐观 headline。
- 连续相同 policy deny 2-3 次后，summary 进入 `blocked_by_policy`。
- policy deny loop 不再显示 `patience_status=inside_patience_window`。

- Python smoke 只保留端到端 hook/daemon 行为，不承担复杂策略判断。

### MCP regression

- `agentcall_session` 默认不返回大段 clean output。
- `include=["clean_tail"]` 才返回 clean tail。
- `select_option` 只允许菜单态。
- summary 暴露 `interaction.options[].semantic`，Codex 不需要猜数字含义。
- `send` 在 `needs_permission` 下返回 blocked hint。
- `send` 在 `blocked_by_policy` 下返回不要等待/改任务/interrupt 的 hint。
- `blocked_by_policy` summary 会区分危险越界和缺少可写 scratch/report 授权。
- working + no attention 时自然语言进入 queued instruction，而不是盲打。
- queued instruction 注入后 summary 计数下降，并记录 injected 时间。

### Real smoke

- 启动一个 PTY worker，触发权限菜单，确认 summary 给出 `interaction.kind=permission_menu`。
- 启动一个动态 workflow 场景，确认 `select_option("1")` 能走通。
- 启动一个 bounded-write PTY route，确认 Claude 通过 Write/Edit/MultiEdit 可写 session scratch/report_path。
- 启动一个 allowed_paths 限制任务，让 Claude 重复尝试越界 Bash，确认进入 `blocked_by_policy`。
- 启动一个显式 read-only route，确认没有自动 writable scratch。
- 触发非只读 Bash 写 scratch，首版应 deny 并给出“改用写工具或请求 limited Bash policy”的建议。
- 工作中发送 `request_report`，确认先 queued，下一次 `PostToolBatch` 后注入。
- 检查 Board：attention 只显示真正需要介入的会话。

## Implementation Order

1. **P0**：实现 bounded-write 默认 route containment：session scratch + report_path + explicit allowed_paths；首版 Bash 仍 `readonly_only`。
2. **P1/P2**：实现 policy denial tracker 与 deny guidance injection，先阻止“健康 working 假象”。
3. **P3/P4**：抽出 `semantic_terminal_text` 和 `extract_tui_signals`，先做 fixture 测试。
4. **P5**：把 `session_summary` 的 headline/current_task/last_error 改用 semantic output，并新增 `interaction` / `policy_block` / `containment`。
5. **P6/P7**：让 MCP 默认返回更小，并让 `session_send` 使用统一 interaction / policy 判断。
6. **P8**：完善 queued instruction 可观测字段。
7. **P9/P10**：Board 展示 blocked_by_policy、interaction 和 queued pending；scratch 记录生命周期元数据。
8. 跑 `python agentcall.py release-check`，再做真实 PTY smoke。

## Acceptance Criteria

- Codex 默认读取 session 时，内容明显短于 clean tail，且没有思考装饰/recap/footer 噪声。
- 权限菜单、workflow 菜单、plan 菜单能被结构化识别。
- 默认 Claude Code worker 是 bounded-write：能通过写工具写 scratch/report/allowed_paths，不能越界写。
- policy deny loop 能被识别为 `blocked_by_policy`，不会继续提示 Codex 耐心等待。
- policy deny summary 能区分危险越界与合理缺授权，并给出 suggested_allowed_paths。
- deny 后能给 Claude 注入一次纠偏上下文，避免机械重试。
- `send/select_option/interrupt` 的错误提示能直接告诉 Codex该用哪个动作。
- 工作中自然语言不会被误以为已经提交给 Claude；queued/injected 状态可见。
- TUI regex 失配只降低 confidence 或进入 warning，不覆盖 hook/daemon/report 状态。
- `python agentcall.py release-check` 通过。
