# AgentCall v6.0 Build And Pressure Test Issue Report

日期：2026-06-11

## Summary

本文记录 v6.0 构建实现期与后续压力测试期暴露的问题。冻结计划 `docs/v6.0-code-plan.md` 未修改；所有新证据写入报告而不是回改 plan。

构建期主要问题：

1. Codex 宿主 MCP transport 曾长期 `Transport closed`，但 daemon HTTP `/api/mcp/call` 可用。
2. token 鉴权开启后，开发诊断脚本未带 `X-AgentCall-Token`，导致 `daemon-health` 误报 401。
3. route target workspace 与 Claude Code process cwd 分离后，相对 `report_path` 容易被解析到错误目录。
4. `prompt_submit_attempted` 路由状态未被 `UserPromptSubmit` ack 逻辑识别，导致已提交 prompt 仍可能显示卡在 prompt gate。
5. report 已经写在 target workspace，但 `accept_report_for_session` 曾按 daemon workspace 解析相对路径，导致错误的 `report_missing_or_empty`。

压力测试主要问题：

1. route 初始 prompt 未被 Claude Code 自动提交，6/6 进入 `prompt_missing`。
2. `pending_prompt_commit_sent` 的语义对监督者不够直观，容易被误读成“任务已提交/已完成”。
3. report 文件已经写出，但 board/session 在 accept 前仍显示 `waiting_input` / `report.ready=false`。
4. worker 仍可能尝试被 policy 拒绝的 Bash 写入路径，需要靠 wrapper policy 纠偏。

其中第 3 点已定位为代码 bug，并已补测试与修复，但本报告生成时尚未重启 daemon 做运行态复验。

## Build-Time Issues

### Build Issue 1：Codex 宿主 MCP transport 曾处于 Transport closed

#### 现象

在 v6.0 实现和初次自测期间，`mcp__agentcall` 命名空间反复返回：

```text
Transport closed
```

但同一时间：

- daemon 进程存在。
- `python agentcall.py daemon-health` 可返回 `status=ok`。
- 直接调用 daemon HTTP `/api/mcp/call` 可以完成 `agentcall_route` / `agentcall_report` smoke。

#### 基础分析

这是 Codex 宿主侧 MCP transport 没有重新绑定新 MCP 进程，不是 daemon 控制面不可用。v6.0 实现过程中临时使用 `/api/mcp/call` 作为旁路，避免把宿主 transport 问题误判成 daemon 或 route 问题。

#### 影响

- 不影响 daemon 本身。
- 会影响当前 Codex 线程内置 MCP 工具是否可直接调用。
- 容易误导监督者以为 AgentCall 整体不可用。

#### 处置

- 在 closure 报告里明确区分：
  - daemon 可用
  - HTTP/MCP control plane 可用
  - Codex 宿主 MCP transport 未重绑
- 用户重启 Codex 后，本轮压力测试中 MCP transport 已恢复，`agentcall_daemon` / `agentcall_board` / `agentcall_route` 均可用。

### Build Issue 2：daemon token 开启后开发诊断脚本误报 401

#### 现象

`/api/*` 开启 token 鉴权后，开发脚本调用 daemon health 时未带 `X-AgentCall-Token`，导致本地诊断出现 401。

#### 基础分析

这是脚本层缺口，不是 daemon 鉴权 bug。daemon 需要 token 是正确行为；`agentcall_dev.py` 应从本地 env 或 `config/agentcall.local.json` 读取 token，并只用于本地诊断请求头。

#### 处置

已修改：

- `scripts/agentcall_dev.py`
  - 新增 `daemon_auth_headers(root)`。
  - `http_json(...)` 支持 headers。
  - `daemon-health` / `doctor` 自动带 `X-AgentCall-Token`。

#### 验证

```powershell
python agentcall.py daemon-health
```

可返回 `status=ok`，不再误报 401。

### Build Issue 3：relative report_path 与 forced Claude cwd 产生歧义

#### 现象

daemon 按配置强制 Claude Code process cwd 为：

```text
D:\guKimi
```

但 route `workspace` 是任务目标，例如：

```text
E:\Project\AgentCall\.agentcall\workspaces\v6-parallel-...
```

早期 handoff 只要求 worker 写：

```text
.agentcall/reports/report.md
```

结果 Claude 可能按 `D:\guKimi` 解释相对路径，把报告写到错误位置，或者至少让监督者难以判断相对路径应归属哪个 workspace。

#### 基础分析

这是 v6.0 构建中最核心的路径语义问题之一：route `workspace` 是任务目标，不能覆盖 Claude process cwd；因此所有交付物路径必须明确“相对谁”。

#### 处置

已修改：

- route handoff 增加 `report_abs_path`。
- handoff 明确写：
  - `report_path` 相对 target workspace。
  - 因为 Claude cwd 可能不同，最终报告必须写入 `report_abs_path`。
- containment 文本增加 `writable_abs`，将 display path 映射到绝对路径。

已加测试：

- `pty_handoff_names_absolute_report_path_when_cwd_differs`

### Build Issue 4：accept_report_for_session 曾按 daemon workspace 解析 report_path

#### 现象

三并发 v6 smoke 中，三个 worker 的报告实际已经存在：

```text
E:\Project\AgentCall\.agentcall\workspaces\v6-parallel-...\ .agentcall\reports\...
```

但 `agentcall_report(action=accept, session_id=...)` 返回：

```json
{
  "ok": false,
  "status": "report_missing_or_empty",
  "validation": {
    "exists": false
  }
}
```

#### 基础分析

`accept_report_for_session` 使用 daemon workspace 拼接相对 `report_path`：

```text
E:\Project\AgentCall + .agentcall/reports/...
```

但真实报告在 route target workspace 下：

```text
E:\Project\AgentCall\.agentcall\workspaces\... + .agentcall/reports/...
```

#### 处置

已修改：

- `crates/agentcall-daemon/src/mcp.rs`
  - `accept_report_for_session` 改为优先读取 route `workspace`，并按 route workspace 解析相对 `report_path`。
  - 返回增加 `report_abs_path`，方便诊断。

已加测试：

- `report_accept_resolves_relative_report_path_against_route_workspace`

#### 结果

同一批真实三并发报告复验后均返回：

```text
ok=true
status=accepted
confidence=high
```

### Build Issue 5：prompt_submit_attempted 未被 UserPromptSubmit ack 识别

#### 现象

route 初始 prompt 未自动提交时，监督者调用：

```text
agentcall_session_send(action=submit_pending_prompt)
```

route 状态会更新为：

```text
prompt_submit_attempted
```

但后续即使 hook 观察到 `UserPromptSubmit`，旧逻辑也可能不把这个 route 状态更新成 `prompt_submitted`。

#### 基础分析

`mark_route_prompt_submitted_locked` 的可 ack 状态白名单漏掉了：

```text
prompt_submit_attempted
```

这会制造一种很糟的假象：实际 prompt 已提交，worker 已经开始工作，但 projection 仍可能看起来卡在 prompt gate。

#### 处置

已修改：

- `crates/agentcall-daemon/src/hooks.rs`
  - `mark_route_prompt_submitted_locked` 允许 `prompt_submit_attempted` 被 `UserPromptSubmit` ack。
  - ack 时同步写入 `task_started=true`、`state=user_prompt_submitted`。

已加测试：

- `user_prompt_submit_marks_submit_attempted_route_prompt_submitted`

### Build Issue 6：report_ready 优先级曾低于 stale prompt gate

#### 现象

早期单 worker smoke 中，worker 已经写出报告并可 accept，但 session summary 仍可能显示：

```text
state=prompt_missing
```

#### 基础分析

worker state 汇总时，过期 prompt gate 状态优先级高于 report artifact / report-ready 信号。对于监督者来说，这是错误的：如果报告已可验收，`report_ready` 应压过旧 prompt gate。

#### 处置

已修改：

- `worker_state.rs`
  - report-ready 相关信号优先于 stale prompt gate。

#### 结果

后续单 worker smoke 可显示 report-ready，并完成：

```text
accept -> high
stop -> lease released
```

### Build Issue 7：hook payload 原始日志过重

#### 现象

v6.0 构建前已有多轮日志膨胀问题，尤其 `PostToolUse` / `PostToolBatch` 容易携带大段工具输入/输出或敏感文本。v6.0 实现过程中继续观察到事件文件过重会拖慢 board/session 排查。

#### 基础分析

hook 原始 payload 不应完整进入事件流。AgentCall 的默认 projection 应依赖结构化摘要，不应被大段 terminal/tool output 拖慢。

#### 处置

已修改：

- `crates/agentcall-daemon/src/hooks.rs`
  - 新增 `redact_hook_payload(...)`。
  - 对长字符串、命令、prompt、content、token/secret/password/API key 类字段进行 redaction。
  - event 中保存 `safe_raw` 而不是完整 raw payload。

#### 风险

redaction 需要平衡可诊断性和安全/性能。当前策略是保守截断，后续如果排障信息不足，需要增加结构化白名单字段，而不是回到大 raw payload。

## Issue 1：route 初始 prompt 全量进入 prompt_missing

### 现象

6 个 route 启动后均先返回：

```json
{
  "state": "prompt_pending",
  "next_action": "wait"
}
```

等待 18 秒后，board 显示 6/6：

```json
{
  "state": "prompt_missing",
  "next_action": "submit_pending_prompt",
  "why": "Route prompt was written to the PTY but UserPromptSubmit was not observed before the ack deadline."
}
```

执行 `agentcall_session_send(action=submit_pending_prompt)` 后，6/6 返回：

```json
{
  "ok": true,
  "status": "pending_prompt_commit_sent",
  "awaiting": "UserPromptSubmit"
}
```

之后 hook 事件显示 `hook.UserPromptSubmit` 被观察到，worker 开始读 handoff 并写报告。

### 基础分析

这说明 route 已经把 handoff prompt 写进 PTY，但初始提交动作没有可靠触发 Claude Code 的 `UserPromptSubmit` hook。当前 `submit_pending_prompt` 是有效自愈路径：它通过 actor 往 PTY 发送一次提交动作，随后 hook ack 出现，任务可以继续。

关键点：

- `pending_prompt_commit_sent` 不是完成态。
- 它只表示 daemon 已经把“提交当前待提交 prompt”的动作发进 PTY。
- 真正的成功条件是随后出现 `hook.UserPromptSubmit`，并且 prompt gate 转为 `user_prompt_submitted`。

### 风险

如果 Codex 误把 `pending_prompt_commit_sent` 当成任务完成，会过早等待报告或发送自然语言，造成控制混乱。

### 建议

- 保留 `submit_pending_prompt`，因为它是当前可用自愈路径。
- 改善返回命名或补充字段，例如：
  - `status=prompt_commit_signal_sent`
  - `awaiting_hook=UserPromptSubmit`
  - `not_completed=true`
- route 初始提交可靠性仍需要后续单独追踪：为什么写 prompt 后没有自然 Enter/commit。

## Issue 2：waiting_input 缺少“到底在等什么”

### 现象

`submit_pending_prompt` 之后，6 个 worker 均进入过：

```json
{
  "state": "waiting_input",
  "why": "Claude Code is waiting for supervisor input.",
  "next_action": "continue"
}
```

但 compact summary 没有解释 Claude 在等什么。

### 基础分析

事件流显示 worker 实际已经读 handoff、写报告、触发 Stop/SubagentStop/Notification。也就是说这里的 `waiting_input` 不是“任务缺必要答案”，更像是 Claude Code turn 结束后的普通 idle/waiting 状态。

当前抽象过粗，把几类不同等待混在一个 `waiting_input`：

- 真正需要回答的问题。
- 权限菜单。
- turn-end idle。
- report 已写出后的等待。
- `SubagentStop` 之后的 checkpoint-like 等待。

### 风险

Codex 会被引导去 `continue`，但如果报告已经写出，正确动作应该是 `accept_report` 或 `stop_worker`。这会增加无效交互。

### 建议

- `waiting_input` 需要带 `pending_interaction.kind` 或更具体状态。
- 至少区分：
  - `idle_after_turn`
  - `question_waiting_answer`
  - `permission_menu`
  - `report_written_waiting_accept`
  - `checkpoint_due`
- 对于已观察到 report write 的 session，`report_ready` 应覆盖普通 `waiting_input`。

## Issue 3：report 已写出但 board/session 未转 report_ready

### 现象

压力测试中，6 个报告文件均已存在，`agentcall_report(action=accept)` 均返回 high：

```json
{
  "ok": true,
  "status": "accepted",
  "confidence": "high",
  "validation": {
    "exists": true,
    "non_empty": true,
    "matched_route_report_path": true,
    "daemon_observed_write": false
  }
}
```

但 accept 前 board 仍显示：

```json
{
  "state": "waiting_input",
  "report": {
    "ready": false
  }
}
```

事件证据显示 hook 确实看到了 Write：

```text
hook.PreToolUse  Write started   target=<target_workspace>/.agentcall/reports/...
hook.PostToolUse Write completed target=<target_workspace>/.agentcall/reports/...
```

### 基础分析

这是明确代码 bug。原因与前一轮修复过的 `accept_report_for_session` 类似：

- route 的 `report_path` 是相对路径。
- worker 的 target workspace 是 `.agentcall/stress/...`。
- Claude Code process cwd 是 `D:\guKimi`。
- hook 事件里的 Write target 是 target workspace 下的绝对路径。
- 旧 `report_write_for_wrapper` 用 daemon workspace 解析相对 `report_path`，导致绝对 target path 与 route report path 匹配失败。

结果：

- hook 看到了 `Write completed`。
- 但没有给 route 打 `report_ready=true`。
- projection 也没有进入 `report_ready`。
- `daemon_observed_write=false`。
- accept 只能靠文件存在兜底成功。

### 当前修复状态

已修改：

- `crates/agentcall-daemon/src/hooks.rs`
  - `report_write_for_wrapper` 改为使用 route `workspace` 解析相对 `report_path`。
  - 新增 `route_workspace(...)`。
  - 新增测试 `post_tool_report_write_uses_route_workspace_for_relative_report_path`。

已验证：

```powershell
cargo test -p agentcall-daemon -p agentcall-mcp --target-dir .agentcall_build\target-v60-test
```

结果：

```text
agentcall-daemon: 154 passed
agentcall-mcp: 9 passed
```

尚未完成：

- 因为重启动作被中断，修复尚未进入当前运行 daemon。
- 需要重建并重启 daemon 后，重新跑至少 1 个 worker 复验 board 能直接显示 `report_ready`。

## Issue 4：worker 仍会尝试被 policy 拒绝的 Bash

### 现象

在 `v6-stress-b-1781158807` 的事件里出现：

```text
hook.PreToolUse
tool=Bash
kind=denied_bash
summary=Bash denied by policy
```

随后该 worker 仍然恢复并使用 Write 写出了报告。

### 基础分析

本轮 objective 已明确要求：

```text
Use Write/Edit for the report, not Bash redirection.
```

但 Claude 仍尝试过 Bash。daemon policy 正确拒绝了它，而且 worker 后续能够改用 Write，说明：

- policy guard 生效。
- 但 worker 遵循路径/写入方式的稳定性仍不足。
- 单次 deny 不应立刻判失败；重复 deny 才应升级为 `blocked_by_policy`。

### 风险

如果任务更复杂，worker 可能重复撞同一 policy，形成 deny loop。

### 建议

- 保留当前 deny guard。
- 对同一 session、同一 tool、同一 reason 的重复 deny 聚合成 `blocked_by_policy`。
- 在 additionalContext 中明确告诉 worker：
  - 不要继续 Bash 写文件。
  - 使用 Write/Edit 到 `report_abs_path`。

## Issue 5：accept high 但 daemon_observed_write=false 的语义偏乐观

### 现象

6 个 accept 均返回：

```json
{
  "confidence": "high",
  "validation": {
    "daemon_observed_write": false,
    "exists": true,
    "non_empty": true
  }
}
```

### 基础分析

这次 `high` 的依据是 route report path 匹配、文件存在且非空；但由于 Issue 3，daemon 没有把 hook write 关联到 report。严格来说，这应该不是最高可信状态，至少应该降到 medium，或者将 `confidence` 拆成：

- `artifact_confidence=high`
- `daemon_write_confidence=low`
- `overall_confidence=medium`

不过 Issue 3 修好后，正常路径应该会重新得到 `daemon_observed_write=true`，这个问题的影响会显著降低。

### 建议

- 短期：等 Issue 3 runtime 复验。
- 中期：accept confidence 不应只返回一个字符串，最好给出分项证据。
