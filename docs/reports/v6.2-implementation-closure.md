# AgentCall v6.2 Implementation Closure

Date: 2026-06-11

## Summary

v6.2 addresses the issues exposed after v6.1: prompt submission was observable, but worker closure still depended too much on Codex manually choosing report paths, inferring target workspaces, and deciding whether a worker was still worth waiting for.

The implementation keeps `docs/v6.2-code-plan.md` frozen and focuses on the planned fixes:

- daemon-minted unique report paths when `agentcall_route` omits `report_path`;
- `request_report` as a route/session state transition;
- report requested/drafting/overdue/accepted worker states;
- project-aware board/session workspace projection;
- compact toolchain context in PTY handoff prompts;
- smoke coverage for 6 concurrent workers with explicit and daemon-minted report paths.

## Root Causes Fixed

| v6.1 symptom | Cause | v6.2 fix |
| --- | --- | --- |
| Parallel workers could collide or force Codex to invent report paths | Report path uniqueness was caller-owned | `agentcall_route` now mints `.agents/agentcall/<route_id>-<session_name>.md` under target workspace when omitted |
| `request_report` behaved like ordinary text | No route-level report request state | `agentcall_session_send(action=request_report)` patches route state with request id, deadline, and report block |
| Codex could not tell whether to wait for report closure | Summary lacked report lifecycle states | Worker state now projects `report_requested`, `report_drafting`, `report_overdue`, and `report_accepted` |
| Board/session mixed daemon repo and target project semantics | Projection did not expose all workspace roles | Board/session include daemon workspace, target workspace, Claude cwd, and report workspace |
| Worker often lacked known local toolchain context | Handoff prompt only carried task text and containment | PTY prompt injects compact toolchain hints from target `.agentcall/toolchain.json` or daemon `config/toolchain.local.json` |

## Code-Level Changes

- `crates/agentcall-daemon/src/routes.rs`
  - added report path minting and duplicate explicit report path warning;
  - route result now projects a normalized report block;
  - PTY prompts include a `Known local toolchain` section when hints are configured.

- `crates/agentcall-daemon/src/mcp.rs`
  - `request_report` records report lifecycle state rather than only sending natural language;
  - report acceptance reads the unified route report path;
  - route MCP responses expose the report block.

- `crates/agentcall-daemon/src/hooks.rs`
  - report writes patch route report status to `report_ready`;
  - tool/hook progress after `report_requested` advances projection to `report_drafting`.

- `crates/agentcall-daemon/src/worker_state.rs`
  - added report lifecycle states and project-aware workspace projection.

- `crates/agentcall-daemon/src/summary.rs` and `src/http.rs`
  - board supports `root` / `workspace` target filtering and reports filter metadata.

- `scripts/agentcall_real_worker_smoke.py`, `scripts/fake_pty_worker.py`, `scripts/agentcall_dev.py`
  - added `--omit-report-path` smoke path to test daemon-minted reports.

- Documentation and skills
  - updated README, CHANGELOG, AGENTS, AgentCall protocol, generated supervisor skill, and plugin skill.

## Residual Notes

- The default `target\debug\agentcall-daemon.exe` was locked by the currently running daemon during local build, so smoke used an isolated ignored target dir: `.agentcall_build\target-v62-smoke`.
- After validation, the old daemon/MCP processes were stopped, `cargo build --workspace` rebuilt `target\debug`, and the daemon was restarted successfully. `python agentcall.py verify-runtime-build` passed against `http://127.0.0.1:3293`.
- The current Codex thread's AgentCall MCP transport did not auto-respawn after the old MCP process was stopped and returned `Transport closed`. This is a host transport reconnection issue, not daemon health; a fresh Codex/plugin MCP connection is needed for in-thread MCP tool calls.
- v6.2 does not change the PTY-first runtime direction and does not reintroduce ACP/SDK as a default path.
