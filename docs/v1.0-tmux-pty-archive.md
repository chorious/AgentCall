# v1.0 PTY/tmux Archive

This document parks the v1.0 PTY/tmux-style direction.

The prototype proved a useful thing: AgentCall can own a terminal process, send
stdin, capture output, render panes in a browser, and preserve enough process
truth to supervise externally visible CLI agents.

That is no longer the short-term development direction.

## What It Proved

- AgentCall can launch owned PTY sessions.
- The browser can render terminal bytes through xterm.js.
- Input, terminal control responses, and resize events can travel over one
  WebSocket.
- The operator can watch and manually intervene in a visible session.
- The same workspace can hold task artifacts, reports, run records, and event
  logs.

## Why It Is Parked

PTY is friendly to the human operator, but it is weak as the primary runtime for
large parent/child agent collaboration:

- State is inferred from screen output instead of structured events.
- Mode switching is fragile and agent-specific.
- Permission handling is UI-shaped rather than protocol-shaped.
- Token/cost/report accounting must be reconstructed after the fact.
- Parent orchestration becomes too dependent on terminal transcripts.

The v2 direction is protocol-first:

```text
ParentOrchestrator
  -> AgentDriver
    -> Claude ACP / Claude SDK / headless JSON / future drivers
  -> structured ChildReport
  -> code-owned validators and acceptance policy
```

## Current Status

The PTY/tmux code remains in the repository as a v1.0 archive and fallback:

- `crates/agentcall-daemon/`
- `web/`
- `viewer/`
- `docs/session-supervisor.md`
- `docs/rust-daemon-architecture.md`

Do not treat these as the active v2 implementation target unless the goal is
explicitly to improve attach/debug/fallback behavior.

## Future Use

PTY may still be useful for:

- Attaching to already-open terminal agents.
- Human-visible debugging.
- Fallback support for agents without ACP/SDK/headless JSON.
- Showing a live terminal pane beside protocol-level state.

The main v2 investment should stay on bounded child lifecycles, ACP/SDK drivers,
structured reports, parent validation, and workflow inspection.
