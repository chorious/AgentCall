# AgentCall v6.2 Code Plan：Worker Closure 与 Project-Aware Supervisor Loop

日期：2026-06-11  
目标分支：`main`  
计划性质：v6.1 发布后的真实使用修正计划。不要回改 `docs/v6.1-code-plan.md`。

---

## 0. 一句话结论

v6.1 已经把 **prompt commit** 从“盲写 PTY bytes”推进到“可观察、可提交、可收敛”的基础层。  
v6.2 的目标不是继续修 prompt gate，而是修真实协作体验中的下一层问题：

> **让 Codex 能放心把小块工作交给 PTY worker，并能低成本完成 report/request/accept/stop 闭环。**

v6.2 聚焦五件事：

1. daemon 可用性预检更直接；
2. route 默认生成唯一 report path；
3. `request_report` 成为一等状态机；
4. worker summary 区分 daemon workspace 与 target project workspace；
5. worker 获得项目工具链/skills 摘要，减少“环境不明”导致的空转。

---

## 1. v6.1 后暴露的问题

输入来源：

- `docs/reports/v6.1-implementation-pressure-and-build-report.md`
- Codex 线程 `019eb2c2-e175-7153-b6c2-317b7dab002d` 的 v6.1 后 AgentCall 调用表现
- 当前 AgentCall board/session 使用体验

### 1.1 已经明显改善的部分

v6.1 后真实线程里出现了正向信号：

- `agentcall_route` 能稳定返回 `prompt_pending`。
- `agentcall_session` 能给出明确下一步 `submit_pending_prompt`。
- `agentcall_session_send(action=submit_pending_prompt)` 能触发后续 `UserPromptSubmit`。
- smoke worker 能输出 `REPORT READY`。
- destructive 控制动作使用 fresh `control_token` 的流程被 Codex 正确执行。
- 3 个 reviewer worker 能并发 route，并且三次 submit prompt 都成功。
- 最终 `agentcall_daemon(status)` 很快，且无残留 worker/lease。

这些证明 v6.1 的 prompt gate 和基础 PTY route 是有效的。

### 1.2 仍然不顺的部分

| 问题 | 真实表现 | 影响 |
|---|---|---|
| daemon liveness 仍需要手动 start | 一开始 session/board 超时，随后 `agentcall_daemon(start)` 恢复 | Codex 仍会先撞错，再修复 |
| report path 并行默认不稳 | Codex 总结“并行 report path 会互相踩” | 多 worker report 验收成本高 |
| `request_report` 不够一等 | Codex 对 3 个 reviewer 发 `request_report`，但后续没有形成清晰 `report_requested -> report_ready` 闭环 | worker 已工作但交付物状态不硬 |
| worker 工具链上下文不足 | Go 问题最终靠 parent 使用 compile skill 解决，不是 worker 自己解决 | worker 报告价值下降，Codex 倾向自己兜底 |
| Codex 仍倾向绕过 worker | reviewer 跑着时，Codex 自己继续落前端/Vic2/测试 | AgentCall 仍像辅助 reviewer，不像可托付执行工厂 |
| project/root 语义混乱 | 对 `E:\GameProject\RKV` 查 board，返回 daemon workspace `E:\Project\AgentCall` | 用户和 Codex 都容易误读当前项目视图 |

## 2. 根因分析

### 2.1 v6.1 修的是 prompt 入口，不是 worker 生命周期闭环

v6.1 的成功标准集中在：

- prompt 是否提交；
- prompt gate 是否收敛；
- report_ready 是否能覆盖 stale prompt；
- accept confidence 是否诚实。

但真实多 worker 工作流还需要另一组状态：

```text
routed
prompt_submitted
working
report_requested
report_drafting
report_ready
accepted
stopped
```

v6.1 对 `report_requested` / `report_drafting` 没有足够硬的投影，所以 Codex 看到 worker 在工作，但不知道该等、催、停、还是接管。

### 2.2 report path 仍由调用者承担过多责任

Codex 现在需要自己决定：

- report 放哪里；
- 多 worker report 是否唯一；
- 相对路径按哪个 workspace 解；
- accept 时是否能被 route 反查。

这不应该是 Codex 的正常负担。  
v6.2 应该让 daemon 在缺少 `report_path` 时自动 mint 一个唯一、安全、项目相关的 report path。

### 2.3 project workspace 与 daemon workspace 没有在 UI/API 上分清

daemon 自己运行在 `E:\Project\AgentCall`，但 worker 目标可能是 `E:\GameProject\RKV`。  
board 当前更像 daemon 全局视图，容易让调用方以为“root 参数没生效”或“查的是错项目”。

这不是单纯显示问题，而是 projection schema 问题：summary/board 必须同时展示：

- `daemon_workspace`
- `target_workspace`
- `claude_cwd`
- `report_workspace`

### 2.4 worker 不知道 parent 已掌握的本地工具链信息

RVK 线程里 Go 最终靠 `local-compile-toolchains` skill 解决。  
但 Claude PTY worker 不一定知道：

- Go wrapper 在哪里；
- Node/npm 是否可用；
- GOCACHE/GOTMPDIR 应该放到项目内；
- 哪些命令需要沙箱外运行；
- 哪些路径可写。

结果是 worker 容易把“我的 PATH 里没有 go”当 blocker，而 parent 已经知道解决方式。

### 2.5 Codex 缺少“何时等待 worker”的信心

v6.1 让 Codex 知道 prompt 进去了。  
但如果 worker 迟迟没有 report，Codex 仍会选择自己继续写代码，因为缺少低成本判断：

- worker 是否真的在读文件；
- report 是否被请求；
- report 是否正在生成；
- worker 是否已经偏离；
- 等它是否值得。

这需要 compact summary 给出更强的 `can_wait` 和 `why`。

## 3. v6.2 Product Goals

### 3.1 Route 自带唯一 report contract

`agentcall_route` 正常调用时，调用者可以不传 `report_path`。daemon 必须自动生成：

```text
<target_workspace>/.agents/agentcall/<route_id>-<session_name>.md
```

要求：

- 每个 route 唯一；
- 绝对路径和相对路径都可投影；
- route/session/accept 使用同一个 resolved path；
- allowed write policy 自动允许该 report path；
- board/session 明确展示 report path。

### 3.2 `request_report` 成为一等状态

`agentcall_session_send(action=request_report)` 不再只是发送自然语言。它必须：

- 写入 route/session 状态；
- 返回结构化状态；
- 生成唯一 request id；
- 在 summary 里显示 `report_requested=true`；
- 如果 worker 有下一次 hook/tool progress，进入 `report_drafting` 或继续 `working`；
- 如果 report 写出，进入 `report_ready`；
- 如果超时，进入 `report_overdue`，而不是无限 `working`。

建议状态：

```text
report_not_requested
report_requested
report_drafting
report_ready
report_overdue
report_accept_failed
report_accepted
```

### 3.3 Project-aware board/session

所有默认 projection 必须分清：

```json
{
  "daemon_workspace": "E:\\Project\\AgentCall",
  "target_workspace": "E:\\GameProject\\RKV",
  "claude_cwd": "D:\\guKimi",
  "report_path": "E:\\GameProject\\RKV\\.agents\\agentcall\\route-1-worker.md"
}
```

`agentcall_board(root=...)` 或 `workspace=...` 过滤语义必须明确：

- 如果按 target workspace 过滤，只显示该项目相关 workers/routes/reports；
- 如果没有 filter，则显示 daemon 全局；
- response 必须声明 `scope` 和 `workspace_filter_applied`。

### 3.4 Worker toolchain context packet

route handoff prompt 自动注入短小工具链摘要：

```text
Known local toolchain:
- Go: use <wrapper path> when available
- Node/npm: use npm from PATH when available
- Go cache/tmp: keep inside target workspace
- Do not treat PATH miss as blocker until checking provided toolchain hints
```

来源：

- repo-local config；
- `agentcall.py doctor`；
- optional `.agentcall/toolchain.json`；
- Codex skill summary only as text input，不让 Python 写 live state。

目标不是让 worker 自己管理复杂环境，而是避免反复踩已知坑。

### 3.5 Codex wait confidence

`agentcall_session` summary 必须给出更有力的判断：

```json
{
  "state": "working",
  "why": "worker observed reading backend files 23s ago; report requested 11s ago",
  "can_wait": true,
  "patience": {
    "status": "inside_patience_window",
    "recommended_sleep_seconds": 60
  },
  "next_actions": ["wait", "request_report"]
}
```

如果已经 request_report 且超时：

```json
{
  "state": "report_overdue",
  "can_wait": false,
  "next_actions": ["inspect_session", "interrupt", "stop"]
}
```

## 5. Implementation Plan

### P0：默认 report path minting

修改点：

- `routes.rs`
  - 当 route 缺 `report_path` 时生成唯一 report path。
  - 将 `report_path` 写入 route `context_packet` 和 `result.report_path`。
  - resolved report path 以 target workspace 为准。
- `hooks.rs`
  - report write 判断使用 resolved route report path。
- `mcp.rs`
  - `accept_report_for_session` 只信 route resolved report path。
  - 返回 `report_abs_path`、`report_rel_path`、`target_workspace`。
- `worker_state.rs`
  - summary report block 总是包含 report path。

验收：

- 同一 workspace 连续 route 6 个 worker，report path 全部不同。
- 缺 `report_path` 时也能 report_ready/accept。
- 手动传 `report_path` 时按调用者路径，但必须防重复或 warning。

### P0：`request_report` 状态化

修改点：

- `mcp.rs`
  - `session_send(action=request_report)` 写 route patch：
    - `report_request_id`
    - `report_requested_at`
    - `report_status=report_requested`
  - 返回 `not_completed=true`、`awaiting=report_write`。
- `projection.rs`
  - 根据 route report request 和 hook progress 推导 `report_drafting` / `report_overdue`。
- `worker_state.rs`
  - 新增 states：
    - `report_requested`
    - `report_drafting`
    - `report_overdue`
    - `report_accepted`

验收：

- request_report 后 summary 不再只是 `working`。
- report 写入前显示 `report_requested` 或 `report_drafting`。
- 超过 deadline 显示 `report_overdue`。
- report 写入后显示 `report_ready`。

### P1：project-aware board/session

修改点：

- `summary.rs`
  - board response 增加：
    - `daemon_workspace`
    - `workspace_filter`
    - `workspace_filter_applied`
  - worker projection 增加：
    - `target_workspace`
    - `claude_cwd`
    - `report_workspace`
- `mcp.rs`
  - `agentcall_board(root=...)` 不再 silently ignore root。
  - 如果 root 只是 target workspace filter，response 必须显式说明。

验收：

- 对 `E:\GameProject\RKV` 查 board 时，response 清楚显示 target filter，而不是只返回 daemon workspace。
- daemon global view 和 project filtered view 可区分。

### P1：worker toolchain context

修改点：

- 新增 daemon read-only toolchain snapshot：
  - `.agentcall/toolchain.json` 或 `config/toolchain.local.json`
  - 不进入 live state writer。
- route handoff prompt 注入 `toolchain_context`。
- `agentcall.py doctor` 输出可生成 toolchain 建议，但不自动写 live state。

验收：

- RVK worker prompt 中出现 Go wrapper/cache hints。
- worker 不因 bare PATH 缺 Go 直接 blocker。
- toolchain hints 缺失时不阻断 route，只显示 `toolchain_context=missing`。

### P2：Codex supervisor skill 更新

修改点：

- 更新 `docs/agentcall-supervisor-skill.md`。
- 重新生成 `.codex/skills/agentcall-supervisor/SKILL.md` 和 plugin skill。
- 新增规则：
  - 优先看 `report_status`；
  - request_report 后等待 deadline；
  - `report_overdue` 才 interrupt/stop；
  - 不再自己猜 report path。

验收：

- `python scripts/generate_agentcall_skill.py --check` 通过。
- skill 明确禁止正常路径传 lease/precondition/idempotency。

### P2：stress smoke 扩展

修改点：

- `scripts/agentcall_real_worker_smoke.py`
  - 增加 no-report-path route 压测。
  - 增加 request_report state smoke。
  - 增加 board project filter smoke。

验收：

```powershell
python agentcall.py smoke real-worker --parallel-workers 6
python agentcall.py smoke real-worker --parallel-workers 6 --omit-report-path
```

必须证明：

- 6 个默认 report path 唯一；
- request_report 可投影；
- accept high 仍需要 daemon evidence；
- stop 后 health/leases 归零。

## 7. Test Plan

### Unit

- route missing report path mints unique path。
- report path resolves against target workspace。
- duplicate explicit report path returns warning or structured refusal。
- request_report patches route state。
- report_overdue activates after deadline。
- board root filter applies to target workspace, not daemon workspace。

### Smoke

- single worker without explicit report_path completes report_ready/accept high。
- 3-worker concurrent without explicit report_path completes。
- 6-worker pressure without explicit report_path completes。
- request_report before report write shows `report_requested`。
- request_report timeout shows `report_overdue`。
- project board filter shows only target project workers。

### Regression

```powershell
cargo test --workspace
python -m pytest -q
python agentcall.py release-check
python agentcall.py smoke real-worker --parallel-workers 6
```

## 8. Acceptance Criteria

v6.2 完成必须满足：

1. Codex 正常 route worker 时可以不传 `report_path`。
2. daemon-minted report path 在 6-worker 并发下唯一。
3. report path 在 route/session/report accept 中解析一致。
4. `request_report` 后 summary 明确显示 report 状态，而不是 generic `working`。
5. report 超时显示 `report_overdue`，不让 Codex 无限等待。
6. project-aware board 能区分 daemon workspace 与 target workspace。
7. worker handoff 包含可选 toolchain context，避免已知 PATH/cache 坑。
8. no-report-path 6-worker pressure 通过。
9. `release-check` 通过。
10. 不新增默认 MCP 工具，不恢复 ACP，不改 v6.1 冻结计划。
