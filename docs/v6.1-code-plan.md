# AgentCall v6.1 Code Plan：Prompt Commit 收敛与 Report Projection 运行态闭环

日期：2026-06-11  
目标分支：`main`  
计划性质：v6.0 压测后的修正计划。不要回改 `docs/v6.0-code-plan.md`，本文件作为 v6.1 执行计划。

---

## 0. 一句话结论

v6.1 不要继续扩大工具面，也不要把 `pending_prompt_commit_sent` 换个名字糊过去。  
v6.1 的核心目标是：

> **route prompt 从“写入 PTY bytes”升级为“可观察、可超时、可自动收敛的 prompt commit contract”。`submit_pending_prompt` 之后必须在有限时间内收敛到 `prompt_submitted` 或明确失败态，不能稳定停在 `pending_prompt_commit_sent` / commit-sent 中间态。**

同时，v6.1 必须把 v6.0 压测中已经暴露的 report projection、waiting_input 语义、运行态复验、hook redaction 一次性收口。

---

## 1. v6.0 暴露的问题摘要

### 1.1 构建期问题

v6.0 构建和自测暴露的问题：

1. Codex 宿主 MCP transport 可能长期 `Transport closed`，但 daemon HTTP `/api/mcp/call` 可用。
2. token 鉴权开启后，开发诊断脚本未带 `X-AgentCall-Token`，导致 `daemon-health` 误报 401。
3. route `workspace` 与 Claude Code process cwd 分离后，相对 `report_path` 容易被解析到错误目录。
4. `prompt_submit_attempted` 未被 `UserPromptSubmit` ack 逻辑识别，导致实际 prompt 已提交但 route/projection 看起来仍卡在 prompt gate。
5. `accept_report_for_session` 曾按 daemon workspace 解析 report path，导致真实报告存在但返回 `report_missing_or_empty`。
6. report 已经写出并可 accept，但 stale prompt gate 曾压过 report-ready，导致 summary 仍可能显示 `prompt_missing`。
7. hook 原始 payload 仍过重，需要继续保持 redaction / compact event 方向。

### 1.2 压测期问题

v6.0 压测暴露的问题：

1. 6/6 route 初始 prompt 未被 Claude Code 自动提交，全部进入 `prompt_missing`。
2. 6/6 调用 `agentcall_session_send(action=submit_pending_prompt)` 后返回 `pending_prompt_commit_sent`，随后 hook 才观察到 `UserPromptSubmit` 并开始工作。
3. `pending_prompt_commit_sent` 的语义对 Codex/监督者不直观，容易被误读为“任务已提交/已完成”。
4. `waiting_input` 过粗，无法说明是在等问题答案、等权限菜单、turn-end idle、report 已写出待验收，还是 checkpoint-like 等待。
5. report 文件已经写出，但 board/session 在 accept 前仍显示 `waiting_input` / `report.ready=false`。
6. worker 仍可能尝试 Bash 写入路径，虽然 wrapper policy 能纠偏，但重复 deny 需要更明确聚合。
7. accept 返回 `confidence=high` 但 `daemon_observed_write=false`，语义偏乐观。
8. 修复已写但未重启 daemon 时，运行态没有复验，容易造成“测试过了但 daemon 仍跑旧代码”。

## 2. v6.1 目标

### 2.1 Product Goals

1. **不稳定复现 `pending_prompt_commit_sent`**
   - 公开返回和 route/session 状态不再使用 `pending_prompt_commit_sent` 作为持久状态。
   - `submit_pending_prompt` 只是发出一次 commit signal，必须在 deadline 内自动收敛为：
     - `prompt_submitted`，或
     - `prompt_commit_unacknowledged` / `prompt_commit_failed`。
   - 任何 session 不得在 commit-sent 中间态停留超过 `PROMPT_COMMIT_ACK_DEADLINE_MS`。

2. **Codex 不再误判 prompt commit**
   - `prompt_commit_signal_sent` 不代表任务开始。
   - 真正的任务开始只能由以下事件关闭 prompt gate：
     - `hook.UserPromptSubmit`；
     - 或后续 `hook.PreToolUse` / `hook.PostToolUse` / `hook.PostToolBatch` 等明确 worker progress；
     - 或报告写出 / report_ready。
   - Summary 必须返回 `not_completed=true` / `awaiting_hook=UserPromptSubmit`。

3. **report_ready 在 accept 前可见**
   - 如果 hook 观察到 Write report，board/session 必须显示 `report_ready`。
   - 如果 daemon 没观察到 write，但 accept 通过文件存在兜底，confidence 必须拆分或降级，不能简单 high。

4. **waiting_input 细分**
   - `waiting_input` 不再作为单一推荐动作来源。
   - 至少区分：
     - `idle_after_turn`
     - `question_waiting_answer`
     - `permission_menu`
     - `report_written_waiting_accept`
     - `checkpoint_due`
     - `prompt_commit_unacknowledged`

5. **构建/运行态一致**
   - 构建后必须能确认 daemon 运行的是新 binary。
   - 压测前必须重启 daemon 或验证 build id。
   - 报告中“已修复但未运行态复验”的状态必须变成 hard fail。

### 2.2 Non-goals

- 不新增大工具面。
- 不引入 SDK/ACP 主线。
- 不把 HTTP raw input 作为 Codex 正式恢复路径。
- 不允许通用 `raw_input` MCP action。
- 不改变 Claude Code process cwd 仍由 `claude_workspace` 控制的原则。
- 不把 terminal screen snapshot 当状态真相。
- 不放松 path containment、binding fail-closed、Bash readonly policy。
- 不让 report accept 自动释放 lease。

## 3. v6.1 核心不变量

### Invariant A：commit-sent 是瞬时尝试，不是 worker state

禁止把以下字符串作为持久 route/session/worker state：

```text
pending_prompt_commit_sent
```

替代语义：

```text
prompt_commit_signal_sent       // action result only, short-lived
prompt_submitted                // observed UserPromptSubmit/progress
prompt_commit_unacknowledged    // commit signal sent but no ack before deadline
prompt_commit_failed            // attempts exhausted / session not in prompt gate
```

### Invariant B：route prompt 的成功条件不是 `input_sent_by_actor`

`input_sent_by_actor` 只表示 daemon 把 bytes 写进 PTY writer。它不表示：

```text
Claude Code 接受了 user turn
UserPromptSubmit 已触发
worker 已开始任务
```

### Invariant C：prompt gate 必须被 worker progress 自动关闭

以下任一事件发生时，prompt gate 立即关闭为 submitted/progress observed：

```text
hook.UserPromptSubmit
hook.PreToolUse
hook.PostToolUse
hook.PostToolBatch
hook.Stop / hook.SubagentStop after task_started
report write observed
report_ready
```

如果 `UserPromptSubmit` hook 丢了但 worker 已经开始 tool use，不得继续显示 prompt gate stuck。

### Invariant D：report_ready 优先于 stale prompt/waiting_input

如果 report artifact 已经写出并匹配 route report path：

```text
worker_state = report_ready
next_action = accept_report
```

不得继续显示：

```text
prompt_missing
waiting_input
continue
```

### Invariant E：direct HTTP input 是救火路径，不是产品路径

正式恢复路径必须是：

```text
agentcall_session_send(action=submit_pending_prompt)
```

不得要求 Codex 调 `/api/sessions/{name}/input` 才能闭环。

## 4. Public State Model

### 4.1 Codex-visible worker states

v6.1 的 `agentcall_session(view=summary)` 和 board attention 应该对 Codex 暴露单一 `state`：

```text
starting
prompt_pending
prompt_missing
prompt_commit_unacknowledged
prompt_submitted
working
idle_after_turn
question_waiting_answer
needs_permission
blocked_by_policy
report_ready
stopping
done
failed
```

内部仍可保留 `liveness_status` / `attention_status` / `turn_status`，但 Codex 默认只看 `state`。

### 4.2 Prompt gate states

Route record 内部保存更细状态：

```text
not_required
created
prompt_written
prompt_pending_ack
prompt_missing
commit_signal_sent
prompt_submitted
prompt_commit_unacknowledged
prompt_commit_failed
aborted_session_exited
```

公开 summary 不直接暴露完整内部状态，只暴露：

```json
{
  "prompt_gate": {
    "state": "prompt_missing",
    "awaiting_hook": "UserPromptSubmit",
    "commit_attempts": 0,
    "can_submit_pending_prompt": true
  }
}
```

### 4.3 submit_pending_prompt action result

`agentcall_session_send(action=submit_pending_prompt)` 返回：

```json
{
  "ok": true,
  "status": "prompt_commit_signal_sent",
  "not_completed": true,
  "awaiting_hook": "UserPromptSubmit",
  "attempt_id": "prompt-commit-route-123-worker-a-1",
  "ack_deadline_ms": 8000,
  "next_observation": "agentcall_session(view=summary)"
}
```

不得返回：

```json
{
  "status": "pending_prompt_commit_sent"
}
```

兼容层可以接受旧字符串，但新代码、测试、文档不得生成它。

### 5.1 Route record 增加 PromptGate v2

在 route `result.prompt_gate` 中存储：

```json
{
  "schema_version": 2,
  "state": "prompt_pending_ack",
  "route_id": "route-123",
  "session_name": "worker-a",
  "prompt_command_id": "cmd-...",
  "prompt_idempotency_key": "route_prompt:route-123:worker-a",
  "prompt_written_at": "2026-06-11T...",
  "ack_deadline_ms": 15000,
  "awaiting_hook": "UserPromptSubmit",
  "task_started": false,
  "submitted_at": null,
  "progress_observed_at": null,
  "commit_attempts": [],
  "active_commit_attempt_id": null,
  "last_error": null
}
```

`commit_attempts` item：

```json
{
  "attempt_id": "prompt-commit-route-123-worker-a-1",
  "kind": "manual_mcp|auto_self_heal",
  "state": "signal_sent|acknowledged|unacknowledged|failed",
  "sent_at": "...",
  "ack_deadline_ms": 8000,
  "command_id": "cmd-...",
  "idempotency_key": "prompt_commit:route-123:worker-a:1"
}
```

### 5.2 新增 helper

实现：

```rust
update_route_prompt_gate_locked(...)
mark_prompt_gate_commit_signal_sent(...)
mark_prompt_gate_submitted_locked(...)
mark_prompt_gate_progress_observed_locked(...)
mark_prompt_gate_commit_unacknowledged_locked(...)
refresh_prompt_gate_timeouts(state)
```

要求：

- `refresh_prompt_gate_timeouts(state)` 在以下入口调用：
  - `agentcall_board`
  - `agentcall_session`
  - `agentcall_session_send`
  - `agentcall_report`
  - daemon health 可选
- 不需要后台 scheduler；用 read-time refresh 即可避免 stale display。
- 如果 `commit_signal_sent` 超过 deadline 且没有 ack/progress，必须 patch 为 `prompt_commit_unacknowledged`。

### 5.3 关闭 prompt gate 的事件

在 `hooks.rs` 中：

- `UserPromptSubmit`：
  - route prompt gate state -> `prompt_submitted`
  - `task_started=true`
  - active attempt -> `acknowledged`
- `PreToolUse` / `PostToolUse` / `PostToolBatch`：
  - 如果 route prompt gate 仍 pending/missing/commit_signal_sent：
    - state -> `prompt_submitted`
    - `task_started=true`
    - `progress_observed_at=event.ts`
    - reason = `progress_without_user_prompt_submit`
- report write observed：
  - state -> `prompt_submitted`
  - task_started=true
  - report_ready=true

这避免 hook 丢失 `UserPromptSubmit` 时，worker 已实际工作但 gate 仍卡住。

### Code targets


### 5.4 新增/修正 action

在 `agentcall_session_send` schema 中保留/新增：

```text
submit_pending_prompt
```

action 行为：

- 只允许在 prompt gate 为：
  - `prompt_missing`
  - `prompt_pending_ack` 且超过 initial grace
  - `prompt_commit_unacknowledged` 且 attempts 未耗尽
- 不允许在：
  - `working`
  - `report_ready`
  - `needs_permission`
  - `blocked_by_policy`
  - terminal
- 不重新发送完整 prompt。
- 只发送 commit signal：
  - 首选：`text=" "`, `enter=true`
  - 可配置 fallback：只发 Enter
- 每个 route prompt 的 manual commit attempts 默认最多 1 次；auto self-heal 默认最多 1 次。
- 如果已经有 active commit attempt 且未超时，返回同一个 attempt 状态，不再重复写 PTY。

### 5.5 返回值更名

旧：

```json
{"status": "pending_prompt_commit_sent"}
```

新：

```json
{
  "ok": true,
  "status": "prompt_commit_signal_sent",
  "not_completed": true,
  "awaiting_hook": "UserPromptSubmit",
  "attempt_id": "...",
  "ack_deadline_ms": 8000,
  "next_observation": "agentcall_session(view=summary)"
}
```

如果超时后再次 summary：

```json
{
  "state": "prompt_commit_unacknowledged",
  "why": "A prompt commit signal was sent, but UserPromptSubmit or tool progress was not observed before the deadline.",
  "next_action": "retry_prompt_commit_or_inspect_screen"
}
```

### 5.6 `send/continue` 不得误入 queue

当 worker state 为：

```text
prompt_missing
prompt_commit_unacknowledged
prompt_pending
```

`agentcall_session_send(action=send|continue)` 必须返回 structured refusal：

```json
{
  "ok": false,
  "status": "prompt_not_submitted",
  "reason": "Route prompt is still pending; do not queue natural-language instructions.",
  "next_action": "submit_pending_prompt",
  "recommended_args": {
    "action": "submit_pending_prompt"
  }
}
```

不得进入：

```text
queued_until_next_hook_injection
```

## Phase 3：route 初始提交可靠性

### Code targets

```text
crates/agentcall-daemon/src/routes.rs
crates/agentcall-daemon/src/hooks.rs
crates/agentcall-daemon/src/session.rs
```

### 5.7 route start 不直接长 prompt 砸 PTY

确认 v6.0 已做 handoff file 后，v6.1 继续强化：

- 完整 objective/criteria/rules 写入 handoff file。
- PTY prompt 使用短单行或短多行，不超过 800 chars。
- Prompt 中必须包含 `handoff_abs_path` 和 `report_abs_path`。
- 不在 prompt 中重复长 acceptance criteria。
- `UserPromptSubmit` raw prompt 不应携带巨长原文。

### 5.8 等 Claude ready 后再写 prompt

route start 后，不要在 Claude TUI 未 ready 时立即提交长 prompt。

策略：

1. spawn PTY；
2. 等 `hook.SessionStart`，最多等待 `PROMPT_READY_WAIT_MS`；
3. 或 terminal screen 出现可输入 prompt；
4. 或 bounded fallback delay；
5. 再写短 handoff prompt；
6. 进入 `prompt_pending_ack`。

注意：MCP route call 仍是短事务。等待/提交可由 daemon 内部 prompt delivery worker 或 read-time state machine 推进，不阻塞 MCP HTTP request。

### 5.9 bounded self-heal

如果 route 初始 prompt 写入后 `PROMPT_ACK_DEADLINE_MS` 内没有 ack/progress：

- daemon 可以自动发一次 commit signal；
- 记录 attempt：
  - `kind=auto_self_heal`
  - `state=signal_sent`
- 如果自动 self-heal 成功，state -> `prompt_submitted`。
- 如果仍失败，state -> `prompt_commit_unacknowledged`，Codex next_action -> `inspect_screen_or_retry_prompt_commit`。

这能减少 6/6 稳定需要人工 `submit_pending_prompt` 的概率。

## Phase 5：Report projection 运行态闭环

### Code targets

```text
crates/agentcall-daemon/src/hooks.rs
crates/agentcall-daemon/src/mcp.rs
crates/agentcall-daemon/src/projection.rs
crates/agentcall-daemon/src/confidence.rs
```

### 5.13 report path 统一解析

已修过的逻辑必须运行态复验：

- 相对 `report_path` 必须按 route `workspace` 解析。
- `report_abs_path` 必须进入 route result。
- hook `Write` target 与 `report_abs_path` 比较时使用 canonical/lexical normalization。
- Windows 路径大小写、`\`/`/`、不存在路径 lexical normalize 都要稳定。

### 5.14 hook report write 直接触发 report_ready

当 `PostToolUse` / `PostToolBatch` 观察到 Write/Edit/MultiEdit 命中 `report_abs_path`：

- route status -> `report_ready`
- route result.report.ready = true
- session projection.report_ready = true
- worker_state = `report_ready`
- save report index
- `daemon_observed_write=true`
- prompt gate 若未关闭，也关闭为 `prompt_submitted` / `progress_observed`

### 5.15 accept confidence 拆分

`agentcall_report(action=accept)` 返回：

```json
{
  "ok": true,
  "status": "accepted",
  "confidence": {
    "overall": "high|medium|low",
    "artifact": "high|medium|low",
    "daemon_write": "high|medium|low",
    "route_match": "high|medium|low"
  },
  "validation": {
    "exists": true,
    "non_empty": true,
    "matched_route_report_path": true,
    "daemon_observed_write": true,
    "report_abs_path": "..."
  }
}
```

规则：

- `exists=true` + `non_empty=true` + route path match + `daemon_observed_write=true` -> overall high。
- `exists=true` + `non_empty=true` + route path match + `daemon_observed_write=false` -> overall medium，不是 high。
- route path mismatch -> low 或 reject。

### 5.16 单次 Bash deny 不直接失败

单次被拒绝 Bash 写入：

- compact event: `denied_bash`
- 不立刻 terminal
- additionalContext 注入明确纠偏：
  - 不要用 Bash 写文件
  - 使用 Write/Edit 到 `report_abs_path`
  - Bash 只做 read-only 探针

### 5.17 重复 deny 聚合

如果同 session、同 tool、同 reason 重复超过阈值：

```json
{
  "state": "blocked_by_policy",
  "next_action": "interrupt_or_request_blocker_report"
}
```

对不同 target 的连续 denied Write，也要按短时间窗口聚合，不能每个 target repeat_count=1 就永远不进入 blocked。

## 8. Acceptance Criteria

v6.1 is accepted only if all are true:

1. `pending_prompt_commit_sent` is not emitted by public MCP/HTTP summary/action result.
2. `submit_pending_prompt` returns `prompt_commit_signal_sent`, `not_completed=true`, `awaiting_hook=UserPromptSubmit`.
3. A commit signal attempt is never stable:
   - within deadline -> `prompt_submitted`, or
   - after deadline -> `prompt_commit_unacknowledged` / `prompt_commit_failed`.
4. `send` / `continue` during prompt gate pending/missing does not enter hook queue.
5. `UserPromptSubmit`, tool progress, or report write closes prompt gate.
6. `report_ready` beats stale prompt gate and generic waiting_input.
7. report-ready is visible before accept when hook observes report write.
8. accept confidence is split; overall high requires daemon_observed_write or equivalent daemon evidence.
9. waiting_input has specific `pending_interaction.kind`.
10. v6.1 smoke does not require direct HTTP `/api/sessions/{name}/input` to close a worker.
11. build/runtime health proves daemon was restarted into new binary before pressure test.
12. hook raw command/content remains redacted by default.
13. 6-worker pressure does not leave silent running/pending states.
