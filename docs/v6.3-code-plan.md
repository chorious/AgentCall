# AgentCall v6.3 Code Plan：Primary Action Control 与 Read/Write Policy 分层

日期：2026-06-11  
目标分支：`main`  
计划性质：v6.2 真实使用反馈后的修正计划。不要回改 `docs/v6.2-code-plan.md`。

---

## 0. 一句话结论

v6.2 已经让 worker closure 变得可观察：route 能生成 report path，`request_report` 能进入状态机，report accept 能用 daemon evidence 判断可信度。

v6.3 要修的是下一层体验问题：

> **AgentCall 不要只告诉 Codex“你能做什么”，而要告诉 Codex“现在唯一应该做什么”。**

当前问题不是 Codex 完全不会用工具，而是 AgentCall 把 **可用动作**、**建议动作**、**恢复动作** 混在同一个 `next_actions` 菜单里。Codex 在健康 `working` 时过早 `request_report`，在 policy block 时过早 `stop`，在 `prompt_pending` 时还需要帮 daemon 做二次提交。

v6.3 聚焦五件事：

1. `primary_action` 成为 Codex 默认执行的唯一动作；
2. daemon 强制 60 秒 patience floor，而不是只靠 skill 文案；
3. prompt pending 的首次 commit 由 daemon 自动自愈，不暴露给 Codex；
4. Read/Grep/Glob 默认不做硬路径 deny，写边界继续强约束；
5. policy block 分类，并给恢复动作，而不是默认 stop。

## 1. v6.2 后暴露的问题

输入来源：

- Codex 线程 `019eb2c2-e175-7153-b6c2-317b7dab002d` 的 v0.3/v0.4/v0.5 三路 AgentCall 调用表现；
- v6.0 / v6.1 压测中反复出现的 prompt submit / wait / request_report 误用；
- 2026-06-11 反馈文档关于 `primary_action`、patience、读写权限分层的建议。

### 1.1 `working` 同时返回 `wait` 和 `request_report`

`worker_state.rs` 当前在健康 `Working` 下返回：

```json
[
  {"kind": "wait"},
  {"kind": "request_report"}
]
```

Codex 很容易把这个数组理解成菜单，并为了“提高效率”选择 `request_report`。真实线程里三个 worker 刚进入 working 不久，Codex 只等约 10 秒就请求报告。

这不是模型坏，而是 API affordance 错。

### 1.2 60 秒耐心只是建议，不是协议

skill 文档写了 60 秒 patience，但 daemon 没有把它变成硬语义。

结果：

- Codex 可以 10 秒后直接 `request_report`；
- daemon 会照办；
- worker 还在阅读/思考时被迫收尾；
- 用户感知为“Claude 慢、Codex 急、AgentCall 体验差”。

### 1.3 `prompt_pending -> submit_pending_prompt` 暴露给 Codex

v6.1/v6.2 把 `submit_pending_prompt` 做成了可恢复动作，但真实体验里它变成每次 route 后 Codex 都要帮 daemon “按一次回车”。

正常产品路径不应该这样。Codex 默认应该看到：

```text
prompt_pending
prompt_submitted
prompt_commit_unacknowledged
```

而不是常规执行：

```text
submit_pending_prompt
```

### 1.4 v0.4 被外部只读资料路径挡住

真实任务中 v0.4 worker 需要读取：

```text
E:\SteamLibrary\steamapps\common\Victoria 2\map\...
```

但 route 只给了：

```json
allowed_paths: ["E:\\GameProject\\RKV"]
read_only: true
```

安全拦截本身符合当前实现，但产品语义错了：这类任务天然需要外部只读参考资料。AgentCall 不应该把读路径和写路径混成同一条硬边界。

### 1.5 policy block 后恢复动作太粗

当 worker 被 read path policy 卡住时，Codex 选择 stop/reopen。更合理的主动作应该是：

```text
reroute_with_reference_paths
```

或者在允许时继续当前任务并补充只读参考路径。

## 2. 根因分析

### 2.1 `next_actions` 是宽菜单，不是控制协议

LLM 看到数组时会主动选择它觉得“推进最快”的动作。  
如果健康 working 同时给 `wait` 和 `request_report`，Codex 就会过早催报告。

v6.3 必须把动作分层：

```text
primary_action      正常路径唯一建议动作
available_actions   技术上可用，但不是默认建议
debug_actions       人类/排障/破局使用
```

兼容字段 `next_actions` 可以暂时保留，但必须从 `primary_action` 派生，不能再作为宽菜单诱导 Codex。

### 2.2 patience 是状态机职责，不是提示词职责

“请等 60 秒”写在 skill 里不够。  
Codex 可能因为任务焦虑、上下文压力或用户催促而提前操作。

daemon 必须拒绝过早动作：

```json
{
  "ok": false,
  "status": "inside_patience_window",
  "reason": "Worker submitted prompt 10s ago; minimum initial wait is 60s.",
  "primary_action": {
    "kind": "wait",
    "sleep_seconds": 50
  }
}
```

### 2.3 prompt commit 是 wrapper 内部细节

`submit_pending_prompt` 是必要的 recovery primitive，但不应该是默认交互路径。  
daemon 已经知道 route prompt、session actor、ack deadline 和 hook progress，因此应在 route start 后做一次 bounded self-heal。

### 2.4 读权限和写权限风险不同

当前本地单用户 AgentCall 的核心风险是写坏项目、污染资产、错误 claim，不是 worker 读取本机参考目录。

默认应强约束：

- `Write`
- `Edit`
- `MultiEdit`
- Bash 写行为

默认不应硬拦：

- `Read`
- `Grep`
- `Glob`

Read 的安全侧应靠 redaction、payload budget、secret 抑制和 strict mode，而不是默认 deny。

## 3. Product Goals

### 3.1 `primary_action` 控制面

`agentcall_session` summary 和 `agentcall_board` worker projection 新增：

```json
{
  "primary_action": {
    "kind": "wait",
    "sleep_seconds": 60,
    "not_before_at": "2026-06-11T10:55:00Z",
    "reason": "inside_patience_window_after_UserPromptSubmit"
  },
  "available_actions": [],
  "debug_actions": [
    {"kind": "inspect_events"}
  ]
}
```

规则：

- `primary_action` 是 Codex 默认应该执行的唯一动作；
- `available_actions` 只放技术上可用但不推荐的动作；
- `debug_actions` 只给人类或排障使用；
- `next_action` 可兼容保留为 `primary_action.kind`；
- `next_actions` 可暂时保留，但不能再列出健康 working 下的 `request_report`。

### 3.2 daemon-enforced patience floor

新增 patience 状态：

```json
{
  "patience": {
    "status": "inside_patience_window",
    "started_at_ms": 1781178469000,
    "not_before_ms": 1781178529000,
    "sleep_seconds": 60,
    "reason": "after_UserPromptSubmit"
  }
}
```

默认规则：

```text
After UserPromptSubmit:
  first_wait_floor_seconds = 60

After meaningful tool progress:
  next_wait_floor_seconds = 60

After report_requested:
  report_wait_floor_seconds = 60
  report_deadline_seconds = 180
```

在 patience window 内：

- `request_report` 返回 `inside_patience_window`，除非 `user_explicit_close=true`；
- `continue` 返回 `inside_patience_window`；
- `stop` / `interrupt` 仍要求 control token，并应被归类为 destructive/debug，不作为 primary action；
- board attention 不应显示健康 worker。

### 3.3 prompt pending 自动自愈

route start 后，daemon 做一次 bounded prompt commit self-heal：

```text
route start
daemon writes prompt
daemon waits SessionStart / PTY ready
daemon auto-commit once
daemon waits UserPromptSubmit or tool progress
```

Codex 默认看到：

```text
prompt_pending within deadline -> primary_action=wait
prompt_submitted -> primary_action=wait
prompt_commit_unacknowledged -> primary_action=inspect_or_restart
```

`submit_pending_prompt` 保留为 debug/recovery action：

```text
debug_actions = ["submit_pending_prompt_once"]
```

但不再作为正常 `prompt_pending` 的 primary action。

### 3.4 Read/Write policy 分层

新增或明确 route 字段：

```text
write_paths / writable_paths:
  硬写入边界。Write/Edit/MultiEdit 必须命中。

report_path:
  daemon 自动加入 writable roots。

scratch_path:
  session 写入沙盒。

reference_paths:
  外部只读参考资料路径，用于 prompt/context/诊断。

read_policy:
  unrestricted | target_plus_reference | strict
```

默认：

```json
{
  "read_policy": "unrestricted",
  "reference_paths": [],
  "write_paths": [
    "<target_workspace>/.agents/agentcall",
    "<scratch_root>"
  ]
}
```

默认策略：

```text
Read/Grep/Glob:
  默认允许本地读取；
  不创建 file claim；
  不触发 path denial；
  做 redaction/budget/secret 抑制。

Write/Edit/MultiEdit:
  必须命中 write_paths / scratch / report_path；
  创建 file claim。

Bash:
  默认 readonly-only；
  不允许重定向、rm/mv/cp/mkdir、PowerShell 写入等；
  如需写，要求 worker 使用 Write/Edit。
```

对 Vic2 本地资源类任务：

```json
{
  "workspace": "E:\\GameProject\\RKV",
  "reference_paths": [
    "E:\\SteamLibrary\\steamapps\\common\\Victoria 2"
  ],
  "write_paths": [
    "E:\\GameProject\\RKV\\.agents\\agentcall",
    "<scratch_root>"
  ]
}
```

### 3.5 policy block 分类与恢复动作

policy denial 不再统一变成“等/停/中断”的粗动作。

新增分类：

```text
blocked_by_write_path
blocked_by_bash_write
blocked_by_missing_reference_path
blocked_by_permission_menu
blocked_by_repeated_denial
```

对应 primary action：

| Block kind | primary_action |
| --- | --- |
| `blocked_by_write_path` | `fix_write_paths_or_request_blocker_report` |
| `blocked_by_bash_write` | `tell_worker_use_write_tool_or_report_blocker` |
| `blocked_by_missing_reference_path` | `reroute_with_reference_paths` |
| `blocked_by_permission_menu` | `select_option` |
| `blocked_by_repeated_denial` | `interrupt_or_stop` |

只有 repeated denial、明显漂移或用户显式要求回收时，`stop` 才能成为 primary action。

## 4. State / Action Matrix

| Worker state | primary_action | 不应该默认出现 |
| --- | --- | --- |
| `prompt_pending` | `wait` with 60s or deadline | `submit_pending_prompt` |
| `prompt_commit_unacknowledged` | `inspect_or_restart` | `request_report` |
| `prompt_submitted` | `wait` | `request_report` |
| `working` with recent progress | `wait` 60s | `request_report`, `continue`, `stop` |
| `working` no progress but inside patience | `wait` remaining | `request_report` |
| `working` no progress after patience | `inspect_session` or `request_report_if_closing` | blind `continue` |
| `idle_after_turn` without report | `request_report` | `stop` |
| `report_requested` inside deadline | `wait` 60s | duplicate `request_report` |
| `report_drafting` | `wait` 60s | `continue` |
| `report_ready` | `accept_report` | `continue` |
| `blocked_by_missing_reference_path` | `reroute_with_reference_paths` | `stop` |
| `blocked_by_write_path` | `fix_write_paths_or_request_blocker_report` | blind `wait` |
| `blocked_by_repeated_denial` | `interrupt_or_stop` | `wait` |
| `needs_permission` / menu | `select_option` | `send`, `continue` |
| `report_overdue` | `inspect_or_interrupt` | `wait` |

## 5. Public Interface Changes

### 5.1 `agentcall_session` summary

Add:

```json
{
  "primary_action": {},
  "available_actions": [],
  "debug_actions": [],
  "patience": {},
  "policy_block": {
    "kind": "blocked_by_missing_reference_path",
    "suggested_reference_paths": []
  }
}
```

Adjust:

- `next_action` becomes compatibility alias for `primary_action.kind`;
- `next_actions` should be derived from primary action only in compact/default summary;
- debug/raw views may expose full action details.

### 5.2 `agentcall_session_send`

Add optional field:

```json
{
  "user_explicit_close": true
}
```

Behavior:

- `request_report` during healthy patience window returns structured refusal unless `user_explicit_close=true`;
- duplicate `request_report` during `report_requested/report_drafting` returns current report state, not a new request;
- `continue` during healthy patience window returns structured refusal with wait instruction.

### 5.3 `agentcall_route`

Add:

```json
{
  "reference_paths": ["..."],
  "write_paths": ["..."],
  "read_policy": "unrestricted"
}
```

Compatibility:

- Existing `allowed_paths` maps to `write_paths` in default policy;
- docs and skill should teach `write_paths/reference_paths`, not `allowed_paths` as a read/write combined concept.

### 5.4 Board

Compact board should:

- hide healthy workers inside patience window from attention;
- show `primary_action` for each worker;
- show `blocked_by_missing_reference_path` with suggested recovery when strict read policy blocks a read;
- not list `request_report` for healthy working workers.

## 6. Implementation Order

### P0：Action 分层

- Add `primary_action`, `available_actions`, `debug_actions` to `WorkerStateView`.
- Change `Working` next action generation:
  - primary: `wait`;
  - no default `request_report`.
- Keep compatibility `next_action` / `next_actions` generated from `primary_action`.
- Update MCP tool response docs and plugin skill.

### P0：Patience hard gate

- Track meaningful progress timestamps:
  - `UserPromptSubmit`;
  - PreToolUse/PostToolUse/PostToolBatch;
  - report_requested;
  - report progress.
- Add patience projection.
- In `mcp_session_send`, reject premature `request_report` / `continue` with `inside_patience_window`.
- Allow override only with `user_explicit_close=true`.

### P1：Prompt commit self-heal

- After route start/prompt gate creation, daemon sends one bounded prompt commit automatically when PTY actor is ready.
- Keep `submit_pending_prompt` as debug/recovery, not primary.
- Add tests for:
  - prompt pending primary action is wait;
  - self-heal attempt recorded;
  - unacknowledged state does not suggest report request.

### P1：Read/Write policy split

- Introduce route model fields: `reference_paths`, `write_paths`, `read_policy`.
- Map current `allowed_paths` to write paths.
- Update hook policy:
  - Read/Grep/Glob default unrestricted;
  - strict modes can enforce target/reference read;
  - write tools remain hard-bounded.
- Keep Bash readonly-only.

### P1：Policy block recovery taxonomy

- Classify denials into block kinds.
- Add suggested recovery fields:
  - `suggested_reference_paths`;
  - `suggested_write_paths`;
  - `reroute_args`;
  - `worker_instruction`.
- Make stop/interrupt debug/destructive unless repeated denial or explicit reclaim.

### P2：Docs / Skills / Smoke

- Update `docs/agentcall-protocol.md` and generated supervisor skill.
- Update `plugins/agentcall/skills/agentcall/SKILL.md`.
- Update README / CHANGELOG.
- Add smoke scenario:
  - three workers;
  - healthy working hides `request_report`;
  - premature `request_report` refused;
  - reference path read allowed by default;
  - write outside allowed path still denied.

## 7. Test Plan

### Unit

- `Working` state projects:
  - `primary_action.kind=wait`;
  - `next_action=wait`;
  - no default `request_report`.
- `request_report` inside initial 60s returns `inside_patience_window`.
- `request_report` with `user_explicit_close=true` succeeds.
- duplicate `request_report` returns existing request.
- `prompt_pending` primary action is wait; `submit_pending_prompt` appears only in debug/recovery.
- `Read/Grep/Glob` outside target workspace allowed under default `read_policy=unrestricted`.
- `Read/Grep/Glob` outside target/reference denied under `read_policy=strict`.
- `Write/Edit/MultiEdit` outside write paths still denied.
- Bash write attempts still denied.
- missing reference path in strict mode maps to `blocked_by_missing_reference_path`.

### Integration / Smoke

- Start three PTY workers for architecture review.
- Confirm after prompt submit:
  - board shows workers as healthy or hidden from attention;
  - summary primary action is wait for at least 60 seconds;
  - premature `request_report` refused.
- Run a Vic2-style task requiring external read reference:
  - default read policy does not block reading external Vic2 path;
  - report still writes only to report path.
- Strict read mode:
  - external read blocked;
  - summary shows `blocked_by_missing_reference_path`;
  - primary action suggests reroute with reference paths, not stop.

### Regression

```powershell
cargo test --workspace
python -m pytest -q
python agentcall.py release-check
python agentcall.py smoke real-worker --parallel-workers 6
```

## 8. Non-goals / 禁止项

- 不回改 `docs/v6.2-code-plan.md`。
- 不恢复 ACP/SDK 作为默认路线。
- 不扩大 MCP 默认工具面。
- 不让 Codex 手写 lease/precondition/idempotency。
- 不把 Python 作为 live state writer。
- 不让 raw PTY/TUI 成为默认状态源。
- 不自动 kill visible PTY worker。
- 不默认限制本地 Read/Grep/Glob。
- 不放松 Write/Edit/MultiEdit 写边界。
- 不允许 Bash 写绕过 write policy。
- 不把 `request_report` 作为健康 `working` 的默认建议动作。
- 不把 `submit_pending_prompt` 作为正常 route 后的默认 Codex 操作。

## 9. Acceptance Criteria

- Healthy `working` summary has exactly one normal primary action: `wait`.
- Codex-facing compact/default summary no longer lists `request_report` for healthy working workers.
- Premature `request_report` returns `inside_patience_window` with remaining wait seconds.
- `user_explicit_close=true` can explicitly bypass patience for report closure.
- Prompt pending path performs one daemon-owned self-heal attempt before asking Codex for debug recovery.
- Default read policy allows external read references without path denial.
- Write tools remain bounded to write paths, scratch, and report path.
- Policy block projections classify missing reference path separately from write denial and repeated denial.
- Board attention does not show healthy workers inside patience window.
- v0.3/v0.4/v0.5 style architecture-review workflow can run without 10-second premature report requests or immediate stop on external reference reads.

---

## 10. Assumptions

- AgentCall remains local single-user tooling in v6.3.
- The main practical safety boundary is write containment, not default read containment.
- Sensitive read leakage is handled by redaction, payload budgets, and optional strict read mode.
- Existing `allowed_paths` callers can be mapped to write boundaries without teaching Codex new low-level safety fields in normal flow.
- Codex follows `primary_action` more reliably than a wide `next_actions` menu.
