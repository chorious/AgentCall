# AgentCall v2.0 Architecture

AgentCall v2 is a lifecycle-bounded multi-agent orchestration runtime.

The core principle:

```text
Parent owns context.
Child owns one execution burst.
Report is the contract.
```

## Requirements

1. A parent command can call Claude through an ACP/SDK driver.
2. Every child lifecycle must finish with a structured report.
3. Child work is block-cut: one invocation, one role, one mode, one report.
4. Parent validates stage outputs, organizes project state, and rejects drift.
5. When parent confidence is low, it delegates a bounded review child.
6. SOP behavior lives in code instead of repeated prompt text.

## Runtime Layers

```text
AgentCall Runtime
  ParentOrchestrator
    ChildCallSpec
    ReportValidator
    ScopeValidator
    AcceptancePolicy
  Agent Drivers
    ClaudeACPDriver
    HeadlessJsonClaudeDriver
    FunctionAgentDriver
    PtyAttachDriver (future fallback)
  Store
    task.json
    report.md
    reports/*.json
    events.ndjson
```

## Current Implementation

The Python v2 package currently implements:

- `ChildCallSpec`: bounded child invocation contract.
- `ChildReport`: structured report contract.
- `REPORT_JSON_SCHEMA`: code-owned report contract used by live drivers.
- `ParentOrchestrator`: planner -> executor -> optional reviewer lifecycle.
- `FunctionAgentDriver`: deterministic driver for tests and simulations.
- `AcpClaudeDriver`: ACP stdio JSON-RPC driver with tested transport.
- `AcpStdioClient`: initializes ACP, creates sessions, sets mode, sends prompts,
  records `session/update`, and answers permission requests.
- `AgentSnapshot`: state events for future UI visualization.
- `workflow simulate`: small-project lifecycle simulation with `scripted`,
  `headless-json`, or `acp` child drivers.
- `workflow inspect`: text cockpit for reports, review status, and state events.

The ACP transport is tested against a fake stdio ACP agent. Live Claude ACP
execution is not used in the simulation yet; the next step is to run the same
driver against `npx -y @agentclientprotocol/claude-agent-acp` and stream its
events through the Rust daemon.

## Acceptance Semantics

Clean acceptance is an event and task status transition:

```text
parent.accepted
task.status = accepted
```

`review.md` is only written when findings require revision, blocker handling,
or a dispute record. A reviewer child may run and return no findings without
creating a review artifact.

## Small-Project Simulation

Run:

```powershell
$env:PYTHONPATH='src'
python -m agentcall --root .agentcall-demo workflow simulate
python -m agentcall --root .agentcall-demo workflow inspect task-0001
```

Expected output:

```text
task_id: task-0001
status: accepted
reports: 2
```

The simulation creates a tiny calculator bug, runs:

1. Planner child in `plan` mode.
2. Executor child in `execute` mode.
3. Parent validation of report contract, lifecycle limit, allowed paths, and tests.
4. Acceptance event without `review.md`.

The runtime also appends `agent.state_changed` events so a UI can render
starting/running/reported/needs-review states without reading terminal output.

The parent also validates that reported `changed_files` are workspace-relative,
remain inside allowed scope, and exist on disk. This keeps the report contract
from becoming a purely narrative claim.

Headless Claude calls pass the same report schema via `--json-schema`; ACP calls
include it in the bounded child prompt and validate the returned JSON before the
parent accepts it.
