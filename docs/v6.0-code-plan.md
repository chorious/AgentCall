# AgentCall v6.0 Code Plan: Slim Codex Control Plane + Reliable PTY Worker Delivery

Date: 2026-06-11
Status: Proposed full-version consolidation plan
Scope: `main` branch after v5.4/v5.5 work, plus the real stress/single-flow reports and prior code-review findings.

## 0. Executive Summary

v6.0 is not another incremental hardening pass. It is a product-level simplification pass.

The core problem is now clear: AgentCall's daemon can own state, run PTY workers, ingest hooks, enforce leases, and recover sessions, but the Codex-facing surface has grown too heavy. Codex currently has to reason about route status, session liveness, attention state, prompt gates, control tokens, preconditions, lease ids, generations, idempotency, cwd/workspace split, historical projections, TUI screen, event raw/debug, policy blocks, reports, and command status. That is too much.

v6.0 should make the daemon absorb that complexity and expose a much smaller contract:

```text
Board -> Worker summary -> Next action -> Action result -> Report
```

Codex should see four things:

1. Which workers exist or need attention.
2. What state each worker is in.
3. Why that state matters.
4. What action is safe next.

Everything else is daemon-internal or debug-only.

This plan intentionally folds prior report findings and code-review feedback into one complete optimization pass. Do not split it into P0/P1/P2. The acceptance criteria are version-level: v6.0 is not done until the Codex-facing control loop is small, reliable, and tested under real PTY pressure.

## 1. Product Target

### 1.1 New Recommended Codex Loop

Codex should only need this loop:

```text
agentcall_daemon(action=status)
agentcall_board(view=compact, filter=attention)
agentcall_route(objective, workspace, allowed_paths, report_path, read_only, session_name)
agentcall_session(name)
agentcall_session_send(name, action=<one of returned next_actions>)
agentcall_report(action=accept, session_id=...)
```

The most important v6.0 rule:

> Codex should not infer daemon internals. Daemon must return the next safe action explicitly.

### 1.2 New Worker State Model

Replace Codex-facing combinations of `liveness_status`, `attention_status`, `turn_status`, route status, and prompt gate with one top-level state enum.

```text
starting
prompt_pending
prompt_missing
working
waiting_input
needs_permission
blocked_by_policy
report_ready
stopping
done
failed
```

The underlying projection may keep richer fields. The default summary must expose only the normalized state and enough explanation to act.

Example:

```json
{
  "schema_version": 2,
  "worker": "stress-v60-control",
  "state": "prompt_missing",
  "why": "Route prompt was written to the PTY but UserPromptSubmit was not observed before the ack deadline.",
  "can_wait": false,
  "next_actions": [
    {
      "kind": "submit_pending_prompt",
      "tool": "agentcall_session_send",
      "args": {"name": "stress-v60-control", "action": "submit_pending_prompt"}
    },
    {
      "kind": "stop",
      "tool": "agentcall_session_send",
      "args": {"name": "stress-v60-control", "action": "stop"},
      "requires_control_token": true
    }
  ],
  "report": {"ready": false},
  "debug_refs": {
    "events": {"view": "events"},
    "screen": {"view": "debug", "include": ["screen"]}
  }
}
```

### 1.3 What Codex Must No Longer See by Default

Hide from recommended MCP schemas and default summaries:

```text
owner_id
owner_lease_id
lease_generation
precondition
projection_last_session_seq
idempotency_key
control_epoch_reason
last_command_status

pty_workflow
persist_context
```

These can remain in debug/raw surfaces for maintainers. They must not be part of the normal Codex loop.

## 2. Source Problems v6.0 Must Resolve

### 2.1 Route Prompt Delivery Is Not Reliable

Stress and single-flow tests proved that `pty.input_sent` and `command.completed` do not mean Claude accepted a user turn. Six PTY workers started, but none reached `UserPromptSubmit`; their route prompts stayed in the Claude input box. MCP `send/continue` then queued instructions because the projection said `working/none`, which provided no recovery path. Direct daemon HTTP input could submit the stuck prompt, proving the PTY worker path is viable but the MCP prompt delivery contract is not.

v6.0 must make prompt delivery first-class:

```text
input_sent_by_actor != task_started
SessionStart != task_started
UserPromptSubmit == task_started
```

### 2.2 Safety Was Implemented at the Wrong Layer

v5.5 correctly moved toward daemon-minted control tokens. v6.0 must finish the simplification: Codex sends intent; daemon mints envelopes, attaches leases, dedupes, validates control freshness, and returns structured refusals.

Codex must not hand-build:

```text
owner_lease_id
lease_generation
projection_last_session_seq
idempotency_key
```

### 2.3 Session Observation Was Too Wide

Earlier `agentcall_session` responses mixed summary, clean tail, events, raw hook payloads, prompts, stdout, and Write content in one payload. v5.4 introduced views, but v6.0 must lock down the contract: default summary is small and actionable; TUI/events/debug/raw are explicit drilldowns.

### 2.4 Board Mixed Runtime Truth and History

`live_daemon_sessions` must only represent in-memory daemon-owned live PTY sessions. Historical projections and route records must not masquerade as live workers.

### 2.5 Projection Had Product-Lying States

Known examples:

- `SubagentStop` could create false `checkpoint_due`.
- Terminal sessions could be revived as `working` by late `command.awaiting_observation`.
- Multiple denied writes could still project `attention=none`.
- Route prompt missing could still project `working/none`.
- `workspace` could be overwritten by Claude process cwd (`D:\guKimi`) instead of task target workspace.

v6.0 must project truth, not last event noise.

### 2.6 Logs and Raw Payloads Still Leak Too Much

Hook payloads can contain `tool_input.command`, environment assignments, API keys, prompt text, and Write content. Redaction and payload budgeting must be applied before events/logs/MCP responses become routine debugging surfaces.

### 3.4 `agentcall_session_send`

#### Recommended Schema

```json
{
  "name": "worker-a",
  "action": "continue|request_report|submit_pending_prompt|select_option|interrupt|stop|kill",
  "text": "...",
  "choice": "1",
  "control_token": "ctl_..."
}
```

#### Remove from Recommended Schema

```text
owner_id
owner_lease_id
lease_generation
precondition
idempotency_key
enter
```

Legacy debug compatibility may continue for one release behind explicit debug flag or raw HTTP API. Codex docs and plugin schema must stop teaching these fields.

#### Action Semantics

```text
continue
  Benign. Daemon decides whether to queue or direct-send based on worker_state.

request_report
  Benign. If worker is mid-turn, queue through hook context. If waiting_input, direct-send.

submit_pending_prompt
  Restricted. Commits existing Claude input prompt. Does not resend full prompt.

select_option
  Only valid for structured permission/menu interaction.

interrupt
  Requires control token. Used for drift/blocker/reclaim.

stop
  Requires control token. Graceful stop request; lease released only after observed exit.

kill
  Requires control token. Process-tree termination; debug/escalation action.
```

### 3.5 `agentcall_report`

#### Recommended Response

```json
{
  "ok": true,
  "status": "accepted",
  "worker": "worker-a",
  "report_path": ".agentcall/reports/worker-a.md",
  "validation": {
    "exists": true,
    "non_empty": true,
    "matched_route_report_path": true,
    "daemon_observed_write": true
  },
  "confidence": "high",
  "next_action": "stop_worker"
}
```

Never return bare `[]` for accept. Report accept does not release lease. It only validates and marks delivery status.

## 4. Internal Architecture Changes

### 4.1 Add `worker_state.rs`

Create:

```text
crates/agentcall-daemon/src/worker_state.rs
```

Responsibilities:

- Normalize projection + route + runtime + prompt gate + policy block into one Codex-facing state.
- Produce `why`, `can_wait`, and `next_actions`.
- Provide a single mapping used by board, session summary, and report guidance.

Suggested types:

```rust
pub(crate) enum WorkerStateKind {
    Starting,
    PromptPending,
    PromptMissing,
    Working,
    WaitingInput,
    NeedsPermission,
    BlockedByPolicy,
    ReportReady,
    Stopping,
    Done,
    Failed,
}

pub(crate) struct WorkerStateView {
    pub(crate) worker: String,
    pub(crate) state: WorkerStateKind,
    pub(crate) why: String,
    pub(crate) can_wait: bool,
    pub(crate) next_actions: Vec<NextAction>,
    pub(crate) report: ReportView,
    pub(crate) debug_refs: DebugRefs,
}
```

Priority order:

```text
failed/orphaned/terminal abnormal
prompt_missing
needs_permission
blocked_by_policy
report_ready
waiting_input
stopping/killing
working
prompt_pending
starting
done
```

Rules:

- `SessionStart` alone is not `working`; route prompt must reach `UserPromptSubmit`.
- `pty.input_sent` is not `task_started`.
- If route prompt gate exceeded deadline without `UserPromptSubmit`, state is `prompt_missing`.
- Multiple denied Write/Bash events in same active worker state -> `blocked_by_policy`, even if each target repeat count is 1.
- Terminal events dominate late command bookkeeping.
- `SubagentStop` is benign unless explicit checkpoint contract exists.

### 4.2 Prompt Gate State Machine

Create or centralize in:

```text
crates/agentcall-daemon/src/prompt_gate.rs
```

States:

```text
registered
waiting_claude_ready
prompt_written
submit_attempted
user_prompt_submitted
ack_missing
submit_failed
aborted_session_exited
```

Stored in route record:

```json
{
  "prompt_gate": {
    "state": "prompt_written",
    "prompt_id": "route_prompt:route-123:worker-a",
    "expected_hook": "UserPromptSubmit",
    "task_started": false,
    "ack_deadline_ms": 15000,
    "submit_attempts": 0,
    "self_heal_attempted": false,
    "last_event_id": "evt-..."
  }
}
```

Events:

- `pty.session_started` -> route can enter `waiting_claude_ready`.
- `hook.SessionStart` -> Claude ready.
- prompt command written -> `prompt_written`.
- commit enter/space-enter sent -> `submit_attempted`.
- `hook.UserPromptSubmit` -> `user_prompt_submitted`, task_started=true.
- deadline exceeded -> `ack_missing`, worker_state=`prompt_missing`.
- `pty.session_ended` before submit -> `aborted_session_exited`.

### 4.3 File-Backed Handoff Prompt

Long route prompts should not be typed wholesale into Claude's TUI input area.

Instead:

1. Daemon writes full handoff file:

```text
.agentcall/tasks/<route_id>/prompt.md
```

2. PTY receives a short prompt:

```text
AgentCall handoff: read and follow <absolute-or-clearly-relative-prompt.md>. Write the final report to <report_path>.
```

3. The handoff file contains:

- objective;
- target workspace;
- allowed paths;
- report path;
- acceptance criteria;
- read/write policy;
- scratch root;
- stop/report instructions.

Benefits:

- Avoids multi-line giant prompt stuck in input box.
- Reduces hook raw prompt payload.
- Improves prompt submission reliability.
- Makes route input idempotent and inspectable.

Acceptance:

- No default route prompt should exceed a small line/byte threshold, e.g. 512-1000 bytes.
- Full details go into handoff file.
- Handoff file is inside daemon workspace and readable by Claude process.

### 4.4 `submit_pending_prompt` Action

Add command type:

```rust
CommandType::SubmitPendingPrompt
```

or map action to a guarded `SendInput` with dedicated payload:

```json
{
  "action": "submit_pending_prompt",
  "commit": "space_enter",
  "prompt_id": "route_prompt:route-123:worker-a"
}
```

Behavior:

- Does not send the full prompt again.
- Sends only commit keystroke: Enter or bounded `" " + Enter`.
- Requires prompt gate state `prompt_written` or `ack_missing`.
- Requires no `UserPromptSubmit` yet.
- Requires live PTY session.
- Requires control token if prompt gate is stale or manual supervisor action is needed.
- Dedupe by prompt id.

Structured success:

```json
{
  "ok": true,
  "status": "pending_prompt_commit_sent",
  "awaiting": "UserPromptSubmit",
  "prompt_gate": {
    "state": "submit_attempted",
    "submit_attempts": 1
  }
}
```

Structured refusal:

```json
{
  "ok": false,
  "status": "not_in_prompt_gate",
  "reason": "Worker already has UserPromptSubmit or is mid-turn.",
  "next_step": "refresh agentcall_session"
}
```

### 4.5 Daemon-Minted Control Remains Internal

Keep `control.rs`, but simplify public exposure.

Current token claims may keep:

```text
session_id
owner_id
owner_lease_id
lease_generation
control_epoch
allowed_actions
expires_at
```

But Codex should see only:

```json
{
  "token": "ctl_...",
  "expires_at": "...",
  "token_required_for": ["interrupt", "stop", "kill"]
}
```

Do not expose owner lease details in default summary. They may appear in debug.

Actor revalidation remains important:

- MCP handler validates token before building command.
- Actor validates command envelope again before destructive effect.

### 4.6 Idempotency Must Be Automatic

Daemon/MCP must generate idempotency by action class.

Recommended rules:

```text
route prompt
  route_prompt:<route_id>:<session_name>

submit_pending_prompt
  submit_pending_prompt:<route_id>:<session_name>:<prompt_id>

continue
  continue:<session>:<worker_state_epoch>

request_report
  request_report:<session>:<worker_state_epoch>

select_option
  select_option:<session>:<interaction_id>:<choice>

send explicit text
  mcp_request_id if available; otherwise generated unique key with retry-window dedupe

stop/kill/interrupt
  action:<session>:<control_epoch>:<token_hash>:<action>
```

Codex should never be required to provide `idempotency_key` in normal flow.

## 5. Projection Rules

### 5.1 Worker State Truth Rules

Implement in projection reducer and `worker_state.rs`.

#### Prompt Truth

```text
pty.input_sent       -> prompt bytes written only
command.completed    -> command dispatch completed only
hook.SessionStart    -> Claude session ready only
hook.UserPromptSubmit -> task/user turn accepted
```

Without `UserPromptSubmit`, route worker cannot be `working` after deadline.

#### Terminal Dominance

Once a terminal event is observed:

```text
pty.session_ended
process.exited
session.actor_failed
session.writer_failed
session.orphaned
```

Later command bookkeeping may update `last_command_status`, but must not change worker state back to `working`.

#### SubagentStop

`SubagentStop` is benign unless:

- route has explicit checkpoint contract; or
- hook payload has `checkpoint_request=true`; or
- hook payload has `agentcall_checkpoint=true`.

Otherwise it must not create attention.

#### Policy Denial

If hook decision has `allowed=false`:

- worker state becomes `blocked_by_policy`.
- `can_wait=false`.
- next actions include `inspect_policy`, `request_report`, `interrupt`, or `stop` depending on context.

Multiple distinct denied writes within a short window count as blocked, even if per-target repeat count is 1.

#### Permission

Permission prompt must produce:

```json
{
  "state": "needs_permission",
  "next_actions": [
    {"kind": "select_option", "choice_required": true}
  ]
}
```

Do not recommend `continue` or natural language send for permission prompt.

#### Report Ready

Report write to route report path produces:

```text
state=report_ready
next_action=accept_report
```

Report accept does not automatically release lease.

### 5.2 Workspace and CWD Truth

Projection must maintain distinct fields:

```text
target_workspace   Task workspace from route
claude_cwd         Claude process cwd / configured claude_workspace
process_cwd        Alias or precise process cwd
scratch_root       AgentCall scratch root for this worker
report_path        Route report path
```

Rules:

- `hook.SessionStart.cwd` updates `claude_cwd`, not `target_workspace`.
- `route.workspace` remains `target_workspace`.
- Summary displays `workspace` as target workspace if retaining legacy field.
- TUI/debug can show all roots.

## 6. Path Policy and Binding

Keep path policy strong, hide detail by default.

### 6.1 Binding

Route worker without reliable wrapper binding:

- no write/edit/bash-write;
- state should surface `binding_untrusted` or `blocked_by_policy`;
- debug may show why.

No guessing by cwd, PID, or first new session.

### 6.2 Canonical Roots

At route creation, materialize and store:

```json
{
  "roots": {
    "target_workspace": "E:\\Project\\AgentCall",
    "claude_cwd": "D:\\guKimi",
    "scratch_root": "D:\\guKimi\\.agentcall\\workspaces\\worker-a"
  },
  "writable_roots": [
    {"kind": "scratch", "abs": "..."},
    {"kind": "report", "abs": "..."},
    {"kind": "allowed_path", "abs": "..."}
  ]
}
```

Hook policy compares absolute normalized paths to absolute writable roots.

### 6.3 Default Exposure

Summary:

```json
{
  "state": "blocked_by_policy",
  "why": "Write denied outside allowed paths.",
  "next_action": "inspect_policy_or_stop"
}
```

Debug/TUI:

```text
path_diagnosis
compared roots
target path
policy block
```

## 8. Lifecycle and Command Closure

### 8.1 Stop vs Kill

Keep separate semantics:

```text
stop -> graceful/normal stop request, lease_release pending observed exit
kill -> force process-tree termination, lease_release pending observed exit
```

Lease release only after waiter observes process exit.

### 8.2 Command Status Closure

For commands marked `awaiting_observation`, close them when observation arrives.

Example:

```json
{
  "command_id": "cmd-stop-...",
  "status": "observed_completed",
  "observed_event": "pty.session_ended",
  "observed_event_id": "evt-...",
  "result": "session_exited"
}
```

Do not leave stop commands permanently `accepted` when session has exited and leases are released.

### 8.3 Route Closure

When session exits:

```text
route.status=session_exited
worker_state=done/stopped/failed
prompt_gate.state=aborted_session_exited if no UserPromptSubmit
```

No route should remain `workflow_status=running` if its live session is gone.

## 9. HTTP and Debug Boundary

### 9.1 Public vs Internal HTTP

Public/local UI:

```text
GET /board
GET /api/board
GET /api/runtime/health
```

Internal MCP:

```text
POST /api/mcp/call
```

Hook ingest:

```text
POST /api/hooks/ingest
```

Debug/manual:

```text
/api/sessions/*
/api/context
/api/transcripts/index
```

Direct HTTP input remains debug/manual. It must not be the recommended Codex rescue path. `submit_pending_prompt` is the product path.

### 9.2 Token Boundary

Keep:

```text
daemon_token -> local HTTP/API auth
control_token -> session high-risk action freshness
idempotency_key -> duplicate side-effect prevention
```

Do not merge these concepts.

## 10. TUI / Terminal Screen

The vt100 screen layer is good and should stay debug-oriented.

Rules:

- Summary must not include terminal screen.
- TUI view is for human dashboard, not default Codex reasoning.
- Debug view can include screen snapshot and deprecated clean tail.
- Terminal screen never drives `worker_state`; it only helps explain or recover prompt states.

Allowed use of screen in v6.0:

- Detect whether prompt text appears to be stuck in input area for `submit_pending_prompt` gate.
- Provide debug snapshot.
- Preserve permission menus in TUI.

Do not infer final worker success/failure from screen text.

## 14. Acceptance Criteria

v6.0 is complete only if all of the following are true:

### Codex Surface

- Codex no longer needs to provide `owner_lease_id`, `lease_generation`, `precondition`, or `idempotency_key` in normal MCP flow.
- Default `agentcall_session(name)` returns one normalized worker state, why, next actions, report info, and optional control token.
- Default board lists only current workers/attention, not history.
- SDK/runtime estimation knobs are absent from recommended route schema.

### Prompt Delivery

- `input_sent_by_actor` is never treated as task started.
- `SessionStart` is never treated as task started.
- Missing `UserPromptSubmit` becomes `prompt_missing`, not `working/none`.
- `submit_pending_prompt` can recover a stuck prompt via MCP.
- Full route handoff lives in a file; typed prompt is short.

### Safety

- Destructive/phase-changing actions use daemon-minted control token.
- Benign actions auto-attach lease and auto-generate idempotency.
- Actor still revalidates destructive commands before effect.
- Path containment and binding fail-closed remain strong.

### Projection Truth

- Terminal sessions cannot be revived by later command events.
- SubagentStop is benign unless explicit checkpoint contract exists.
- Policy denials fold into blocked_by_policy.
- Target workspace and Claude cwd are distinct.

### Lifecycle

- Stop/kill release leases only after observed exit.
- Stop/kill command status closes after observed exit.
- Route status closes when session exits.
- Report accept returns structured validation and does not release lease.

### Payload / Debug

- Summary <= 8 KB.
- TUI <= 20 KB.
- Compact events <= 48 KB.
- Raw prompt/tool output/Write content absent from compact views.
- Hook command/env secrets are redacted.
- Debug/raw surfaces remain available explicitly.

### Stress

- Single worker full lifecycle passes via MCP only.
- Six concurrent workers never silently stall as `working/none` before UserPromptSubmit.
- All sessions and leases return to baseline after stop.

## 16. Codex Implementation Guardrails

For local Codex implementing this plan:

1. Do not add new public MCP tools unless absolutely necessary. Prefer simplifying existing tools.
2. Do not expose daemon internals to solve Codex confusion.
3. Do not make `raw_input` a product action. Implement `submit_pending_prompt` only under prompt gate.
4. Do not classify `SessionStart` as worker progress beyond Claude readiness.
5. Do not retype full route prompt on retry.
6. Do not remove path policy, binding, or leases. Hide them from Codex, but keep daemon enforcement.
7. Do not let tests pass by weakening attention states. If worker is stuck, say stuck.
8. Do not use terminal screen as state truth. Use hooks/projection/prompt gate.
9. Do not return bare arrays for report accept.
10. Do not log raw secrets, raw commands, or Write content in compact surfaces.

The north star: every default Codex call should answer one question—**what should I safely do next?**
