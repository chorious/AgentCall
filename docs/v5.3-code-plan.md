# AgentCall v5.3 Code Plan: Hard Gate Closure + Real-Run Feedback

## Summary

v5.3 的目标是给 v5.0-v5.2 做真正收口，而不是继续扩展新能力。

本轮同时处理两类问题：

1. v5.0-v5.2 plan 已经明确要求、但实现审计确认没有闭合的 hard gates。
2. 刚才真实拉起两个 AgentCall review worker 后暴露出的运行问题：report 已写但 projection 不认、worker 完成后仍 waiting、MCP/log 路径仍重、只读审查 worker 会自行拉 subagent 并漂移。

一句话：

```text
v5.3 让 AgentCall 从“集成里程碑”变成“可签收的受控 worker 底座”。
```

v5.3 完成后，才能把 v5.0-v5.2 标为 closed。否则当前状态仍应称为 integration milestone。

## Non-goals

- 不引入新 runtime。
- 不恢复 ACP 作为默认路线。
- 不新增 MCP canonical 工具名。
- 不重写整个 daemon 架构。
- 不把所有 side-index JSON 一次性迁入 SQLite。
- 不实现分布式队列。
- 不让 PTY worker 自动越权修代码。
- 不把 raw PTY / transcript / hook artifacts 塞进 board 默认返回。
- 不把 Claude 自然语言当成报告完成的唯一证据。
- 不依赖 TUI 文本判断 worker 完成。

## P0: Close v5.0/v5.1/v5.2 Hard Gates

### P0.1 Precondition Seq Validation

当前问题：

- `CommandPrecondition.projection_last_session_seq` 已解析。
- 但没有和当前 projection seq 比较。
- `rejected_precondition` 在源码中不存在。
- 这让 stale send 防线成为空壳。

实现要求：

- 在命令进入 actor 执行前校验 `precondition.projection_last_session_seq`。
- 建议位置：
  - `actor.rs::submit_session_command` 或 `actor.rs::execute_command` 前置校验。
  - 可新增 `commands.rs::validate_command_precondition(...)`。
- 读取当前 session projection 的 `projection_last_session_seq`。
- 当 provided seq 与 current seq 不一致时：
  - command 不得写入 PTY。
  - 返回结构化错误 `rejected_precondition`。
  - 写 command rejected event。
  - 不改变 session liveness 为 working。

验收测试：

- `session_send_precondition_seq_mismatch_rejected`
- `projection_seq_precondition_prevents_stale_send`
- `precondition_match_allows_command`
- `missing_precondition_for_destructive_command_rejected`

### P0.2 Actor Failure / Writer Closed / Orphaned Projection

当前问题：

- actor loop panic 会静默结束。
- writer write error 只返回给 caller，不改变 projection。
- reader I/O 错误会静默 break。
- session 可能继续显示 running / working。

实现要求：

- actor loop 必须有 failure guard。
- actor panic、channel closed、writer closed、writer write error 都必须投影为失败态。
- 增加统一状态：

```text
liveness_status = failed_or_orphaned
attention_status = failed
needs_attention = true
last_error_brief = <原因>
```

- 写入事件：
  - `session.actor_failed`
  - `session.writer_closed`
  - `session.reader_failed`
  - `session.orphaned`

- daemon 启动时扫描 runtime state：
  - 如果 projection/active_sessions 里有 running session，但 daemon 没有 child ownership，标记 `orphaned_after_daemon_restart`。
  - 不得继续投影为 live daemon session。

验收测试：

- `actor_panic_marks_session_failed_or_orphaned`
- `writer_closed_marks_session_failed`
- `reader_error_marks_session_failed`
- `orphaned_projection_not_reported_alive`
- `daemon_restart_marks_unowned_running_session_orphaned`

### P0.3 Control Channel Priority And RawWrite Isolation

当前问题：

- `ActorControlCommand::RawWrite(Vec<u8>)` 仍在 control inbox。
- actor 只有一条 unbounded mpsc。
- stop / interrupt / kill 没有优先级保证。
- 类型层仍违反 v5.1 hard correction。

实现要求：

- 从 `ActorControlCommand` 中移除 `RawWrite(Vec<u8>)`。
- 光标位置回复等小字节写入走独立 path：
  - `ActorAuxWriteCommand`
  - 或 `PtyAuxWriter`
  - 不得与 user command / stop / interrupt 竞争同一 control queue。

- 将 actor 控制命令至少拆成优先级：

```text
high priority:
  kill
  stop
  interrupt

normal priority:
  send
  select_option
  report_request
  continue
```

- 如果继续使用 std channel，也必须在 actor 内优先 drain high-priority queue。
- stop / interrupt 不得被大量 queued send 阻塞。

验收测试：

- `control_inbox_does_not_carry_raw_pty_bytes`
- `stop_command_has_priority_over_queued_submit`
- `interrupt_command_has_priority_over_queued_submit`
- `high_volume_aux_output_does_not_delay_interrupt`

## P1: Real-Run Report Completion Closure

### P1.1 Report Write Must Update Projection

真实问题：

- 两个 worker 都写出了：
  - `reports/review_v52_functional_implementation.md`
  - `reports/review_v52_code_robustness.md`
- Hook events 捕获到 `Write`。
- 但 board/session summary 仍显示：
  - `report_ready=false`
  - `files_written_count=0`
  - `waiting_input`

实现要求：

- 当 hook 看到 `Write/Edit/MultiEdit` 命中 route/context packet 的 `report_path`：
  - 更新 session projection：

```json
{
  "report_ready": true,
  "report_source": "hook_write",
  "reports": ["<report_path>"],
  "files_written_count": 1,
  "last_report_path": "<report_path>"
}
```

- report path 必须从 route context packet / active session metadata 中读取，不靠 TUI。
- 如果 report 文件存在且路径匹配，即使 Claude 没说“完成”，board 也应显示 report ready。
- `agentcall_report(action=accept)` 应能读取该 report 并计算 ConfidenceLedger。

验收测试：

- `hook_write_to_report_path_marks_report_ready`
- `report_ready_does_not_depend_on_tui_text`
- `agentcall_report_accept_reads_hook_written_report`

### P1.2 Worker Done State After Report

真实问题：

- worker 写出 report 后仍停在 `waiting_input`。
- board attention 继续提示需要介入，但这其实是“可验收/可回收”状态。

实现要求：

- 新增或明确区分：

```text
liveness_status:
  working | idle | waiting_input | completed | failed_or_orphaned

delivery_status:
  none | report_ready | report_accepted | report_failed_contract

attention_status:
  none | waiting_input | needs_permission | blocked_by_policy | failed | report_ready
```

- 对 utility worker：
  - 写出有效 report 后，默认 attention 应为 `report_ready`，不是普通 `waiting_input`。
  - `required_next_step = accept_report_or_stop_worker`。
  - 如果 report 被 accept，可以释放 workspace/owner lease，并允许 stop。

验收测试：

- `utility_worker_report_ready_beats_waiting_input`
- `accepted_report_releases_worker_lease`
- `completed_worker_not_listed_as_attention_waiting_input`

### P1.3 Read-only Worker Must Not Spawn Implementation Subagents

真实问题：

- `review-v52-code-robustness` 是只读审查 worker。
- 它内部调用了 `TaskCreate`，SubagentStop 的 `last_assistant_message` 出现 `fix the top 5 priority issues in the report`。
- 虽然没有改生产代码，但这说明只读审查任务会漂移成“修复任务”语义。

实现要求：

- route/read_only worker 的 hook policy 增加 TaskCreate 约束。
- 当 containment mode 为 `read_only`：
  - 允许 TaskCreate 仅用于 review/research/report 子任务。
  - 如果 TaskCreate description/activeForm/subject 包含 fix/implement/modify/write code 等实现语义，deny 或标记 drift。
- 首版可先 conservative：
  - read_only worker 默认禁止 TaskCreate。
  - 需要允许时必须显式 `allow_subagents=true`。

推荐首版：

```text
read_only + allow_subagents=false:
  TaskCreate => deny with reason read_only_subagent_disabled
```

验收测试：

- `read_only_route_denies_taskcreate_by_default`
- `read_only_route_can_allow_review_subagent_explicitly`
- `taskcreate_drift_sets_attention_status`

## P2: MCP And Log Hot Path From Real Slowdowns

### P2.1 Board/Session Must Not Parse Huge Logs On Default Path

真实问题：

- `.agentcall/events/recent.ndjson` 接近 2MB。
- `PostToolUse/recent.ndjson` 超过 3MB。
- full board / session operations 仍可能读取大文件并 JSON parse。

实现要求：

- `agentcall_board(view=compact, filter=attention)` 只读 projection store。
- `agentcall_session` 默认只读 projection。
- 任何 include raw/clean/events/plan 的请求必须：
  - 有独立 limit。
  - 有 byte cap。
  - 返回 `truncated=true`。

- 禁止 default MCP response 返回：
  - full route history。
  - full legacy detached sessions。
  - raw event payload。
  - hook tool_response。

验收测试：

- `compact_board_does_not_call_read_events`
- `default_session_does_not_read_transcript_or_events`
- `full_board_caps_events_and_routes`
- `mcp_response_under_128kb_for_busy_workspace`

### P2.2 Replace Event Linear Scan With Projection Flags

真实问题：

- `session_has_seen_hook_event` 每次扫描 events。
- `last_supervisor_instruction_injected_at` 也扫描 events。
- 活跃 worker 越跑越慢。

实现要求：

- 在 projection 中维护 hot flags：

```json
{
  "seen_hooks": {
    "UserPromptSubmit": true,
    "PostToolBatch": true,
    "Write": true
  },
  "last_hook_at": "...",
  "last_post_tool_batch_at": "...",
  "last_supervisor_instruction_injected_at": "...",
  "last_report_write_at": "..."
}
```

- `mcp_session_send` 不再通过 events 扫描判断是否 seen hook。
- summary 不再通过 events 扫描查 injection 时间。

验收测试：

- `session_send_uses_projection_seen_hook_flag`
- `last_injected_at_from_projection_not_event_scan`
- `busy_event_log_does_not_slow_session_send`

### P2.3 Log Rotation Must Enforce Real Per-file Caps

真实问题：

- hook 分类日志仍出现 3MB+ recent。
- archive 里出现 4MB+ 文件。
- 这会拖慢 rg、board debug、人工排查。

实现要求：

- 每个 log category 有明确 cap：

```text
.agentcall/events/recent.ndjson <= 2MB
.agentcall/logs/hooks/*/recent.ndjson <= 1MB
archive chunk <= 2MB
tool artifacts externalized by default
```

- 写入前/后都要检查 size。
- 如果单条 event 超过 cap：
  - payload 必须 artifact 化。
  - recent 只保留 pointer。

- 提供诊断脚本：

```text
python scripts/agentcall_dev.py logs-health
```

输出：

```text
largest recent logs
largest archives
largest artifact categories
events count
oversized files
```

验收测试：

- `hook_recent_log_rotates_under_cap`
- `oversized_hook_payload_externalized_before_write`
- `logs_health_reports_oversized_files`

### P2.4 MCP Timeout Policy

真实问题：

- `agentcall-mcp` 10s READ_TIMEOUT 在大状态下容易造成 transport closed。

实现要求：

- MCP daemon client timeout 分类更细：

```text
connect_timeout
read_timeout
daemon_busy_timeout
response_too_large
```

- 默认 board/session compact 必须在 1s 内返回。
- full/debug 请求允许更长 timeout，但必须显式 debug include。
- 不用单纯把所有 timeout 拉长掩盖慢路径；timeout 增加只能作为 debug/full path 的兜底。

验收测试：

- `compact_board_returns_under_budget_with_large_logs`
- `daemon_client_classifies_read_timeout`
- `debug_full_request_can_use_extended_timeout`

## P3: PTY Interaction Latency And Stop Semantics

### P3.1 Remove Blocking Sleeps From Actor Critical Path

真实问题：

- `write_input` 里 `80ms sleep` 阻塞 actor。
- interrupt 里 `250ms sleep` 阻塞 actor。

实现要求：

- actor 不在处理命令时 sleep。
- 如果 TUI 需要 delay：
  - 使用 delayed task / timer thread。
  - 或把 write 拆成两个 queued low-level write，但 high-priority stop/interrupt 可插队。

验收测试：

- `send_enter_delay_does_not_block_actor_stop`
- `interrupt_redirect_delay_does_not_block_kill`

### P3.2 Stop And Kill Semantics Split

v5.1 未闭口问题：

- StopSession 和 KillSession 都走强杀。

实现要求：

```text
stop:
  request graceful exit
  wait bounded grace
  interrupt if needed
  kill only after grace exceeded

kill:
  immediate process tree kill
```

- stop result 不得谎称 completed，应该返回：

```text
stop_requested
graceful_exited
kill_escalated
kill_completed
```

验收测试：

- `stop_attempts_graceful_before_kill`
- `kill_immediately_kills_process_tree`
- `stop_timeout_escalates_to_kill`

## P4: Test And Release Honesty

### P4.1 Pytest Must Be Reproducible On Windows

真实问题：

- 自评说 17 passed。
- OPUS 实测 3 passed / 14 errors，主要是 Windows 文件锁 teardown。

实现要求：

- 修 pytest 临时目录清理：
  - 使用 per-test temp path。
  - teardown retry with backoff。
  - 不删除仍被 daemon/process 持有的目录，先 stop process。
- `python -m pytest -q` 必须可重复绿。

验收测试：

- 本地 `python -m pytest -q` clean pass。
- `scripts/agentcall_dev.py release-check --skip-cargo` clean pass。

### P4.2 Version And Sign-off Language

当前问题：

- README / plugin 是 v4.3.0。
- Rust crate 是 3.0.0。
- docs 有 v5.x plans。
- 自评写了 core gates complete，但 deep audit 证明 hard gates 未闭合。

实现要求：

- v5.3 完成前，不得写 “v5.2 complete/sign-off”。
- 新增 `docs/reports/v5.3-closure-status.md`：
  - 列出 v5.0-v5.2 hard gates。
  - 每项必须有 implementation evidence 和 test evidence。
- CHANGELOG 加 v5.3 条目时只能写：

```text
v5.3 closes v5 hard gates and real-run feedback issues.
```

不能提前发布 v5.3。

## Implementation Order

按顺序做，不允许先做低优先级外观整理：

1. P0.1 precondition seq validation。
2. P1.1 report write projection。
3. P1.2 utility worker report-ready lifecycle。
4. P0.2 actor/writer/reader/orphan failure projection。
5. P2.2 event linear scan -> projection flags。
6. P2.1 compact MCP path hard cap。
7. P2.3 log cap + logs-health script。
8. P3.1 actor sleep removal。
9. P0.3 RawWrite isolation + priority control。
10. P3.2 stop/kill semantic split。
11. P1.3 read-only TaskCreate/subagent drift control。
12. P4 pytest reproducibility。
13. P4 closure status document and version language cleanup。

## Acceptance Criteria

v5.3 只有在以下全部成立时才算完成：

- stale `projection_last_session_seq` command 被拒绝，且有测试。
- actor/writer/reader/orphaned 失败会投影到 board attention，且有测试。
- `RawWrite(Vec<u8>)` 不在 `ActorControlCommand`。
- report path 写入后 projection 显示 `report_ready=true`。
- 刚才这类 utility worker 写 report 后不再停留为普通 `waiting_input`。
- read-only worker 默认不能自行拉 implementation subagent。
- compact board/session 不读取大 events/transcript。
- `session_send` 不再线性扫描 events 判断 hook seen。
- hook recent logs 和 events recent logs 有真实 size cap。
- actor command path 不再被 fixed sleep 阻塞 stop/kill。
- stop 和 kill 语义不同。
- `python -m pytest -q` 在 Windows 本机 clean pass。
- `cargo test -p agentcall-daemon -p agentcall-mcp` pass。
- `python scripts/agentcall_arch_audit.py` pass。
- 新增 `docs/reports/v5.3-closure-status.md`，明确 v5.0-v5.2 closure evidence。

## Real-run Regression Scenario

v5.3 必须复跑这次暴露问题的真实场景：

1. 用 AgentCall route 拉两个 read-only PTY utility workers：
   - functional implementation review
   - code robustness review
2. 两个 worker 只能写各自 report。
3. report 写出后 board 必须显示：

```text
report_ready=true
attention_status=report_ready
required_next_step=accept_report_or_stop_worker
files_written_count>=1
```

4. `agentcall_report(action=accept)` 能接受 report。
5. accept 后 worker lease 释放，session 可 stop。
6. board compact response 小于 128KB。
7. `.agentcall/logs/hooks/*/recent.ndjson` 不超过 cap。

## Notes

v5.3 不是新版本炫技，而是收口版本。

如果 P0 hard gates 没闭合，任何 “v5 complete” 都不成立。

如果真实 worker report 写出后 board 仍不认，AgentCall 就还没有完成它最核心的监督职责。
