# AgentCall v5.1 Code Plan：SessionActor + 进程所有权 + 并发 PTY Worker

## Summary

v5.1 解决 “Codex 控得住 Claude Code worker”。

v5.0 只让 Codex 看得快、看得准；v5.1 必须把每个 Claude Code PTY worker 变成 daemon 明确拥有的 session actor：

```text
一个 session 一个 actor
一个 actor 一个 PTY writer
所有输入走 command inbox
所有 command 有 idempotency_key / command_id / precondition
Windows 进程树由 ProcessController 管理
```

一句话：

```text
v5.1 让 Claude Code worker 不再失控。
```

## Non-goals

- 不做多 runtime 泛化。
- 不恢复 ACP 默认 runtime。
- 不引入 Kafka/Redis。
- 不要求完整 SQLite 默认 store。
- 不让 MCP bridge 直接管理 actor。
- 不允许外部绕过 actor 写 PTY stdin。
- 不把 Claude report 当最终事实。
- 不让 PTY output 与 control command 共用一个无界队列。
- 不把 `interrupt.sent` 当成 command completed。
- 不把 Windows Job Object 留到 v5.2。

## Hard Corrections From GPT Pro

v5.1 必须强化 actor 边界，而不是把旧函数搬进 `actor.rs`：

1. control inbox 与 PTY output channel 必须分离，避免输出洪水阻塞 stop/interrupt。
2. PTY writer 必须 move into actor，不能继续暴露在 `Arc<Session>` 上。
3. command registry 用 append-only log + small index，不用单个 `commands.json` rewrite。
4. write command 必须携带 `owner_id / owner_lease_id / lease_generation`。
5. interrupt 只有观察到 idle/可输入后才能 completed。
6. WindowsJobProcessController 是 v5.1 必做项，至少 `kill_tree` 要真实验证。
7. TUI/PTY output 继续 hint-only，不得 authoritative。
8. `ActorControlCommand` 只能承载小型控制消息；PTY output 不得以大 `Vec<u8>` 形式进入 control inbox。
9. `interrupt.sent`、`stop.sent`、`kill.sent` 都只是控制信号，不是完成状态；必须观察到 idle/exited/killed 后才能 `command.completed`。
10. 如果 Windows Job Object 无法在 v5.1 实装，就必须把验收降级为 `kill_tree_best_effort` 并显式标红；不得写“已支持 kill tree”。
11. actor panic、writer closed、process orphaned 必须投影为 `failed_or_orphaned`，不得继续显示 healthy running。
12. high-volume PTY output 只能影响 tail/brief，不得阻塞 `interrupt` / `stop` / `kill`。

## Implementation Gate

v5.1 不以“代码里出现 actor.rs”为验收，而以外部无法绕过 actor 控制 PTY 为验收。任何 PR 合并前必须满足：

```text
1. Session 类型不持有可写 PTY stdin handle。
2. PtyWriter 只能 move 进 SessionActor，不能被 Arc<Session>、route、mcp、http、runtime 直接拿到。
3. control command 和 PTY output 不能共用同一个大对象无界队列。
4. interrupt/stop/kill 的 command completed 只能来自观察结果，不能来自 sent。
5. owner_id / owner_lease_id / lease_generation 对写操作必填。
6. commands registry 必须 append-only，可重建 index；不得使用单个 commands.json rewrite 作为事实源。
7. actor panic / writer closed / process orphaned 必须有状态投影和事件，不能沉默失败。
```

如果 Windows Job Object 无法在本轮落地，文档、runtime_health、测试名和验收标准必须统一降级为 `portable_pty_best_effort`，不能对 Codex 或用户展示“kill tree guaranteed”。这条不允许靠措辞糊过去。

### SessionActor

`actor.rs`：

```rust
pub(crate) struct SessionActor {
    pub session_id: String,
    pub run_id: String,
    pub owner_id: String,
    pub owner_lease: OwnerLease,
    pub workspace_lease: Option<WorkspaceLease>,
    pub session_view: Arc<Session>,
    pub writer: PtyWriter,
    pub control_rx: std::sync::mpsc::Receiver<ActorControlCommand>,
    pub output_rx: std::sync::mpsc::Receiver<PtyOutputSignal>,
    pub projection: SessionProjectionV1,
    pub seq: u64,
}

pub(crate) enum ActorControlCommand {
    Start,
    Submit(CommandEnvelopeV1),
    HookEvent(EventEnvelopeV1),
    ProcessExited(String),
    RefreshProjection,
}

pub(crate) struct PtyOutputSignal {
    pub output_seq: u64,
    pub bytes_seen: usize,
    pub brief: Option<String>,
}
```

P1 可先用 `std::sync::mpsc` 和 `std::thread`，不要一开始引入 Tokio。但不能使用单个无界 channel 承载所有内容。

如果继续使用 stdlib channel，必须满足：

```text
control path:
  只传小型 command envelope。
  stop/interrupt/kill 不得等待 PTY output drain。

output path:
  不传 raw Vec<u8> 大块输出。
  由 PTY reader 写 replay/log，并向 actor 发送 coalesced signal。
  output signal 可以丢弃或合并，但 control command 不能被丢弃。
```

也就是说，v5.1 的最低实现不是“两个无界队列就算分离”，而是 control path 在行为上必须具备优先级。若做不到 bounded queue，就必须用“output 不进 control inbox”的结构兜住。

通道规则：

```text
control channel:
  Start / Submit / Interrupt / Stop / Kill / RefreshProjection
  small queue
  high priority

output channel:
  bounded/coalesced signal
  不直接传大 Vec<u8>
  不得阻塞 control command
```

如果 stdlib channel 无法 bounded，先采用 “PTY reader 写 replay/log + actor 只接收 coalesced signal”。

严禁把 `PtyOutput(Vec<u8>)` 放入 `ActorControlCommand`。高频输出只能进入 replay/log writer 或 coalesced output signal；`StopSession` / `InterruptTurn` / `KillSession` 必须优先于输出处理。

```rust
pub(crate) struct Session {
```

`PtyWriter` 只 move 进 `SessionActor`。`mcp.rs`、`routes.rs`、`summary.rs` 不能获得 writer。

类型边界验收：

```text
Session 不包含 writer。
ActorHandle 只暴露 command sender。
runtime / route / mcp / http 只能 submit CommandEnvelopeV1。
任何 raw write helper 必须位于 actor-private 路径，或仅用于测试。
```

## ProcessController

`process.rs`：

```rust
pub(crate) trait ProcessController: Send + Sync {
    fn spawn_pty_session(&self, spec: SpawnSpec) -> Result<ProcessHandle, String>;
    fn interrupt_turn(&self, handle: &ProcessHandle) -> Result<InterruptResult, String>;
    fn graceful_stop(&self, handle: &ProcessHandle) -> Result<StopResult, String>;
    fn kill_tree(&self, handle: &ProcessHandle) -> Result<KillResult, String>;
    fn heartbeat(&self, handle: &ProcessHandle) -> Liveness;
    fn detect_orphan(&self, binding: &SessionBinding) -> OrphanStatus;
}
```

P1 Windows 现实约束：

- interrupt：尝试 Ctrl+Break / existing PTY interrupt。
- stop：先 graceful，再 interrupt，再 kill。
- kill：最终必须能 kill Claude 及子进程树。Windows 下 v5.1 必须实现 Job Object 或等价可验证方案。
- orphaned：不能假装 alive。无法确认 ownership 时标记 `orphaned`。

必须做两层实现：

```text
PortablePtyProcessController
  only for non-Windows or fallback

WindowsJobProcessController
  required on Windows
  owns Job Object
  kill_tree uses TerminateJobObject or equivalent verified implementation
```

验收规则：

```text
如果 WindowsJobProcessController 未落地：
  acceptance 不得写 windows_job_kill_cleans_child_process_tree 通过。
  runtime_health 必须显示 process_controller=portable_pty_best_effort。
  stop/kill summary 必须提示 child process cleanup not guaranteed。

如果 WindowsJobProcessController 已落地：
  runtime_health 显示 process_controller=windows_job。
  测试必须覆盖 parent + child process tree cleanup。
```

### PR 3：Idempotency and precondition durable enough

修改：

```text

```

代码级任务：

- P1 将 v5.0 的 idempotency 从 helper 提升为 append-only command registry：

```text
.agentcall/state/commands.ndjson

```

- `commands.ndjson` 是事实来源；`commands.index.json` 是可重建索引。
- 流程：

```text
1. append command.accepted / command.deduped / command.rejected event
2. append command registry line
3. update small index snapshot
4. index 丢失或损坏时从 commands.ndjson 重建
```

- command registry line 字段：

```json
{
  "command_id": "...",
  "session_id": "...",
  "idempotency_key": "...",
  "fingerprint": "...",
  "status": "accepted|started|completed|failed|deduped|cancelled",
  "created_at": "...",
  "updated_at": "..."
}
```

- actor 执行前检查 precondition：
  - projection seq mismatch -> rejected_precondition。
  - turn running + command=send_input -> queued or rejected_busy。
  - wrong owner -> rejected_owner。

测试：


代码级任务：

- `Session.status: Mutex<String>` 逐步替换或映射到 `ProcessState`。
- `interrupt` command：
  - 不销毁 session。
  - 产生 `interrupt.requested` / `interrupt.sent`。

  - 测试不能把 best effort 伪装成 kill tree 通过。
- daemon 启动时：
  - 读取 projection snapshot。
  - 对无法确认 process ownership 的 running session 标记 orphaned。

测试：

## Acceptance Criteria

- `agentcall_session_send` 不再绕过 actor 直接写 PTY。
- PTY writer 类型上只归 actor 所有。
- control command 不会被高频 PTY output 阻塞。
- 同 session 并发 send 有确定顺序或被拒绝。
- Codex 重试相同 idempotency key 不重复写入。
- running 状态下自然语言输入返回 queued/rejected_busy。
- permission menu 只能走 structured select_option。
- stop/interrupt/kill/orphaned 语义清晰且有观察结果事件。
- Windows 上 `kill_tree` 可以真实清理子进程树。
- actor panic、writer closed、process orphaned 不得被 summary 继续显示为健康 running。
- 高频 PTY output 不得阻塞 `interrupt` / `stop` / `kill`。
- `agentcall_board(scope=mine)` 可只看当前 owner。
- `cargo test -p agentcall-daemon -p agentcall-mcp` 通过。
