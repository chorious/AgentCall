# AgentCall v5.0 Architecture Refresh：实时控制面与审计层解耦

## Summary

v5.0 的核心判断：AgentCall 当前的性能瓶颈不是 Rust 慢，也不是 MCP 协议天然慢，而是 **实时控制面查询混入了审计/历史重构路径**。

Codex 通过 MCP 查询 AgentCall 时，大多数时候只需要回答一个问题：

```text
现在有没有 worker 需要我介入？如果没有，我应该等多久？
```

但当前 daemon 仍会在部分路径里临时拼装 full board、扫描 session summary、读取 routes/reports/events/transcripts/legacy sessions，再裁剪成 compact 结果。结果是轻量状态查询被接到了重审计电路上。

v5.0 的目标不是继续扩功能，而是重建分层：

- **Hot path**：实时控制面，只读内存/小投影，面向 Codex 默认 MCP 调用。
- **Warm path**：最近窗口和 report/route 索引，面向普通诊断。
- **Cold path**：完整 logs、transcripts、archives、artifacts，面向显式 debug/doctor。

## 背景结论

### MCP 电路

当前请求电路：

```text
Codex App
  -> stdio MCP transport
    -> agentcall-mcp.exe
      -> HTTP POST /api/mcp/call
        -> agentcall-daemon
          -> mcp_board / mcp_session / mcp_route
            -> board_state / session_summary / routes / hooks / logs
      <- JSON
    <- MCP text content
  <- Codex tool result
```

`agentcall-mcp.exe` 本身是薄网关，但它是 stdio 单通道，同步处理请求。一个慢 daemon call 会阻塞后续 MCP call。MCP 层真正需要修的是 timeout、默认轻量化、响应限流和 timing log；业务复杂度不应留在 MCP bridge。

### Daemon 电路

当前最重路径集中在：

```text
board_state()
  -> attention_items()
    -> session_summary() × live sessions
      -> clean output / events / routes / bindings / policy / plan artifact
  -> events / reports / routes / legacy sessions / runtime_health
```

尤其需要注意：

- `filter=attention` 能提前返回，但返回前仍要计算 `attention_items()`。
- `view=compact` 如果没有 attention filter，仍可能先构造 full board 再裁剪。
- `session_summary()` 默认路径包含太多昂贵推断，部分路径会扫描 transcript。
- `runtime_health()` / board 查询路径里存在清理状态等写行为。

```text
zellij -> PTY / daemon / session runtime

crush   -> hooks / permission / session / MCP / skills
```

## 目标架构

### 1. MCP Bridge：薄、稳、有超时

`agentcall-mcp` 的职责只应是：

- stdio JSON-RPC 协议适配。
- daemon bootstrap/status。
- 将工具列表和工具调用转交 daemon。
- 对 daemon HTTP 设置 connect/read/write timeout。
- 记录 MCP timing log，区分排队、daemon、序列化、stdout 写回。
- 对返回内容做大小上限保护。

不应该在 MCP bridge 内实现业务 workflow、session summary、file scan、report scan。

### 2. Daemon Hot Path：实时控制面

Hot path 是 Codex 默认路径，只回答低成本问题：

```json
{
  "attention": [],
  "running": [
    {
      "session": "name",
      "liveness_status": "working",
      "attention_status": "none",
      "last_progress_age_seconds": 17,
      "patience_status": "inside_patience_window"
    }
  ]
}
```

Hot path 只能读取：

- daemon 内存 session registry。
- session projection。
- board projection。
- 小型 route/report projection。
- policy/permission projection。

Hot path 禁止默认读取：

- full transcript。
- raw output log。
- legacy sessions。
- archives。
- full hook logs。
- full routes/reports 历史。

### 3. Projection Layer：事件进入时更新，不在查询时考古

新增或强化投影层：

```text
session_projection
  - session name
  - runtime pid/cwd/status
  - hook binding
  - liveness_status
  - attention_status
  - pending_interaction
  - policy_block
  - last_progress_at
  - report_ready

board_projection
  - attention list
  - running compact list
  - recent completed/failed summaries
  - health compact fields

route_projection
  - route_id
  - session_name
  - status/workflow_status
  - required_next_step
  - patience hints

report_projection
  - report path
  - status
  - task/session binding
  - updated_at
```

事件来源包括：

- PTY reader progress。
- hook ingest。
- route/session_send。
- stop/interrupt/select_option。
- report accept/request。
- periodic maintenance。

查询路径只读 projection，不重放日志。

### 4. Cold Path：审计、debug、doctor

完整日志和历史是必要的，但必须显式请求：

- `.agentcall/events/recent.ndjson`
- `.agentcall/events/archive/**`
- `.agentcall/logs/hooks/**`
- `.agentcall/artifacts/hooks/**`
- Claude transcript JSONL
- raw PTY output
- legacy session state

这些路径应该由 `logs doctor`、debug endpoint、explicit full board 或专门脚本使用，不进入 MCP 默认工具返回。

## Non-goals

- 不恢复 ACP 作为默认 runtime。
- 不让 Python 重新成为 live state writer。
- 不把 MCP bridge 做成业务服务。
- 不让 compact board 默认扫描 full history。
- 不让 session summary 默认读取 transcript。
- 不在 GET board/health 热路径里做重清理或大量写 I/O。
- 不用 TUI regex 覆盖 hook-derived 状态。
- 不自动 kill 用户可见 worker，除非后续明确设计 lifecycle policy。

## P0 Plan

### P0-1：真正的 compact/attention fast path

- `agentcall_board(view=compact, filter=attention)` 不调用 full `session_summary()`。
- attention 使用 `session_projection` 中已有字段。
- 默认 MCP board 参数改为 compact attention。
- session list 默认只返回 live running + attention，不返回 exited/legacy。

验收目标：

- attention board 目标 < 300ms。
- 返回大小目标 < 16KB。

### P0-2：session_summary 分层

将 `agentcall_session` 拆为默认 summary 与 explicit includes：

- 默认：只读 `session_projection`。
- `include=clean_tail`：读取/清洗 output tail。
- `include=plan`：读取 plan artifact 或 transcript。
- `include=raw`：debug-only。

验收目标：

- 默认 session summary 不读 transcript。
- 默认 summary 目标 < 300ms。

### P0-3：MCP timeout + timing log

- MCP -> daemon connect timeout。
- write/read timeout。
- MCP timing log 写入 `.agentcall/logs/mcp/recent.ndjson`。
- timeout 返回结构化错误，不拖满 Codex 120s。

timing 字段：

- `request_received_at`
- `daemon_request_ms`
- `daemon_response_bytes`
- `serialize_ms`
- `stdout_write_ms`
- `total_ms`

### P0-4：GET 查询不做重写入

- `runtime_health()` / `board_state()` 中的 cleanup 移到 background maintenance。
- GET 热路径不持有 `state_writer` 做文件写入。

### P0-5：减少重复读取

单个 daemon request 内复用：

- events tail。
- routes state。
- runtime bindings。
- pending supervisor instructions。
- policy denials。

短期可以 request-local cache，长期由 projection 替代。

## P1 Plan

- route/report 维护 recent index，不每次扫目录。
- legacy sessions 从默认 board 移除，改为 debug section。
- hook artifact 加 TTL/size budget。
- policy denial 聚合按 `(tool, target, reason)`，不被其他 allowed tool 清零。
- TUI clean output 增量化，不每次 clone + 全量清洗。
- stop/report-ready 后自动标记 worker 已交付，阻止继续执行“fix P0”一类漂移 prompt。

## P2 Plan

- 参考 Vector 引入 bounded channel / backpressure 指标。
- 参考 Zellij 将 PTY bus、screen/output projection、background jobs 拆清。
- 参考 Crush 将 hooks engine、permission engine、session persistence、pubsub 分离。
- 考虑 SQLite 或 sled 作为 projection/index 存储，但不是 P0。

## 当前证据

三份性能报告已经产出：

- `docs/reports/perf_audit_mcp_control.md`
- `docs/reports/perf_audit_state_logs.md`
- `docs/reports/perf_audit_pty_io.md`

共同结论：

- MCP bridge 应保持薄，但需要 timeout/timing/response cap。
- daemon 查询路径目前承担太多 full-state 构造。
- `session_summary()` 是 N 倍放大的关键负载节点。
- transcript/full output/log archives 必须退出默认 summary/board。

## 推荐下一步

1. 先实现 P0-1 和 P0-2，证明 MCP 慢的主因可以被 hot path 拆分消除。
2. 再实现 P0-3，用 timing log 明确后续瓶颈。
3. 然后根据真实 timings 决定是否进入 projection cache / background maintenance。

一句话原则：

```text
AgentCall daemon 是实时控制面，不是每次请求都现场考古的审计器。
```
